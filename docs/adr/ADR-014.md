# ADR-014 — Peer Review: Cost-Free Assessment Through Collective Correction

**Status:** Accepted  
**Date:** April 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-003 (Batch Generation), ADR-005 (Adaptive Learning Path), ADR-007 (AI Pipeline), ADR-013 (Learning Flow)

---

## Context

The final task of each topic is a `TODO_ACTION` — the user photographs something
real that demonstrates they have applied what they learned. A learner studying
sustainable architecture photographs a building they identified. A learner
studying digital marketing photographs a campaign they created. The task is
inherently open-ended: there is no single correct answer that a multiple-choice
system can evaluate.

Three mechanisms exist for evaluating open-ended submissions:

1. **Human expert review** — an admin or educator reviews each submission.
   Accurate, but does not scale. At 10,000 users, this is a full-time job.

2. **AI evaluation** — a vision model or LLM reviews the photo against the
   task criteria. Accurate at small scale, but costs money on every submission
   and introduces a dependency on model availability.

3. **Peer review** — enrolled learners review each other's submissions.
   Zero AI cost, zero admin involvement, and — critically — the act of reviewing
   someone else's work is itself a learning activity.

The third option is the correct choice for this system. This ADR documents the
design decisions that make peer review work correctly: reviewer selection,
majority vote finalization, social feedback, and roadmap integration.

---

## Decision Summary

| Concern | Decision |
|---------|----------|
| Evaluation mechanism | Peer review — 3 verdicts per submission |
| Reviewer selection | Priority: unreviewed first, then partial, then null |
| Randomness | Application-level `.random()`, not `ORDER BY RANDOM()` |
| "Never same reviewer twice" | Application filter + DB unique constraint |
| Finalization | Synchronous on 3rd verdict — majority of 3 wins |
| Reviewer reward | 10 points per completed review, added to enrollment |
| Submitter reward | Full points if ≥2 CORRECT verdicts, 0 if ≥2 INCORRECT |
| Social feedback | Verdict percentage breakdown returned after each review |
| Roadmap integration | `PENDING_REVIEW` and `REVIEW_PENDING` node statuses |
| Port boundary | `TaskService` calls `PeerReviewUseCase` interface — not `PeerReviewService` directly |

## The Pedagogical Case for Peer Review

Peer review is not a cost-cutting measure that happens to have pedagogical
value — it is a pedagogically superior assessment mechanism for applied tasks.

**Reviewing requires understanding.** To judge whether another learner's
photograph correctly demonstrates the concept, the reviewer must recall and
apply the same knowledge they just learned. This is retrieval practice — one
of the most evidence-backed learning techniques. A learner who reviews three
peer submissions after completing a topic retains more than a learner who simply
received a grade.

**Social learning signals.** When a reviewer sees "67% of reviewers marked
this as correct," they receive two signals simultaneously: confirmation or
challenge of their own judgment, and evidence of how the broader learner
community understands the task criteria. This is formative feedback without
a teacher.

**Distributed validation reduces individual bias.** A single AI model or a
single human reviewer brings a fixed perspective. Three independent learner
assessments, aggregated by majority vote, produce a more robust signal than
any single evaluator — especially for open-ended tasks where the criteria
involve real-world judgment, not factual correctness.

## Decision 1 — Reviewer Selection Algorithm

### The problem

When a user submits their photo, the system should immediately offer them a
peer submission to review. The selection must satisfy:

- **Never the user's own submission** — obvious correctness requirement
- **Never the same submission to the same reviewer twice** — prevents double-voting
- **Priority: unreviewed first** — submissions with 0 reviews need attention most
- **Random within priority tier** — prevents systematic bias toward early submitters

### The algorithm

Selection is computed in the application layer, not the database layer:

```
1. Load all PENDING_REVIEW TODO_ACTION assignments for the topic,
   excluding the current user's own (JPQL join: assignment → enrollment → topic)

2. For each candidate assignment, load its existing peer_reviews
   → filter out: any where current user is already assigned
   → filter out: any where review count ≥ 3 (already finalized, guard)

3. Categorise remaining into:
   - unreviewed: review count == 0
   - partial:    review count in 1..2

4. Select:
   - unreviewed.isNotEmpty() → pick unreviewed.random()
   - else partial.isNotEmpty() → pick partial.random()
   - else → return null (tell user no reviews available yet)

5. Create PeerReview(verdict=null, assignedAt=now) and save
```

### Why application-level selection, not a DB query

The alternative is `ORDER BY RANDOM()` in SQL with a subquery to exclude
ineligible candidates. This works but has two problems at scale:

`ORDER BY RANDOM()` forces a full table scan and sort on every call — the
database cannot use any index. For 10,000 users each submitting once,
the `task_assignments` table grows to 10,000 rows. A full sort on every
reviewer assignment request is unnecessary CPU spent in PostgreSQL.

The application-level approach loads only the relevant candidates
(filtered by topic, task type, and status — all indexed), then applies
eligibility filters and randomness in memory. The population per topic is
always small — hundreds at most, not millions. The memory cost is negligible.

### Why `ORDER BY RANDOM()` is the wrong primitive here

`ORDER BY RANDOM()` produces a uniform random ordering across the entire
result set. What we need is not a uniform random — we need a prioritised
random: "unreviewed first, then partial, uniformly at random within each tier."
This two-tier priority cannot be expressed cleanly in SQL without a CASE in
the ORDER BY, which is both harder to read and still forces the full sort.
The application-level approach expresses the intent directly.

### "Never same reviewer twice" — double safety

The eligibility filter in step 2 excludes any submission where the current
user already has a `PeerReview` record (regardless of verdict status). This
is the primary enforcement mechanism.

The database provides a secondary guarantee: `peer_reviews` has a `UNIQUE`
constraint on `(submission_assignment_id, reviewer_user_id)`. If a race
condition somehow bypasses the application filter and attempts to insert a
duplicate, the database rejects it. The system cannot corrupt data even under
concurrent load.

## Decision 2 — Finalization: Synchronous on the 3rd Verdict

### The problem

When the 3rd verdict is submitted, four things must happen atomically:

1. Update `TaskAssignment.isCorrect` and `pointsAwarded`
2. Update `TaskAssignment.status` to `SUBMITTED` (or `LATE`)
3. Add points to the submitter's `Enrollment.totalPointsEarned`
4. Add review reward points to each of the 3 reviewers' enrollments

The question is: should this happen synchronously in the same HTTP request
as the 3rd verdict, or asynchronously in a background job?

### Decision: synchronous

Finalization is synchronous. The 3rd verdict request triggers finalization
and returns the result in the same response.

**Why not async:**

Async finalization (via `@Async` or an event) would mean the 3rd reviewer
receives `{ finalized: false }` in their response — even though finalization
is about to happen. The reviewer would need to poll or receive a push
notification to see the final result. This is unnecessary complexity for an
operation that takes milliseconds.

The finalization work is bounded: count verdicts (1 query), update 1 assignment,
load and update at most 4 enrollments (1 submitter + 3 reviewers). All within
a single database transaction. The P99 latency of this operation is
well under 100ms — no reason to move it off the request thread.

Async is the right choice when work is unbounded in time (AI generation,
file processing) or when failure must not affect the response (background
cleanup). Neither applies here.

**The finalization flow:**

```
3rd verdict submitted
  │
  ├── Save verdict + reviewedAt = now
  │
  ├── Load all 3 verdicts for this submission
  │
  ├── Count CORRECT verdicts
  │     ≥ 2 CORRECT → isCorrect = true, pointsAwarded = template.pointsReward
  │     ≥ 2 INCORRECT → isCorrect = false, pointsAwarded = 0
  │
  ├── Determine late penalty:
  │     submittedAt > dueAt → status = LATE (half points applied)
  │     else → status = SUBMITTED
  │
  ├── Save updated TaskAssignment
  │
  ├── Load submitter's Enrollment → add pointsAwarded → save
  │
  ├── For each of 3 reviewers:
  │     Load Enrollment by (reviewerUserId, topicId)
  │     If ACTIVE → add REVIEW_POINTS_REWARD (10) → save
  │
  └── Return VerdictResult(finalized=true, submitterPointsAwarded=...)
```

### Reviewer points and topicId derivation

Each reviewer earns 10 points added to their enrollment for the same topic.
The `topicId` is not stored on `PeerReview` directly. It is derived by:

```
review.submissionAssignmentId
  → TaskAssignment.enrollmentId
  → Enrollment.topicId
```

One extra DB read per finalization — acceptable given this happens once
per submission across the topic's entire lifecycle.

`REVIEW_POINTS_REWARD = 10` is defined as a `const val` in `PeerReviewService`,
not in the domain model. The value is a product decision, not a domain
invariant. Moving it to configuration (`application.yml`) is a future
enhancement if per-topic configurable review rewards become a requirement.

## Decision 3 — Social Feedback: Verdict Percentage Breakdown

### The mechanic

After submitting a verdict, the reviewer sees the current distribution
of all verdicts on that submission:

```json
{
  "verdict": "CORRECT",
  "currentVerdicts": { "correct": 2, "incorrect": 1, "total": 3 },
  "finalized": true,
  "submitterPointsAwarded": 80
}
```

This is returned in the same response as the verdict submission — no
additional request needed.

### Why this matters

**Confirmation of judgment.** The reviewer just assessed something
subjective. Seeing "2 of 3 reviewers agreed with you" validates their
assessment and reinforces their understanding of the task criteria.
Seeing "you were the minority" creates a productive moment of reflection:
"Why did I see this differently?"

**Social proof without social pressure.** The breakdown is shown after
the verdict is submitted — the reviewer cannot see what others said before
making their own judgment. This prevents anchoring bias (the tendency to
agree with the first assessment you see). Sequencing matters: verdict
first, then social signal.

**Transparency builds trust.** Learners are more willing to accept
a peer-reviewed outcome when they can see the exact vote distribution.
"You received 2 CORRECT and 1 INCORRECT, so the outcome is CORRECT"
is more trustworthy than a silent automatic grade.

### What is NOT shown

The identity of the other reviewers is never exposed. `PeerReviewAssignmentResponse`
contains only `reviewId` and `submissionPhotoUrl`. No username, no avatar,
no profile link. Anonymity protects reviewers from social pressure and
prevents learners from gaming the system by coordinating verdicts with friends.

## Decision 4 — Roadmap Integration

### The problem

The roadmap is the user's primary mental model of their progress. Peer review
introduces two new states that must be represented:

1. **The user has submitted their photo** but the 3 verdicts are not yet in.
   Their submission is waiting. They cannot advance to a "completed" state yet.

2. **The user has been assigned a peer review** but has not yet submitted
   their verdict. They have a pending obligation.

Both states must be visible on the roadmap so the user understands their
current situation without needing a separate notifications system.

### Two new `RoadmapNodeStatus` values

**`PENDING_REVIEW`** — shown on the user's own `TODO_ACTION` node when
`TaskAssignment.status == PENDING_REVIEW`. The task was submitted but awaits
3 verdicts. The node is "in progress" — not locked, not complete.

**`REVIEW_PENDING`** — an additional node appended to the roadmap for each
`PeerReview` record assigned to the user with no verdict yet. These nodes
use `dayIndex = -1` (a sentinel value meaning "not a day-indexed task") and
appear after the regular task sequence. The client renders them as an action
item: "Complete your peer reviews."

### Why `dayIndex = -1` for review nodes

Review obligations are not day-indexed — they appear when another user
submits, which can happen at any time. Using `dayIndex = -1` signals to the
client that these nodes are outside the sequential day structure and should
be rendered differently (e.g. as a side-panel or floating action rather than
a step in the road).

### Why roadmap, not a separate notifications endpoint

A separate endpoint (`GET /notifications/pending`) would require the client
to poll two endpoints on load instead of one. Every screen that displays
pending reviews would need to know about this second endpoint. The roadmap
as a single source of truth for "what do I need to do" is simpler, more
cohesive, and consistent with how the rest of the learning state is exposed.

## Decision 5 — Port Boundary: TaskService → PeerReviewUseCase

### The problem

`TaskService.submitPhoto()` needs to select and assign a peer review
immediately after the user submits their photo. The review selection
logic lives in `PeerReviewService`. The question is how `TaskService`
should call it.

### Decision: inject `PeerReviewUseCase` interface, not `PeerReviewService`

```kotlin
// TaskService constructor — correct
class TaskService(
    ...
    private val peerReviewUseCase: PeerReviewUseCase  // interface
) : TaskUseCase

// NOT this — violates Clean Architecture
class TaskService(
    ...
    private val peerReviewService: PeerReviewService  // concrete class
) : TaskUseCase
```

**Why this matters:**

`TaskService` is in `application/enrollment/`. `PeerReviewService` is in
`application/peerreview/`. Both are in the application layer. Allowing
application services to directly reference each other by concrete type
creates tight coupling between bounded contexts — a change to
`PeerReviewService`'s constructor propagates to `TaskService`.

By injecting the `PeerReviewUseCase` interface (defined in `application/port/`),
`TaskService` depends on an abstraction. Spring injects `PeerReviewService`
at runtime. `TaskService` has no compile-time knowledge of `PeerReviewService`'s
implementation details.

This is the same Port/Adapter pattern applied inward (between application
services) rather than outward (between application and infrastructure).
The principle is consistent: depend on abstractions, not concretions.

## Alternatives Considered

### AI photo evaluation

**Rejected.** A vision model (GPT-4V, Claude Vision, Gemini Vision) could
evaluate whether a photo correctly demonstrates the task. This would eliminate
the need for peer review entirely.

**Why rejected:**
- Cost: at 10,000 users × 1 TODO_ACTION per topic, that is 10,000 AI evaluation
  calls per topic. At ~$0.01–0.05 per call, this is $100–500 per topic. Peer
  review is $0.
- Reliability: vision models hallucinate. A model might pass a submission that
  shows the wrong thing, or fail one that shows the right thing. Human judgment,
  aggregated across 3 reviewers, is more contextually accurate for domain-specific
  applied tasks.
- Dependency: AI evaluation adds a latency-sensitive dependency on an external
  API to the submission path. Peer review has no external dependencies.
- Pedagogical value: AI evaluation teaches the learner nothing. Peer review
  teaches all three reviewers.

### Single reviewer (1 verdict)

**Rejected.** A single reviewer introduces high variance — one reviewer may
misunderstand the criteria, be inattentive, or have a personal bias. One vote
is not a robust signal.

### Five reviewers (majority of 5)

**Rejected.** Five reviewers significantly reduces the likelihood that enough
reviewers are available early in a topic's lifecycle. At 3 reviewers, a
submission can be finalized as soon as 3 other learners have submitted their
own photos. At 5, the wait is substantially longer for early enrollees.
Three is the minimum that makes majority vote meaningful (2-1 split is
still a clear majority).

### Unanimous verdict (all 3 must agree)

**Rejected.** Unanimity is too strict for open-ended visual tasks. A 2-1
split on a subjective photograph is a valid majority — it means two independent
reviewers agreed. Requiring all three to agree would frequently leave submissions
unresolved, requiring re-review or admin intervention.

### Weighted verdicts (first reviewer counts more)

**Rejected.** There is no principled basis for the first reviewer's judgment
being more reliable than the third's. Weighting would require a rationale
that does not exist. Equal weight, majority wins, is the simplest and most
defensible rule.

### Asynchronous finalization

**Rejected.** See Decision 2. The work is bounded and fast. Async complexity
is unjustified when synchronous finalization completes in milliseconds.

## Consequences

- Every `TODO_ACTION` submission enters `PENDING_REVIEW` status immediately,
  regardless of how many reviewers exist at submission time.
- A submission stays `PENDING_REVIEW` indefinitely until 3 verdicts are collected.
  There is no automatic timeout or fallback outcome. If a topic has fewer than
  3 active reviewers, submissions may never finalize — this is a known limitation
  at very small cohort sizes.
- Reviewer anonymity is enforced at the API layer: `PeerReviewAssignmentResponse`
  never includes reviewer identity.
- The verdict percentage breakdown creates a genuine learning moment: reviewers
  see how the community assessed the same submission after committing to their
  own judgment.
- Adding a new evaluation mechanism (e.g. admin override, AI re-evaluation on
  dispute) requires only a new method on `PeerReviewUseCase` — no controller
  or domain model changes are needed.
- The `REVIEW_POINTS_REWARD = 10` constant in `PeerReviewService` is the first
  candidate for externalisation if per-topic configurable review rewards are
  requested.
