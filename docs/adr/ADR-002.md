# ADR-002 — Communication Architecture: Hybrid REST + gRPC

**Status:** Superseded by ADR-006 (gRPC layer removed)  
**Date:** March 2026  
**Author:** Radesh Govind

---

## Context

The system has two communication boundaries that have different characteristics:

1. **KMP Client ↔ Spring Boot Server** — cross-platform mobile clients syncing
   task data, reporting completions, authenticating.

2. **Spring Boot Server ↔ AI Agent** — internal call to generate course tasks,
   originally designed as a separate deployed service (Cloudflare Workers).

Each boundary has different requirements. A single communication protocol
is not the optimal choice for both.

---

## Decision

**Hybrid approach:**

| Boundary | Protocol | Reason |
|----------|----------|--------|
| Client ↔ Server | REST / HTTP | Mobile-friendly, cacheable, stateless |
| Server ↔ AI Agent | gRPC | Typed contracts, binary efficient, streaming |

---

## REST — Client to Server

**Chosen because:**
- KMP/Ktor has first-class HTTP support without additional setup
- HTTP caching semantics (ETag, Last-Modified) benefit infrequent batch sync
- Stateless pull-based sync fits the offline-first model perfectly
- Universally debuggable — any HTTP tool works

**Why not gRPC here:**
- Mobile gRPC requires additional libraries and setup complexity on KMP
- HTTP caching not available over gRPC
- Payload difference between JSON and protobuf is negligible for batch sync

**Why not WebSockets here:**
- No real-time bidirectional communication needed
- Offline-first means the client doesn't need server push
- Connection overhead unjustified for once-per-day sync

---

## gRPC — Server to AI Agent (original design)

**Chosen because:**
- AI agent was designed as a separate Cloudflare Workers deployment
- Both ends under our control — typed protobuf contracts eliminate serialization bugs
- Binary encoding more efficient for large batch task payloads
- Kotlin coroutine-native via `grpc-kotlin`
- Server-streaming RPC allows progress reporting on large batch jobs

**Proto contract defined:**
```protobuf
service TaskGenerationService {
  rpc GenerateTasks(GenerateTasksRequest) returns (GenerateTasksResponse);
  rpc GenerateTasksStreaming(GenerateTasksRequest) returns (stream TaskResult);
  rpc HealthCheck(HealthCheckRequest) returns (HealthCheckResponse);
}
```

---

## Status: Superseded

The gRPC boundary was removed when the AI agent architecture changed.
See **ADR-006** for the full explanation and migration rationale.

The proto file is retained in `docs/proto/` as a historical artifact and
demonstration of the original design intent.

**Key insight preserved:** The `TaskGenerationPort` interface that replaced gRPC
is designed so the transport layer can be reintroduced at this boundary if the
AI agent needs to scale independently. The module boundary today becomes the
service boundary tomorrow with minimal changes.
