# ADR-003 — AI Content Generation: Batch vs On-Demand

**Status:** Accepted (Hybrid)  
**Date:** March 2026  
**Author:** Radesh Govind

---

## Context

The application requires AI-generated task content for learners. Two fundamental
approaches exist for when to invoke the LLM:

- **On-demand:** Generate tasks at runtime when a user requests them
- **Batch:** Pre-generate tasks on a schedule, serve from cache

The choice has significant implications for latency, cost, personalization,
and system complexity.

---

## Decision

**Hybrid approach — batch for the fixed learning path, on-demand for adaptive branches.**

| Content Type | Strategy | Reason |
|-------------|----------|--------|
| Fixed course path tasks | Nightly batch `@Scheduled` | Predictable, cacheable, cost-controlled |
| Adaptive subtasks (struggle branches) | On-demand | Personalised, triggered by user behaviour |

---

## Batch Generation — Fixed Path

**How it works:**
- Spring `@Scheduled(cron = "0 0 2 * * *")` runs at 02:00 daily
- Generates tasks for all active courses across all configured domains
- Stores results in PostgreSQL with pgvector embeddings for deduplication
- Semantic deduplication via pgvector prevents content fatigue

**Why batch for fixed path:**

| Concern | Batch | On-demand |
|---------|-------|-----------|
| User-facing latency | ✅ Zero (pre-generated) | ❌ LLM inference delay |
| Cost scaling | ✅ Scales with domains, not users | ❌ Scales with user count |
| Predictability | ✅ Known window, controllable | ❌ Spikes on peak usage |
| Personalization | ❌ Same content per domain/level | ✅ Per-user |

At 10,000 users across 5 courses — batch makes ~50 LLM calls per night.
On-demand would make ~10,000 calls per day. Cost difference is orders of magnitude.

**Resilience strategy:**
1. `@Retryable` — 3 attempts with exponential backoff (2s, 4s, 8s)
2. Circuit breaker (Resilience4j) — opens after repeated failures
3. Static fallback — pre-authored JSON tasks always available

---

## On-Demand Generation — Adaptive Branches

**How it works:**
- Triggered when `StruggleDetectionService` identifies a struggling user
- Generates personalised subtasks contextualised to the user's specific difficulty
- Searches pgvector for similar past struggles before generating (reuse first)
- Generated branches stored with embeddings for future reuse

**Why on-demand for adaptive content:**
- Struggle patterns are unpredictable — cannot be pre-generated
- Content must be contextualised to the specific user's error pattern
- Volume is low — only triggered for struggling users, not all users
- Latency is acceptable — user already paused, expecting help

---

## Similarity-Based Reuse (the smart part)

Before generating new adaptive content, the system queries pgvector:

```
similarity > 0.90  →  reuse existing branch entirely
similarity > 0.65  →  merge existing + generate complement  
similarity < 0.65  →  generate fresh, store for future reuse
```

This means adaptive generation cost decreases over time as the system
accumulates a library of struggle patterns and their resolutions.

---

## At Scale Considerations

| Scale | Batch | On-Demand Adaptive |
|-------|-------|-------------------|
| 1,000 users | ~50 calls/night | Low volume, fast |
| 100,000 users | ~50 calls/night (unchanged) | pgvector reuse kicks in heavily |
| 1,000,000 users | Partition batch by region | Dedicated adaptive generation workers |

The batch cost curve is flat. The adaptive cost curve grows sublinearly
due to pgvector reuse increasing as the pattern library grows.

---

## Consequences

- Two distinct generation pipelines with different triggers and lifecycles
- pgvector is load-bearing for both deduplication and similarity-based reuse
- Batch failure does not affect adaptive generation and vice versa
- Monitoring must track both pipelines independently in Prometheus/Grafana
