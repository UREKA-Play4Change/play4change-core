# ADR-008 — Client Architecture: Decompose + MVI + Domain-Driven Feature Slices

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-004 (i18n Strategy), ADR-006 (Module Boundary)

---

## Context

The KMP client requires a navigation model, a state management pattern, and
a code organisation strategy that satisfy four constraints simultaneously:

1. **Multiplatform correctness** — the same logic must compile and run on
   Android and iOS without platform-specific forks in business logic.
2. **Lifecycle safety** — coroutine scopes and navigation state must survive
   configuration changes (Android) and be cancelled correctly on destruction.
3. **Testability** — screens must be testable without a running device, and
   the AI pipeline and network calls must be replaceable with deterministic
   fakes at zero cost.
4. **Architectural clarity for a single-person project** — the structure must
   be self-documenting so that any examiner reading the source tree can
   immediately understand where each concern lives.

No standard pattern satisfies all four without deliberate design choices.

---

## Decision

**Three-layer client architecture:**

| Layer | Pattern | Responsibility |
|-------|---------|---------------|
| Navigation | Decompose `ChildStack` | Lifecycle-aware screen routing with serialisable back stack |
| State management | MVI via `BaseComponent` | Unidirectional data flow: Events → State → UI |
| Code organisation | Domain-Driven Feature Slices | Self-contained vertical slices per feature |

These three patterns are chosen together. Each one reinforces the others
and cannot be evaluated in isolation.

---

## Navigation: Decompose ChildStack

### Decision

Use [Decompose](https://github.com/arkivanov/Decompose) with a `ChildStack`
backed by `StackNavigation<Config>`.

### Rationale

The primary alternatives were:

| Option | Android only | KMP | Lifecycle-aware | Back stack serialisable |
|--------|-------------|-----|----------------|------------------------|
| Jetpack Navigation | ✅ | ❌ | ✅ | ❌ |
| Voyager | ✅ | ✅ | Partial | ❌ |
| Decompose | ✅ | ✅ | ✅ | ✅ |

**Why Decompose wins in this context:**

Jetpack Navigation is Android-only. Voyager lacks the lifecycle integration
required for coroutine scope management on both platforms. Decompose is the
only option that provides a `ComponentContext` — a platform-agnostic lifecycle
owner that integrates with both Android's `LifecycleOwner` and iOS's
`UIViewController` lifecycle.

The `@Serializable Config` sealed interface means the back stack survives
Android process death and is restored automatically. This is not a nice-to-have
— it is a correctness requirement for any app that handles authentication state.

**Stack safety contract:**

The navigation model enforces two invariants to prevent circular or unbounded stacks:

- `navigateToHome()` and `navigateToLogin()` use `replaceAll` — they clear the
  entire back stack and make the target the sole entry. A user cannot navigate
  "back" into a previous login flow.
- Modal screens (Profile, About, Explore, Task) use `push` with a guard:
  `if (childStack.value.active.configuration != Config.X) navigation.push(...)`.
  This prevents the same screen being stacked on top of itself through repeated
  rapid taps.

**Transition direction:**

Screen transitions use stack size comparison to determine direction:
`targetState.items.size >= initialState.items.size` → forward (slide from right).
`targetState.items.size < initialState.items.size` → backward (slide from left).
This is the only correct approach for Decompose's `ChildStack` — a global
"slide left" flag cannot distinguish push from pop because the same `AnimatedContent`
receives both events.

## State Management: MVI via BaseComponent

### Decision

Every screen is backed by a `DefaultXxxComponent` that extends `BaseComponent<S, E>`,
where `S : ComponentState` and `E : ComponentEvents`. Screens are pure
composables that receive `(state: S, onEvent: (E) -> Unit)`.

### Core infrastructure

```
BaseComponent<S, E>
  │
  ├── MutableValue<S>          — Decompose's observable value, not StateFlow
  ├── CoroutineScope(Main + SupervisorJob)
  ├── effects: Channel<Effect> — one-time navigation/toast events
  └── doOnDestroy { scope.cancel() }

StatefulComponent<S>
  └── safeLaunch { }           — sets isLoading, catches exceptions → AppError

BaseView(component) { state, onEvent → ... }
  └── subscribeAsState()       — converts Value<S> to Compose State<S>
  └── ErrorDialog              — shown automatically when state.error != null
```

### Why MutableValue over StateFlow

Decompose's `MutableValue` integrates with the component's lifecycle directly —
it does not require a coroutine collector and does not produce a memory leak when
the component is destroyed. `StateFlow` requires collection inside a
`LaunchedEffect`, which introduces a frame of delay before the initial state is
rendered. `MutableValue` renders synchronously on the first composition.

### Events vs effects

The architecture distinguishes two categories of outputs from a component:

- **State updates** — reflected in `S` and observed by the composable. These
  include loading flags, data, and validation errors. The composable re-renders.
- **Effects** — one-time signals emitted via `Channel<Effect>` and collected
  with `LaunchedEffect(component)`. Navigation is always an effect, never a state
  field. Putting navigation targets in state creates the risk of re-triggering a
  navigation on recomposition, which is a documented bug class in ViewModel-based
  architectures.

This distinction is the Decompose-native equivalent of `SingleLiveEvent` or
`SharedFlow(replay=0)` in other architectures.

### Why not ViewModel + StateFlow

ViewModels are Android-specific. On KMP, `androidx.lifecycle.ViewModel` is
available through Compose Lifecycle extensions but carries Android lifecycle
semantics that do not map cleanly to iOS. Decompose's `ComponentContext` is
the platform-agnostic lifecycle owner that makes KMP viable without
`expect/actual` forks in the presentation layer.

## Code Organisation: Domain-Driven Feature Slices

### Decision

Every feature is a self-contained vertical slice under `features/<feature>/`:

```
features/<feature>/
├── domain/
│   ├── model/          ← pure Kotlin data classes, zero imports from Android/Compose
│   └── repository/     ← interfaces only — no implementations
├── data/
│   └── mock/           ← implements repository interfaces with deterministic fake data
├── presentation/
│   ├── XxxState.kt     ← data class implementing ComponentState
│   ├── XxxEvents.kt    ← sealed class implementing ComponentEvents
│   ├── XxxEffect.kt    ← sealed class extending BaseComponent.Effect
│   ├── XxxComponent.kt ← interface (contract, no implementation)
│   └── DefaultXxxComponent.kt ← extends BaseComponent, injected via Koin
└── ui/
    └── XxxScreen.kt    ← @Composable only, receives (state, onEvent), no logic
```

### Dependency rule

Dependency flow is strictly downward and inward:

```
ui → presentation → domain ← (never imports) data
                              data → domain (implements interfaces)
```

Features never import from other features. Shared types live in `core/`.
This is enforced structurally — not by convention — because Kotlin's package
visibility provides no cross-module protection within a single Gradle module.
The rule is enforced by code review and is documented here so it survives
contributor turnover.

### Why DDD over a layered architecture

A traditional layered architecture (`presentation/`, `domain/`, `data/` at the
top level) groups files by technical type. In practice this means a change to
the "home" feature touches files in three separate top-level directories.
Feature slices group files by business capability. A change to the home feature
touches only `features/home/`. This locality reduces cognitive load and makes
the codebase navigable without a mental map of the layer structure.

## Mock Data Layer

### Decision

All repository implementations in `data/mock/` use hardcoded, realistic data
with simulated network delays. Production implementations will live in
`data/network/` and replace mock implementations via Koin's DI binding.
Zero application code outside `data/` changes when switching from mock to real.

### Mock data design principles

Mock data is not placeholder data. It is designed to:

1. **Be domain-realistic** — task content uses real sustainability questions,
   not "Option A / Option B". An examiner interacting with the demo sees
   educational content that reflects the project's actual domain.

2. **Simulate latency** — every mock repository method calls `delay(600)` to
   `delay(1500)` before returning. This ensures loading states are visible
   during development and prevents the UI from being designed only for the
   instant-load case.

3. **Cover edge cases** — the mock roadmap includes a completed adaptive branch
   node alongside main-path nodes, so the UI is always tested against the full
   state space, not just the happy path.

### Switching to production

The production transition requires:
1. Create `data/network/RealXxxRepository.kt` implementing `XxxRepository`
2. Change the Koin binding in `di/features/XxxModule.kt`:
   `single<XxxRepository> { RealXxxRepository(get()) }`
3. Nothing else changes. No screen, component, or domain model is touched.

This is a verifiable claim — the `domain/repository/` interfaces are the
stability guarantee. Any change to them is a breaking change that must be
reviewed as a contract change, not an implementation detail.

## String Resources and Internationalisation

All UI strings use `stringResource(Res.string.key)` via the Compose Multiplatform
resources system. Hardcoded strings in composables are treated as build violations.

Resource files live at `commonMain/composeResources/`:
- `values/strings.xml` — English (default, required)
- `values-pt/strings.xml` — Portuguese (minimum objective)
- `values-fr/strings.xml` — French (extended objective)

This is consistent with ADR-004's decision to handle UI translation client-side
and error translation server-side. The `Res.string` type-safe accessors are
generated at build time — a missing translation key is a compile error, not a
runtime crash.

## Dependency Injection: Koin

Koin is the DI framework for the KMP client. Alternatives considered:

| Option | KMP | Compile-safe | Annotation processing |
|--------|-----|-------------|----------------------|
| Hilt | Android only | ✅ | ✅ (kapt) |
| Dagger | Android only | ✅ | ✅ (kapt) |
| Koin | ✅ | Runtime | ❌ |

Hilt and Dagger are excluded because kapt (Kotlin annotation processing) is
not available on the iOS KMP target. Koin is runtime-verified rather than
compile-time-verified, which is a known tradeoff. The risk is mitigated by:
- Integration tests that boot the Koin graph and verify all bindings resolve
- A flat, explicit module structure (`di/features/XxxModule.kt`) that makes
  missing bindings immediately obvious during development

## Consequences

- Adding a new feature requires creating a new slice directory. The structure
  is a checklist — if any of the four subdirectories is missing, the feature
  is incomplete.
- Navigation logic lives exclusively in `DefaultRootComponent`. Composables
  never call navigation APIs directly. This is verifiable by searching for
  `navigation.push` — it must only appear in one file.
- The `Effects` channel pattern means navigation cannot be triggered by
  recomposition. Every navigation action has a clear, traceable cause: an
  event fired by user interaction.
- Replacing any repository mock with a real implementation is a one-line
  change in one DI module. This is the testability guarantee.
