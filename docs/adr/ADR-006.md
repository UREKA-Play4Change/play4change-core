# ADR-006 — AI Agent: gRPC Service → Gradle Module Boundary

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind

---

## Context

The original architecture (ADR-002) designed the AI agent as a separate deployed
service — specifically a Cloudflare Workers TypeScript deployment. This required
a network boundary between the Spring Boot server and the AI agent, for which
gRPC was the correct choice.

The AI stack was subsequently changed to LangChain4j + Mistral running on the
JVM alongside the Spring Boot server. This eliminated the network boundary
between server and AI logic.

Two questions arose:
1. Should gRPC be kept even though both ends now run in the same JVM?
2. How should the AI agent be structured to remain provider-agnostic (not tied to Mistral)?

---

## Decision

**Remove gRPC. Replace with a Gradle module boundary using the Provider pattern.**

### Module Structure

```
:ai-agent:api       ← interfaces + data models only (no LangChain4j)
:ai-agent:langchain ← LangChain4j + Mistral implementation
:server             ← depends on :ai-agent:api only
```

The server never imports anything from `:ai-agent:langchain`. It only knows
about the interface defined in `:ai-agent:api`. The implementation is injected
by Spring at runtime.

### The Contract

```kotlin
// :ai-agent:api — the only thing :server knows about
interface TaskGenerationPort {
    suspend fun generateTasks(request: GenerationRequest): Either<AppError, GenerationResult>
    suspend fun generateAdaptiveBranch(context: StruggleContext): Either<AppError, AdaptiveBranch>
    suspend fun healthCheck(): Boolean
}
```

### Swapping Providers

To switch from Mistral to OpenAI tomorrow:
1. Create `:ai-agent:openai` module
2. Implement `TaskGenerationPort`
3. Change one line in `settings.gradle.kts`
4. Zero changes in `:server`

---

## Why gRPC Was Removed

| Concern | gRPC (same JVM) | Module boundary |
|---------|----------------|-----------------|
| Network overhead | ❌ Serialise + deserialise in same process | ✅ Direct function call |
| Complexity | ❌ Proto compilation, stubs, channels | ✅ Just an interface |
| Debugging | ❌ gRPC tooling required | ✅ Standard Kotlin debugging |
| Correctness | ❌ Wrong abstraction for same-JVM | ✅ Right abstraction |

Calling gRPC between two modules in the same JVM is the wrong abstraction.
The overhead of serialization, deserialization, and network stack exists
without any of the benefits (different process, different language, independent scaling).

---

## Provider Pattern — LLM Agnosticism

The original requirement to not be tied to Mistral is achieved through the
Provider pattern rather than through a network boundary.

```
TaskGenerationPort (interface in :ai-agent:api)
        │
        ├── LangChain4jMistralAdapter  (:ai-agent:langchain) ← current
        ├── LangChain4jOpenAIAdapter   (:ai-agent:openai)    ← future
        └── LangChain4jGeminiAdapter   (:ai-agent:gemini)    ← future
```

Each adapter is a complete, independently testable implementation.
Switching providers is a configuration change, not a code change in business logic.

---

## Future: Reintroducing the Network Boundary

If the AI agent needs to scale independently — for example if LLM inference
becomes the bottleneck at 100k+ users — the module boundary is already
the correct seam for extraction.

Migration path:
1. Extract `:ai-agent:langchain` into a separate deployable service
2. Create `:ai-agent:grpc-client` that implements `TaskGenerationPort`
   by calling the remote service via gRPC
3. Swap the Spring bean — zero changes in `:server`

The interface designed today becomes the service contract tomorrow.
The module boundary today becomes the network boundary when scale demands it.

## Historical Artifacts

The original proto definition is retained in `docs/proto/task_generation.proto`
as a demonstration of the original gRPC design intent and as documentation
of the architectural evolution.

The Cloudflare Workers TypeScript `ai-agent/` folder has been removed from
the active codebase and its contents archived in `docs/archive/cloudflare-agent/`.

---

## Consequences

- `:server` has a compile-time dependency on `:ai-agent:api` only
- `:server` has zero knowledge of LangChain4j, Mistral, or any LLM library
- Adding or switching LLM providers requires no changes to server business logic
- The `TaskGenerationPort` interface is the stability guarantee — it must be
  versioned carefully as it is the contract between two modules
- Integration tests mock `TaskGenerationPort` — unit tests in `:ai-agent:langchain`
  test the actual LLM calls in isolation
