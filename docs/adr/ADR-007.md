# ADR-007 — AI Content Generation Pipeline

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-003 (Batch vs On-Demand), ADR-005 (Adaptive Architecture), ADR-006 (Module Boundary)

---

## Context

The platform requires two distinct categories of AI-generated educational content,
each with fundamentally different generation characteristics:

1. **Fixed-path tasks** — multiple-choice tasks pre-generated for every module in
   every active course. Predictable volume, cacheable, latency-insensitive.

2. **Adaptive subtasks** — personalised remediation branches generated when a
   struggling user is detected. Unpredictable trigger, latency-tolerant (user
   already paused), volume low but highly context-sensitive.

Both categories share the same LLM provider and embedding model, but differ in
trigger, prompt strategy, deduplication approach, and reuse logic. A single
unified generation pipeline would entangle concerns that have different
operational lifecycles and failure modes. A completely separate pipeline for
each would duplicate infrastructure.

The challenge: design a generation architecture that serves both content
categories cleanly, is provider-agnostic (not coupled to Mistral), degrades
gracefully when the LLM is unavailable, and accumulates a reuse library over
time to reduce long-term inference costs.

---

## Decision

**A single port interface with two distinct operation contracts, backed by a
prompt-separated adapter that routes to different generation strategies per
content type.**

The implementation is structured across four layers with strict dependency
directionality:

```
:ai-agent:api      ←  interfaces + domain models only
      ↑
:ai-agent:langchain ←  LangChain4j + Mistral implementation
      ↑
:server             ←  business logic; depends on :ai-agent:api only
```

The `TaskGenerationPort` interface (in `:ai-agent:api`) defines two generation
operations and a health check. The `LangChain4jTaskGenerationAdapter` (in
`:ai-agent:langchain`) is the only class in the entire codebase that imports
LangChain4j or knows about Mistral. The server calls the port and nothing else.

---

## The Port Contract

```kotlin
interface TaskGenerationPort {

    /**
     * Generate tasks for a fixed course learning path.
     * Called by the nightly batch job and on course creation.
     *
     * @return tasks with embeddings for pgvector deduplication
     */
    suspend fun generateTasks(
        request: GenerationRequest
    ): Either<AppError, GenerationResult>

    /**
     * Generate an adaptive branch for a struggling user.
     * Called on-demand by StruggleDetectionService.
     *
     * Implementation MUST apply similarity-based reuse before generating:
     *   similarity > 0.90 → reuse existing branch
     *   similarity > 0.65 → merge existing + generate complement
     *   similarity < 0.65 → generate fresh, store for future reuse
     */
    suspend fun generateAdaptiveBranch(
        context: StruggleContext
    ): Either<AppError, AdaptiveBranch>

    /** Health check — used by circuit breaker and Actuator health indicator. */
    suspend fun healthCheck(): Boolean
}
```

The port returns `Either<AppError, T>` on every operation. Errors are typed
domain objects — infrastructure exceptions never cross the module boundary.
See ADR-001 for the error handling rationale.

## Fixed-Path Task Generation

### Trigger

A Spring `@Scheduled(cron = "0 0 2 * * *")` batch job runs nightly at 02:00.
It iterates over all active courses and calls `generateTasks()` per module.
The same operation is also called synchronously on course creation to seed
initial content immediately.

### Request Model

```kotlin
data class GenerationRequest(
    val courseId: String,
    val moduleId: String,
    val subjectDomain: String,       // e.g. "sustainability", "digital_literacy"
    val audienceLevel: AudienceLevel,
    val language: String,            // ISO 639-1
    val taskCount: Int,
    val moduleObjective: String,     // drives prompt context
    val existingEmbeddings: List<FloatArray> = emptyList()
)
```

`existingEmbeddings` is populated on incremental runs so the prompt can
instruct the model to generate content that complements what already exists.

### Prompt Architecture

Two static objects handle prompt construction — neither knows about LangChain4j
or Mistral, making them independently testable.

**`TaskGenerationPrompt.system()`** establishes the model's role and enforces
the output schema. Key constraints injected at the system level:

- Output must be a valid JSON array. No preamble, no markdown fences.
- Exactly four options per task; correct answer always at index 0.
- Wrong options must be plausible — not obviously incorrect.
- Difficulty must match `audienceLevel` exactly.
- Tasks must be completable in 5–15 minutes.

**`TaskGenerationPrompt.user(request)`** injects the runtime context:
domain, module objective, audience level, language, task count, and a
conditional instruction when existing embeddings are present:

```
"Important: N tasks already exist for this module.
 Generate semantically distinct tasks that complement the existing content."
```

### Response Parsing and Hardening

The model response is processed in `parseTasksFromJson()`. LLMs frequently
wrap JSON output in markdown fences despite explicit instructions. The parser
strips these defensively before deserialisation:

```kotlin
val cleaned = rawJson.trim()
    .removePrefix("```json").removePrefix("```")
    .removeSuffix("```").trim()
```

Each element in the parsed array is processed inside `runCatching { }.getOrNull()`.
A malformed task does not abort the batch — valid tasks are persisted, invalid
ones are skipped. This is a deliberate resilience decision: partial content is
better than no content.

### Embedding and Deduplication

After parsing, each task is embedded using the Mistral embedding model:

```kotlin
val embedding = embeddingModel.embed("$title $description").content().vector()
val isDuplicate = deduplicationService.isDuplicate(embedding, moduleId)
```

`PgVectorDeduplicationService.isDuplicate()` executes a cosine distance query
scoped to the module:

```sql
SELECT 1 FROM task_templates
WHERE module_id = ?
  AND 1 - (embedding <=> ?::vector) > ?
LIMIT 1
```

Tasks above the similarity threshold (`ai.dedup.full-reuse-threshold`, default
0.90) are marked `GenerationStatus.DUPLICATE` and excluded from persistence.
This prevents content fatigue from accumulating across nightly batch runs.

### Result Model

```kotlin
data class GenerationResult(
    val tasks: List<GeneratedTask>,
    val metadata: GenerationMetadata
)

data class GenerationMetadata(
    val tasksRequested: Int,
    val tasksGenerated: Int,
    val tasksDeduplicated: Int,
    val tokensUsed: Long,
    val generationTimeMs: Long,
    val providerName: String
)
```

`GenerationMetadata` is logged at INFO level after every batch run and drives
the Grafana dashboard. It answers operational questions: how many tasks were
actually generated vs requested? How many were deduplicated? What was the
token cost? Which provider was used?

### Observability

Generation duration is measured with a Micrometer `Timer`:

```kotlin
val timer = Timer.start(meterRegistry)
// ... generation ...
val durationMs = timer.stop(
    meterRegistry.timer("ai.generation.duration", "type", "fixed_path")
) / 1_000_000
```

Failure counter incremented on any exception:

```kotlin
meterRegistry.counter("ai.generation.failures", "type", "fixed_path").increment()
```

Deduplication counter incremented per skipped task:

```kotlin
meterRegistry.counter("ai.dedup.duplicates_skipped", "module_id", moduleId).increment()
```

These three metrics are sufficient to answer: is the batch running? Is it
failing? Is it generating fresh content or mostly hitting duplicates?

## Adaptive Branch Generation

### Trigger

`generateAdaptiveBranch()` is called on-demand by `StruggleDetectionService`
when a user struggle is detected. The trigger conditions are configurable per
course:

- User fails the same task more than `N` consecutive attempts
- User explicitly requests help
- Task window expires without any attempt

### Context Model

```kotlin
data class StruggleContext(
    val userId: String,
    val taskId: String,
    val moduleId: String,
    val courseId: String,
    val subjectDomain: String,
    val audienceLevel: AudienceLevel,
    val language: String,
    val attemptCount: Int,
    val errorPattern: ErrorPattern,
    val moduleObjective: String,
    val taskDescription: String
)

enum class ErrorPattern {
    CONCEPTUAL_MISUNDERSTANDING,
    PROCEDURAL_ERROR,
    KNOWLEDGE_GAP,
    UNCLEAR_INSTRUCTIONS,
    UNKNOWN
}
```

`ErrorPattern` is the diagnostic classification of the struggle. It is the
most important field in the context — it determines both the pgvector similarity
query and the prompt strategy. `UNKNOWN` always triggers fresh generation
because the diagnosis is absent.

### Similarity-Based Reuse Pipeline

Before any LLM call, the adapter runs a similarity search against past struggles:

```
1. Embed the struggle context:
   struggleText = "${errorPattern} ${taskDescription} ${subjectDomain}"
   embedding = embeddingModel.embed(struggleText)

2. Query pgvector for similar past struggles:
   SELECT ab.id, 1 - (se.embedding <=> ?::vector) AS similarity
   FROM struggle_events se
   JOIN adaptive_branches ab ON ab.struggle_event_id = se.id
   WHERE ab.status = 'COMPLETED'
   ORDER BY se.embedding <=> ?::vector
   LIMIT 1

3. Apply strategy based on similarity:
   > 0.90  →  FULL_REUSE    (zero LLM cost)
   > 0.65  →  PARTIAL_REUSE (proportional LLM cost)
   < 0.65  →  FRESH_GENERATION (full LLM cost, amortised over future reuses)
```

The thresholds are externalised to `application.yml`:

```yaml
ai:
  dedup:
    full-reuse-threshold: 0.90
    partial-reuse-threshold: 0.65
```

They are initial values. The correct values are empirical and domain-dependent.
Prometheus tracks strategy distribution (`ai.adaptive.reuse` by strategy tag)
so the thresholds can be tuned against real usage data after launch.

### Full Reuse

Similarity above 0.90 indicates the same struggle pattern has been resolved
before. The existing branch is loaded from `adaptive_tasks` and returned
with a new `branchId` UUID (so the new user's progress is tracked
independently) and `reusedFromBranchId` pointing to the source:

```
ReuseStrategy.FULL_REUSE -> {
    meterRegistry.counter("ai.adaptive.reuse", "strategy", "full").increment()
    loadExistingBranch(match.branchId!!, ReuseStrategy.FULL_REUSE)
}
```

Zero LLM inference calls. Zero embedding calls beyond the initial similarity
query. The user receives personalised help in milliseconds.

### Partial Reuse

Similarity between 0.65 and 0.90 indicates partial overlap. The number of
reused tasks is proportional to the similarity score:

```kotlin
val reuseCount = (existing.subtasks.size * match.similarity).toInt()
    .coerceAtLeast(1)
val generateCount = defaultSubtaskCount - reuseCount
```

At similarity 0.80 with 3 default subtasks: 2 are reused, 1 is generated fresh.
At similarity 0.66: 1 is reused, 2 are generated fresh. The blend is continuous,
not stepped.

The fresh complement is generated with `StruggleAnalysisPrompt` (see below)
and appended to the reused tasks. The result is a branch that is partially
grounded in proven remediation content and partially tailored to the new
context.

### Fresh Generation

Similarity below 0.65 means no useful prior exists. The full subtask sequence
is generated:

```
ReuseStrategy.FRESH_GENERATION -> {
    meterRegistry.counter("ai.adaptive.reuse", "strategy", "fresh").increment()
    generateFreshBranch(context, struggleEmbedding)
}
```

The generated branch is intended to be stored with its struggle embedding for
future reuse (see Known Gaps below).

### Adaptive Prompt Architecture

**`StruggleAnalysisPrompt.system()`** establishes a different model persona
from task generation — an "adaptive learning coach" rather than a "content
designer". The system prompt emphasises scaffolding toward success, not
generating standalone tasks:

- Start with the simplest related concept
- Each subtask builds on the previous
- The final subtask returns the student to the original task level
- Points reward is lower (5–50) than main tasks (10–100) to reflect smaller scope
- Framing must be encouraging — not remediation language

**`StruggleAnalysisPrompt.user(context, subtaskCount)`** injects the full
diagnostic context. Critically, `ErrorPattern` is expanded to a natural
language prompt instruction specific to each pattern:

| ErrorPattern | Prompt directive |
|---|---|
| `CONCEPTUAL_MISUNDERSTANDING` | "Start by re-explaining the concept differently" |
| `PROCEDURAL_ERROR` | "The student understands but applies wrong steps. Focus on procedure." |
| `KNOWLEDGE_GAP` | "Fill the prerequisite gap before returning to the task." |
| `UNCLEAR_INSTRUCTIONS` | "Provide clearer, more explicit guidance." |
| `UNKNOWN` | "Provide general scaffolding from basic to intermediate." |

This is the key distinction between adaptive and fixed-path generation: the
prompt carries a diagnosis and a remediation strategy, not just a domain and
level. The model is directed toward a specific therapeutic path, not asked
to generate generically on a topic.

### Branch Result Model

```kotlin
data class AdaptiveBranch(
    val branchId: String,
    val subtasks: List<GeneratedTask>,
    val reuseStrategy: ReuseStrategy,
    val reusedFromBranchId: String?  // null if freshly generated
)
```

`reuseStrategy` and `reusedFromBranchId` are persisted alongside the branch.
This creates an audit trail of how content was sourced — observable from
Grafana and useful for understanding content quality over time.

## Graceful Degradation — The Three-Layer Fallback

A core design requirement is that the application must run without an active
LLM provider. Three layers enforce this:

**Layer 1 — Conditional bean activation.**
`LangChain4jConfig` is annotated with `@ConditionalOnExpression` that evaluates
the `ai.mistral.api-key` property. If the key is absent or set to `not-set-yet`,
the entire LangChain4j bean graph is never loaded. Spring starts cleanly.

**Layer 2 — NoOp fallback adapter.**
`NoOpTaskGenerationAdapter` is annotated `@ConditionalOnMissingBean(TaskGenerationPort::class)`.
When the real adapter isn't loaded, this activates automatically and returns
`ServiceUnavailable.DependencyUnavailable` for every call. No null checks, no
`if (aiEnabled)` branching anywhere in business logic.

**Layer 3 — Service-level mock fallback.**
`CourseService` folds on the `Either` result from `generateTasks()`. On
`Either.Left` (AI unavailable), it falls back to `mockTask()` — pre-authored
placeholder content that keeps the course creation flow intact. The course is
created, the module is seeded, and the user is unblocked. Mock tasks are
replaced on the next successful nightly batch run.

```kotlin
aiResult.fold(
    ifLeft = { error ->
        log.warn("AI generation failed for dayIndex={} — using mock fallback", dayIndex, error)
        taskTemplateRepository.save(mockTask(module, dayIndex, poolIndex, domain))
    },
    ifRight = { result ->
        // persist real AI tasks
    }
)
```

This fallback chain means: CI pipelines run without secrets, development
environments work without a key, and production degrades to mock content
rather than failing on course creation.

## pgvector — The Shared Infrastructure Layer

`PgVectorDeduplicationService` serves both pipelines and is the load-bearing
component for content quality and cost control.

**Why pgvector over a dedicated vector database (Qdrant, Pinecone, Weaviate):**

At the content volumes of this platform — thousands of tasks and struggle
events, not millions of vectors — pgvector's performance characteristics are
equivalent to a dedicated vector database. Keeping vector storage in the
existing PostgreSQL instance eliminates a deployment component, removes a
network hop, simplifies backup and recovery, and avoids vendor lock-in on a
second infrastructure dependency. The migration path to a dedicated vector
database is well-defined if scale demands it, because the query interface
(`isDuplicate`, `findSimilarStruggle`) abstracts the storage layer.

**The cosine distance operator:**

pgvector's `<=>` operator computes cosine distance. Similarity is derived as
`1 - (embedding <=> ?::vector)`. A result of 1.0 is identical, 0.0 is
orthogonal. All SQL queries using this operator are in `PgVectorDeduplicationService`
via `JdbcTemplate` — not JPQL, which does not support custom operators.
Standard CRUD on entities uses JPA. This split is intentional: the right tool
for each operation.

**The `VectorConverter`:**

Hibernate does not natively understand `vector(384)` as a column type.
`VectorConverter implements AttributeConverter<FloatArray?, String?>` handles
the bidirectional conversion between Kotlin `FloatArray` and the pgvector
string format `[0.1,0.2,...]`. The entity column additionally requires
`@ColumnTransformer(write = "?::vector")` to ensure the string is cast to
the vector type on write. Both are required — the converter handles Java-side
serialisation; the transformer handles the PostgreSQL-side type cast.

## Known Gaps and Planned Improvements

### 1 — Struggle embedding not stored on fresh generation (bug)

`generateFreshBranch()` receives the `struggleEmbedding` parameter but does
not call `deduplicationService.storeStruggleEmbedding()` after branch
persistence. Freshly generated branches are therefore never candidates for
future similarity matching — the reuse library does not grow from fresh
generations.

**Impact:** The self-improving cost model is broken for the most expensive
generation path. Full and partial reuse rates will not increase over time.

**Fix:** After persisting the new branch, call
`deduplicationService.storeStruggleEmbedding(struggleEventId, embedding)`.
This requires adding `struggleEventId` to `StruggleContext` (currently absent).

### 2 — `runBlocking` inside `@Transactional` in CourseService

`CourseService.createCourse()` is a `@Transactional` method that calls
`runBlocking { taskGenerationPort.generateTasks(...) }`. This blocks a thread
from the Spring thread pool for the full duration of the Mistral inference
call (typically 3–30 seconds), holding an open database transaction and
connection for that period.

**Impact:** Under concurrent course creation, this can exhaust the thread pool
and the connection pool simultaneously.

**Fix:** Separate concerns. Save the course and module in one transaction
(commit). Generate tasks outside any transaction. Save generated tasks in
a second transaction. Alternatively, migrate `CourseService` to a coroutine-aware
execution context.

### 3 — No field-length validation on parsed LLM output

The prompt specifies maximum field lengths (title ≤ 60 chars, description ≤ 300
chars, hint ≤ 150 chars). The parser validates structural correctness (required
fields present, correct types) but does not enforce these constraints. The model
occasionally violates them, and over-length content is persisted as-is.

**Fix:** Add explicit truncation or rejection in `parseTasksFromJson()` before
the deduplication check.

### 4 — No circuit breaker on LLM calls

If Mistral is unavailable, every adaptive branch request waits for the full
`timeoutSeconds` (60s default) before returning `Either.Left`. The `healthCheck()`
method is exposed to the Actuator health indicator but is not wired to a
Resilience4j circuit breaker.

**Fix:** Wrap LangChain4j calls with `@CircuitBreaker(name = "mistral")`. The
circuit breaker should fail fast after repeated failures, protecting the
thread pool from accumulating blocked requests. The `NoOpTaskGenerationAdapter`
already provides the correct fallback behaviour once wired.

## Alternatives Considered

### Alternative A — Fully on-demand generation for all content

Rejected. Fixed-path tasks are predictable and domain-static. Generating them
at user request time introduces 1–5 second inference latency on the first
interaction, which is fatal to the engagement loop. Cost scales with user
count rather than domain count — at 10,000 users across 5 courses, on-demand
would make ~10,000 calls per day vs ~50 for batch.

### Alternative B — Fully pre-generated content including adaptive branches

Rejected. Struggle patterns are user-specific and triggered by individual
behaviour — they cannot be predicted and pre-generated. A pre-generated
"remediation library" without per-user diagnostic context would be generic
and less effective than targeted generation. The pgvector reuse strategy
achieves near-equivalent cost characteristics over time without sacrificing
personalisation.

### Alternative C — Single prompt for both content types

Rejected. Fixed-path tasks and adaptive subtasks have different constraints:
different point ranges (10–100 vs 5–50), different structural goals (standalone
vs scaffolded sequence), different personas (content designer vs learning coach),
and different quality criteria (novelty vs diagnostic targeting). Separating
prompts into `TaskGenerationPrompt` and `StruggleAnalysisPrompt` makes each
independently tunable without risking regressions in the other.

### Alternative D — LangChain4j AI Services (declarative interface)

LangChain4j provides an `@AiService` annotation that can generate
implementations from an annotated interface — eliminating the need for manual
`chatModel.generate()` calls and JSON parsing. This was evaluated and rejected
for the current stage: the declarative approach reduces observability and
control over the raw request/response cycle, which is valuable during initial
prompt tuning. Migration to `@AiService` is a viable refactor once prompts
are stable.

## Consequences

- `TaskGenerationPort` is the stability contract between `:server` and the AI
  layer. Any change to its signatures is a breaking change requiring coordinated
  migration across both modules.
- `PgVectorDeduplicationService` is load-bearing for both content quality and
  cost control. It must be covered by integration tests using a real pgvector
  instance (Testcontainers).
- Similarity thresholds (0.90, 0.65) are initial values. Prometheus metrics on
  strategy distribution must inform post-launch tuning.
- Adding a new LLM provider requires only a new implementation of
  `TaskGenerationPort` in a new Gradle module. Zero changes to `:server`.
- Prompt changes are isolated to `TaskGenerationPrompt` and `StruggleAnalysisPrompt`.
  Neither imports any infrastructure dependency, making them unit-testable in isolation.
