# ADR-001 — Result Type: Custom Implementation → Arrow Either

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind

---

## Context

The application needs a type-safe error handling mechanism that avoids exceptions
for expected domain failures. The goal is Railway Oriented Programming — a pipeline
where each step either continues on the happy path or short-circuits with a typed error.

---

## Decision

**Phase 1 — Custom Implementation (initial)**

Built a custom `Result<T, E>` sealed class with:
- `Result.Success<T>` and `Result.Failure<E>`
- `ResultScope` with `bind()`, `ensure()`, `ensureNotNull()`
- `resultWrapper` / `suspendResultWrapper` for coroutine-safe error capture
- `ResultControlFlowException` for internal short-circuit flow

This was built from scratch intentionally to deeply understand the pattern before
adopting a library.

**Phase 2 — Migration to Arrow (production decision)**

Migrated to [Arrow](https://arrow-kt.io/) `Either<E, A>` and `Raise<E>` DSL.

The custom `AppError` interface and sealed error hierarchy are **retained** —
they are domain-specific and Arrow does not provide them.

---

## Rationale

| Concern | Custom | Arrow |
|---------|--------|-------|
| Understanding the pattern | ✅ Built it | — |
| Production maintenance | ❌ Ours to maintain | ✅ Actively maintained |
| Kotlin engineer recognition | ❌ Unknown API | ✅ Industry standard |
| Additional utilities | ❌ Build ourselves | ✅ mapError, recover, zip, etc. |
| KMP support | ✅ | ✅ |

Building custom proved comprehension. Shipping with Arrow proves pragmatism.
Both matter. The sequence matters.

---

## Migration Map

| Custom | Arrow equivalent |
|--------|-----------------|
| `Result<T, E>` | `Either<E, T>` |
| `AppResult<T>` | `Either<AppError, T>` |
| `Result.Success(data)` | `Either.Right(data)` |
| `Result.Failure(error)` | `Either.Left(error)` |
| `resultWrapper { }` | `either { }` |
| `suspendResultWrapper { }` | `either { }` (suspend-aware) |
| `.bind()` | `.bind()` (same API) |
| `ensure(condition) { error }` | `ensure(condition) { error }` (same) |
| `ensureNotNull(value) { error }` | `ensureNotNull(value) { error }` (same) |
| `.fold(onSuccess, onFailure)` | `.fold(onLeft, onRight)` |

**What stays unchanged:** `AppError` interface, all sealed error subclasses,
`messageKey`, `params`, `httpStatus` — these are domain concepts Arrow has
no opinion on.

---

## Consequences

- All service and repository functions return `Either<AppError, T>`
- Controllers call `.getOrThrow()` extension which bridges to Spring exception handling
- The `common` module retains the `AppError` hierarchy — Arrow is added as a dependency
- Future engineers recognise the pattern immediately without reading custom code
