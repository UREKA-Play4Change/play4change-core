# ADR-011 — Passwordless Authentication: Magic Link + OAuth 2.0 (Google & Facebook)

**Status:** Accepted  
**Date:** April 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-001 (Either error handling), ADR-008 (Client Architecture), ADR-009 (Deployment)

---

## Context

Play4Change needs to authenticate users across Android and iOS without
asking them to create or remember a password. Passwords are the leading
cause of account abandonment in mobile apps — they require the user to
invent a secret, remember it, and type it correctly on a small keyboard.

The application serves learners who may open the app once a day for a few
minutes. That user must not be blocked by credential friction. The
authentication mechanism must be:

- **Frictionless** — minimal steps, no password creation or recovery flows
- **Secure** — resistant to token theft, replay attacks, and session hijacking
- **Stateless-friendly** — works with a JWT-based API that has no server session
- **Multiplatform** — works identically on Android and iOS without platform forks
- **Extensible** — supports multiple identity providers under one user model

No single mechanism satisfies all users. A learner without a Google or
Facebook account needs a fallback. A learner who prefers social login should
not be forced to check their email. The solution must support both.

---

## Decision

**Two complementary passwordless mechanisms, both implemented server-side:**

| Mechanism | Flow | Best for |
|-----------|------|----------|
| Magic Link | Email → one-time token → JWT | Users without social accounts |
| OAuth 2.0 | Provider ID token → server verification → JWT | Users with Google or Facebook |

Both paths converge at the same output: a `TokenPair` (access token + refresh
token) issued by the Play4Change server. The client always talks to our server —
it never stores provider tokens or calls provider APIs directly.

## How Authentication Works — The Mental Model

Before describing the implementation, the underlying concepts must be clear.

### What a JWT is

A **JSON Web Token (JWT)** is a self-contained string that encodes claims about
a user — their ID, role, and when the token expires. It has three parts separated
by dots:

```
header.payload.signature
```

- **Header** — algorithm used to sign the token (e.g. `HS256`)
- **Payload** — the claims: `{ "sub": "user-id", "role": "USER", "exp": 1234567890 }`
- **Signature** — a cryptographic hash of header + payload, signed with a secret key

The server can verify any JWT it receives by recomputing the signature. If it
matches, the token is authentic. If it does not, the token was tampered with.
The server never needs to look up the token in a database — verification is
purely mathematical. This is what makes JWTs stateless.

### Why two tokens

A single JWT is valid until it expires. If stolen, it grants access until
expiry. Two-token rotation solves this:

- **Access token** — short-lived (15 minutes). Used on every API call.
  If stolen, the attacker has a 15-minute window.
- **Refresh token** — long-lived (7 days). Stored securely. Used only to
  obtain a new access token when the current one expires.

The refresh token is stored in the database so it can be revoked. The
access token is not stored — it is verified cryptographically.

### Token family reuse detection

When a refresh token is used, it is immediately marked as used and replaced
with a new one. If an old (used) refresh token is ever presented again, it
means a token was stolen — the legitimate user and the attacker are both trying
to use tokens from the same family. The entire family is revoked, forcing
re-authentication. This is the industry-standard approach to refresh token
theft detection.

## Mechanism 1 — Magic Link

### What it is

The user provides only their email address. The server sends a one-time link
to that address. Clicking the link authenticates the user. No password is
ever created, stored, or transmitted.

### Why it works

Email is already a proof of identity — if you can receive email at an address,
you control that address. A magic link delegates identity verification to the
email provider (Gmail, Outlook, etc.) rather than asking the user to invent
a new credential.

### The flow, step by step

```
1. User enters email in the app
        │
        ▼
2. POST /auth/magic-link
   Server generates a cryptographically random 32-byte token
   Server stores: SHA-256(token) in magic_link_tokens table
                  (raw token is NOT stored — only its hash)
   Server sets expiry: now + 15 minutes
        │
        ▼
3. Server sends email via Resend API:
   "Click here to sign in: https://app.play4change.com/auth/verify?token=<raw_token>"
        │
        ▼
4. User clicks the link in their email client
        │
        ▼
5. GET /auth/verify?token=<raw_token>
   Server computes SHA-256(raw_token)
   Server looks up the hash in magic_link_tokens
   Server checks: not expired, not already used
   Server marks the token as used (one-time use enforced)
        │
        ▼
6. Server finds or creates the User record by email
   Server issues TokenPair (access token + refresh token)
   Server returns TokenPair to client
        │
        ▼
7. Client stores tokens. User is authenticated.
```

### Why the token is hashed before storage

If the database is compromised, an attacker would find only SHA-256 hashes —
not the raw tokens. SHA-256 is a one-way function: given the hash, you cannot
recover the raw token. The raw token travels only in the email link and in the
single verify request. It is never persisted.

This is the same principle as password hashing — you never store what you
verify; you store a one-way transformation of it.

### Why 15-minute expiry

A magic link is only useful to someone who has access to the email inbox at
that moment. The 15-minute window is short enough to limit the exposure if
the email is forwarded or the link is visible in a notification — and long
enough for the user to click it without frustration.

### Email provider: Resend

Magic links require a transactional email service. The application uses
[Resend](https://resend.com) — a REST API for sending email. The choice
over SMTP or Spring Mail is deliberate:

| Concern | SMTP / Spring Mail | Resend |
|---------|--------------------|--------|
| Setup | Requires SMTP server config, credentials | Single API key |
| Reliability | Dependent on SMTP server uptime | Managed delivery infrastructure |
| Deliverability | Depends on sender reputation | Resend manages SPF/DKIM/DMARC |
| Dependencies | `spring-boot-starter-mail` (JavaMail) | Plain HTTP POST |
| Dev fallback | Requires real SMTP or MailHog | `ConsoleEmailAdapter` when key absent |

The `ResendEmailAdapter` is a `@Primary` bean active when `RESEND_API_KEY` is
set. The `ConsoleEmailAdapter` is a `@ConditionalOnMissingBean(EmailPort::class)`
fallback that prints magic links to stdout during local development — no email
infrastructure needed to develop or test the flow.

```kotlin
// ResendEmailAdapter — production
@Component
@Primary
@ConditionalOnProperty(name = ["resend.api-key"], matchIfMissing = false)
class ResendEmailAdapter(private val props: ResendProperties) : EmailPort {
    fun send(to: String, subject: String, html: String) {
        // POST https://api.resend.com/emails
        // Authorization: Bearer ${props.apiKey}
        // Body: { "from": props.from, "to": to, "subject": subject, "html": html }
    }
}

// ConsoleEmailAdapter — local development fallback
@Component
@ConditionalOnMissingBean(EmailPort::class)
class ConsoleEmailAdapter : EmailPort {
    fun send(to: String, subject: String, html: String) {
        log.info("MAGIC LINK → $to\n$html")
    }
}
```

## Mechanism 2 — OAuth 2.0 (Google + Facebook)

### What OAuth 2.0 is — the actual problem it solves

OAuth 2.0 solves a specific problem: **how does a third-party application
(Play4Change) verify that a user is who they claim to be, without the user
giving Play4Change their Google or Facebook password?**

The answer is delegation. Google and Facebook act as identity providers (IdP).
They authenticate the user using their own credentials. They then issue a
signed certificate — an **ID token** — that says:
"We, Google, confirm that the user with email X and ID Y authenticated with us
at time Z. Here is a signed token proving this."

Play4Change verifies the signature on that certificate. If valid, we trust
the identity claim without ever seeing the user's Google or Facebook password.

### The flow, step by step

```
1. User taps "Continue with Google" in the app
        │
        ▼
2. The mobile OS opens Google's authentication screen
   (This happens on the device — Play4Change server is not involved yet)
        │
        ▼
3. User authenticates with Google
   Google issues an ID token to the mobile app
   The ID token is a signed JWT containing: email, name, sub (Google user ID),
   issued-at, expiry, audience (our Google Client ID)
        │
        ▼
4. POST /auth/oauth
   Body: { "provider": "GOOGLE", "idToken": "<Google's JWT>" }
        │
        ▼
5. Server verifies the ID token — this is the critical step
   (detailed below for each provider)
        │
        ▼
6. Server finds or creates the User record by (provider, providerId)
   Server issues our own TokenPair (access + refresh)
   Server returns TokenPair to client
        │
        ▼
7. Client stores our tokens. User is authenticated.
   The Google ID token is discarded — we never store it.
```

### Why the server must verify the token (not just trust it)

If the client simply sent `{ "email": "user@gmail.com" }` and the server
believed it, any attacker could claim to be any user by typing any email.
The ID token is Google's cryptographic proof that the claim is genuine.

Verifying the ID token is non-negotiable. Without it, OAuth provides no
security — only the illusion of it.

### Google: JWKS Signature Verification

Google signs its ID tokens using RSA private keys. It publishes the
corresponding public keys at a well-known endpoint (JWKS — JSON Web Key Set):

```
https://www.googleapis.com/oauth2/v3/certs
```

Verification steps:

```
1. Extract the key ID (kid) from the ID token's header
   The kid tells us which of Google's public keys to use

2. Fetch Google's JWKS (cached for 1 hour to avoid hammering Google's servers)

3. Find the public key matching the kid in the JWKS response

4. Reconstruct the RSA public key from the JWK's n (modulus) and e (exponent)
   n and e are base64url-encoded big integers

5. Use JJWT to verify the ID token's signature using the reconstructed public key
   If the signature is invalid → reject (token was tampered with)
   If the signature is valid → proceed

6. Validate claims:
   - iss (issuer) must be "https://accounts.google.com" or "accounts.google.com"
   - aud (audience) must contain our GOOGLE_CLIENT_ID
   - exp (expiry) is validated automatically by JJWT

7. Extract: sub (Google's user ID), email, name
```

**Why not use Google's tokeninfo endpoint?**

Google provides a shortcut: `GET https://oauth2.googleapis.com/tokeninfo?id_token=...`
It validates the token and returns the claims. This is the "lazy" approach and
it was the previous implementation.

The problem: it makes a network call to Google on every login. If Google's
tokeninfo endpoint is slow or unavailable, our login is broken. JWKS verification
is entirely local after the initial key fetch — the only network dependency is
the hourly JWKS refresh, which is cached. One slow network call every hour is
strictly better than one slow network call per login.

```kotlin
// GoogleOAuthAdapter — key verification logic
class GoogleOAuthAdapter(
    private val googleProperties: GoogleProperties,
    private val objectMapper: ObjectMapper
) : OAuthVerifierPort {

    @Volatile private var jwksCache: Pair<Long, String>? = null
    private val JWKS_URL = "https://www.googleapis.com/oauth2/v3/certs"

    override fun verify(provider: AuthProvider, idToken: String): OAuthClaims {
        val kid = extractKid(idToken)            // from JWT header (unverified)
        val publicKey = findPublicKey(kid)       // from JWKS cache
        val claims = Jwts.parser()
            .keyLocator { _ -> publicKey }       // JJWT verifies signature here
            .build()
            .parseSignedClaims(idToken)
            .payload
        // validate iss, aud, exp (exp validated by JJWT automatically)
        return OAuthClaims(email = claims["email"], name = claims["name"], sub = claims.subject)
    }
}
```

### Facebook: Graph API Verification

Facebook does not use JWKS. Facebook's ID tokens are opaque access tokens,
not signed JWTs. The correct verification approach is different:

```
1. Call the Facebook Graph API with the access token:
   GET https://graph.facebook.com/me?fields=id,name,email&access_token={token}

2. If Facebook returns a user object → token is valid and belongs to that user
   If Facebook returns an error → token is invalid or expired → reject

3. Extract: id (Facebook's user ID), name, email
```

This is the correct approach for Facebook. There is no signature to verify
locally — we trust Facebook's Graph API as the authority on whether a token
is valid. The call is direct and simple: if Facebook says the token is good,
it is good.

## Role-Based Access Control

The authentication system supports two roles: `USER` and `ADMIN`.

The role is:
1. **Stored** in the `users` table (`role VARCHAR(10) DEFAULT 'USER'`)
2. **Embedded** in the JWT as a `"role"` claim when the token is issued
3. **Read** from the JWT by `JwtAuthFilter` on every request — no database
   lookup required
4. **Propagated** through refresh token rotation — the role stored on the
   refresh token is re-embedded when a new access token is issued

```
JWT payload example (admin user):
{
  "sub": "user-uuid",
  "email": "admin@play4change.app",
  "role": "ADMIN",
  "iat": 1712345678,
  "exp": 1712346578
}
```

Spring Security translates the role claim into a `GrantedAuthority`:

```kotlin
// JwtAuthFilter
val (userId, role) = tokenService.parseAccessToken(token)
SecurityContextHolder.getContext().authentication =
    UsernamePasswordAuthenticationToken(
        userId,
        null,
        listOf(SimpleGrantedAuthority("ROLE_$role"))
    )
```

Route protection in `SecurityConfig`:

```
.requestMatchers("/admin/**").hasRole("ADMIN")
.anyRequest().authenticated()
```

An `ADMIN` user logs in through the same magic link or OAuth flow as any
other user. The role is set directly in the database — there is no separate
admin registration endpoint. This is intentional: admin access is a
privilege granted out-of-band (by a database operator), not self-service.

## Token Lifecycle

```
Magic Link / OAuth Login
        │
        ▼
TokenPair issued:
  access_token  (JWT, 15 min, not stored in DB)
  refresh_token (opaque, 7 days, stored as SHA-256 hash in refresh_tokens)
        │
┌───────┴────────────────────────────────────┐
│                                            │
▼                                            ▼
Access token valid                    Access token expired
Use on every API call                         │
                                              ▼
                                   POST /auth/refresh
                                   Body: { refreshToken: "..." }
                                             │
                                   Server verifies:
                                   - SHA-256(token) matches DB record
                                   - not expired
                                   - not already used
                                             │
                                   ┌─────────┴─────────────────┐
                                   │                           │
                                   ▼                           ▼
                              Valid token              Token already used
                              Mark old as used         → Revoke entire family
                              Issue new TokenPair      → Force re-login
                              Return to client         (theft detected)
```

## Database Schema

```sql
-- One-time use tokens for magic link
CREATE TABLE magic_link_tokens (
    id          VARCHAR(36)  PRIMARY KEY DEFAULT gen_random_uuid()::text,
    token       VARCHAR(255) NOT NULL UNIQUE,   -- SHA-256 hash, not raw token
    email       VARCHAR(255) NOT NULL,
    expires_at  TIMESTAMPTZ  NOT NULL,
    used        BOOLEAN      NOT NULL DEFAULT false,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_magic_link_token ON magic_link_tokens(token);

-- Refresh tokens for session continuation
CREATE TABLE refresh_tokens (
    id          VARCHAR(36)  PRIMARY KEY DEFAULT gen_random_uuid()::text,
    token_hash  VARCHAR(255) NOT NULL UNIQUE,   -- SHA-256 hash
    user_id     VARCHAR(36)  NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    family_id   VARCHAR(36)  NOT NULL,          -- for reuse detection
    role        VARCHAR(10)  NOT NULL DEFAULT 'USER',  -- propagated on refresh
    expires_at  TIMESTAMPTZ  NOT NULL,
    used        BOOLEAN      NOT NULL DEFAULT false,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_refresh_token_hash   ON refresh_tokens(token_hash);
CREATE INDEX idx_refresh_family_id    ON refresh_tokens(family_id);
CREATE INDEX idx_refresh_user_id      ON refresh_tokens(user_id);

-- Users — unified table for all auth providers
CREATE TABLE users (
    id                VARCHAR(36)  PRIMARY KEY DEFAULT gen_random_uuid()::text,
    email             VARCHAR(255) NOT NULL UNIQUE,
    name              VARCHAR(255),
    avatar_url        TEXT,
    role              VARCHAR(10)  NOT NULL DEFAULT 'USER',
    provider          VARCHAR(20)  NOT NULL,   -- MAGIC_LINK | GOOGLE | FACEBOOK
    provider_id       VARCHAR(255),
    preferred_language VARCHAR(10) NOT NULL DEFAULT 'en',
    audience_level    VARCHAR(20)  NOT NULL DEFAULT 'BEGINNER',
    created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
    UNIQUE (provider, provider_id)
);
```

## Alternatives Considered

### Option A — Password-based authentication (username + password)

The classic approach: user creates a password, server stores a bcrypt hash,
login verifies against the hash.

**Pros:**
- No external dependencies — works without email or social accounts
- Universally understood by developers

**Cons:**
- Password creation adds friction at registration — users abandon flows
- Password recovery requires email anyway — making email mandatory while adding complexity
- bcrypt storage, reset flows, brute-force protection must all be implemented
- Passwords are the #1 cause of account breaches (credential stuffing, reuse)
- Delivers worse security and worse UX than the alternatives

**Verdict: Rejected.** Passwords solve a problem the app does not have
(offline identity) while introducing problems it does not need.

---

### Option B — Firebase Authentication

Google's managed auth service. Handles magic links, Google sign-in, Facebook
sign-in, and many more providers with a single SDK.

**Pros:**
- Extremely fast to integrate — weeks of auth work reduced to days
- Managed token verification, refresh, and revocation
- SDKs for Android, iOS, and web

**Cons:**
- **Vendor lock-in** — all user identity lives in Firebase. Migration is painful
  and partial (Firebase does not export password hashes in a usable format)
- **Loss of control** — the user table is Firebase's, not ours. Custom fields
  (role, audienceLevel, preferredLanguage) require a parallel database anyway
- **Cost at scale** — Firebase Authentication is free up to 10,000 MAU, then
  $0.0055/MAU. At 1,000,000 users: ~$5,500/month for authentication alone
- **Educational value** — implementing auth from first principles demonstrates
  understanding of the security model. Using Firebase abstracts exactly the
  concepts that must be understood

**Verdict: Rejected.** The cost of lock-in and loss of control outweigh the
integration speed advantage for a platform targeting institutional deployment.

---

### Option C — Supabase Auth

Open-source Firebase alternative with self-hosted option.

**Pros:**
- Self-hostable — no vendor lock-in if self-hosted
- Supports magic links and OAuth out of the box
- Postgres-based — compatible with the existing stack

**Cons:**
- Still an external service dependency even when self-hosted (another container to manage)
- Auth schema is Supabase's — integrating custom user fields (role, audienceLevel)
  requires working around their schema or duplicating users into our own table
- The self-hosted deployment adds operational complexity for what is ultimately
  a moderately complex but well-understood problem

**Verdict: Rejected.** The custom user model requirements make Supabase's auth
schema a poor fit. Managing the impedance mismatch between Supabase's user
table and ours would be more complex than implementing the auth layer directly.

---

### Option D — Keycloak (enterprise SSO)

The standard enterprise identity provider with OIDC/SAML support.

**Pros:**
- Feature-complete — MFA, federation, LDAP/AD integration
- Used by institutions (universities often have Keycloak or Active Directory)
- Well-documented for Spring Boot integration via `spring-security-oauth2-resource-server`

**Cons:**
- Severe overengineering for current scale
- Keycloak itself requires 512MB–1GB RAM minimum — significant for a deployment
  where the entire application runs in 512MB
- OIDC integration complexity is substantial for a mobile-first app that needs
  custom token claims
- No simpler path for magic links — would still require custom implementation

**Verdict: Rejected.** The correct decision for an institution network at scale.
The wrong decision for the current deployment stage. Documented here as the
upgrade path when multi-institution SSO becomes a requirement.

## What Was Implemented — Summary

| Component | File | Decision |
|-----------|------|---------|
| Magic link token generation | `MagicLinkService.kt` | SHA-256 hash stored; raw token in email only |
| Magic link email delivery | `ResendEmailAdapter.kt` | Resend REST API; `ConsoleEmailAdapter` as dev fallback |
| Google ID token verification | `GoogleOAuthAdapter.kt` | JWKS local verification with 1h cache; not tokeninfo endpoint |
| Facebook token verification | `FacebookOAuthAdapter.kt` | Graph API call — correct for opaque tokens |
| JWT issuance | `TokenService.kt` | access (15m) + refresh (7d); role embedded as claim |
| Token refresh + reuse detection | `TokenService.kt` | Family-based revocation on reuse |
| Route protection | `SecurityConfig.kt` | `/admin/**` requires `ROLE_ADMIN` |
| Role propagation | `JwtAuthFilter.kt` | `ROLE_$role` authority from JWT claim; no DB lookup |
| Local dev fallback | `ConsoleEmailAdapter.kt` | `@ConditionalOnMissingBean(EmailPort::class)` |

## Configuration

All sensitive values come from environment variables. No secrets are hardcoded.

```yaml
# application.yml
jwt:
  secret: ${JWT_SECRET}          # HMAC-SHA signing key — min 256 bits
  access-ttl-minutes: 15
  refresh-ttl-days: 7

resend:
  api-key: ${RESEND_API_KEY:}    # empty → ConsoleEmailAdapter active
  from: ${RESEND_FROM:noreply@play4change.app}

google:
  oauth:
    client-id: ${GOOGLE_CLIENT_ID:}  # must match aud claim in Google ID tokens

app:
  base-url: ${APP_BASE_URL:http://localhost:8080}  # used to construct magic link URLs
```

## Security Properties

| Property | How it is achieved |
|----------|--------------------|
| No passwords stored | Magic link delegates to email; OAuth delegates to provider |
| Tokens not stored in plaintext | SHA-256 hash stored; raw token only in transit |
| Short access token lifetime | 15-minute expiry limits stolen token exposure |
| Refresh token theft detection | Family-based reuse detection; entire family revoked on reuse |
| Admin route isolation | Spring Security `hasRole("ADMIN")` at route level |
| No provider token storage | Google/Facebook tokens discarded after verification |
| JWKS local verification | Google tokens verified without Google network call per login |

---

## Consequences

- A user without a Google or Facebook account can still authenticate via magic link
- A user with either social account never needs to remember a credential
- Adding a new OAuth provider (Apple, GitHub) requires: one new `OAuthVerifierPort`
  implementation, one new `AuthProvider` enum value, and one route registration — no
  changes to `OAuthService`, `TokenService`, or any domain model
- Admin access is granted by a database operator updating `users.role` directly —
  there is no self-service admin registration endpoint by design
- The `ConsoleEmailAdapter` fallback means the full auth flow is testable locally
  with no email infrastructure — magic links print to the application log
