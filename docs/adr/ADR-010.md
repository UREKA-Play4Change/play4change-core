# ADR-010 — BaseView Scaffold Upgrade: Edge-to-Edge Insets, Navigation Drawer, and Log-Out Placement

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-008 (Client Architecture: Decompose + MVI + DDD)

---

## Context

Two visual defects and one UX design problem were identified during device
testing that shared a common architectural root.

**Defect 1 — Double gap at screen edges.**
Every screen showed an anomalous gap at the top and bottom — wider than the
intended content padding and inconsistent across devices. The status bar and
navigation bar areas were not being absorbed correctly, producing a visible
stack of: system bar height + hardcoded content padding.

**Defect 2 — Inconsistent window inset handling.**
Because each screen managed its own layout independently, there was no single
place to fix edge-to-edge behaviour. A fix in one screen had no effect on
others. Any future device-specific inset requirement — foldable hinge, floating
keyboard, custom navigation gesture zone — would need to be applied to every
screen individually.

**Design problem — log-out placement.**
The log-out action was placed at the bottom of the Profile screen, after the
badges section. This placement has two flaws. First, it requires scrolling to
the end of a growing list to reach a destructive global action. Second, log-out
is semantically a session-level action — it terminates the entire application
session, not a profile-level setting. Placing it inside a feature screen
conflates session management with user data display.

All three problems had the same architectural cause: `BaseView` was a thin
wrapper around a `Box` with a hardcoded `padding(24.dp)`, delegating all layout
responsibility to individual screens. Screens accumulated compensating
workarounds that masked the root cause without fixing it.

---

## Decision

**Three coordinated changes applied at the `BaseView` layer:**

1. Replace the `Box + padding` implementation with a `Scaffold` that handles
   window insets via `WindowInsets.safeDrawing`, eliminating the double gap
   for all screens simultaneously.

2. Extend `BaseView` with optional `topBar`, `bottomBar`, `drawerState`, and
   `drawerContent` parameters, making it the single structural contract for
   every screen in the application.

3. Move log-out from `ProfileScreen` to a `ModalNavigationDrawer` accessible
   from `HomeScreen`, positioned at the bottom of the drawer panel below all
   navigation items.

---

## The Gap: Root Cause Analysis

Android's edge-to-edge mode (enabled via `enableEdgeToEdge()` in
`MainActivity`) instructs the system to draw application content behind the
status bar and navigation bar. The application is then responsible for
consuming window insets so content is not obscured by system UI.

The previous `BaseView` implementation did not consume insets:

```kotlin
// Previous implementation — incorrect
Box(
    modifier = Modifier
        .fillMaxSize()
        .padding(24.dp),   // hardcoded — not inset-aware
    contentAlignment = Alignment.Center
) {
    content(state, component::onEvent)
}
```

On a device with a 24dp status bar, the visual result was:
- 24dp status bar height (system, not consumable by the app)
- 24dp top padding (hardcoded in BaseView)
- Total visible gap at top: 48dp

`Scaffold` with `contentWindowInsets = WindowInsets.safeDrawing` resolves this
by absorbing the system bar heights into `innerPadding`, which is then applied
to the content area. The content padding becomes exactly what the screen
requests — not the system bar height plus what the screen requests.

```kotlin
// Correct implementation
Scaffold(
    topBar = topBar,
    bottomBar = bottomBar,
    contentWindowInsets = WindowInsets.safeDrawing
) { innerPadding ->
    Box(
        modifier = Modifier
            .fillMaxSize()
            .padding(innerPadding),
        contentAlignment = contentAlignment
    ) {
        content(state, component::onEvent)
    }
}
```

`enableEdgeToEdge()` must be called in `MainActivity` before `super.onCreate()`
for the inset system to function. Without it, `WindowInsets.safeDrawing` returns
zero on all sides and the Scaffold inset handling has no effect.

## The New BaseView Contract

`BaseView` now accepts four optional structural parameters in addition to the
existing component and content arguments:

```kotlin
@Composable
fun <S : ComponentState, E : ComponentEvents> BaseView(
    component: BaseComponent<S, E>,
    topBar: @Composable () -> Unit = {},
    bottomBar: @Composable () -> Unit = {},
    drawerState: DrawerState? = null,
    drawerContent: @Composable (ColumnScope.() -> Unit)? = null,
    contentAlignment: Alignment = Alignment.TopStart,
    content: @Composable (state: S, onEvent: (E) -> Unit) -> Unit,
)
```

**`topBar`** — rendered in the Scaffold's `topBar` slot. Screens that need a
`TopAppBar` pass it here. Screens that do not (Splash, Login) leave it as the
default empty lambda. The Scaffold correctly handles the status bar inset
regardless of whether a TopAppBar is present.

**`bottomBar`** — rendered in the Scaffold's `bottomBar` slot. Content in this
slot is always visible regardless of scroll position. Screens that need a fixed
bottom action (task submission, for example) pass it here. The Scaffold
automatically applies navigation bar inset padding to this slot.

**`drawerState` + `drawerContent`** — when both are non-null, `BaseView` wraps
the Scaffold in a `ModalNavigationDrawer`. When either is null, no drawer is
rendered. This conditional wrapping means the drawer exists only for screens
that need it, without any per-screen branching logic.

**Why optional parameters over subclasses or separate composables:**
Subclassing would require a separate `BaseViewWithDrawer`, `BaseViewWithBottomBar`,
and so on — an explosion of types for what is fundamentally configuration of
the same layout container. Optional parameters with sensible defaults preserve
the single entry point while covering all layout combinations used in the app.

## Navigation Drawer: Log-Out Placement Rationale

Log-out terminates the user's authenticated session and redirects to the login
screen. It is a global action with application-wide consequences.

**Why it does not belong on Profile screen:**
The Profile screen displays user data — progress, badges, course status. It is
a read screen. Placing a destructive session action at the end of a scrollable
list of user data creates two problems: the action is hard to find (requires
scrolling) and is semantically misplaced (session management next to badge
display). A user looking for log-out should not need to scroll past their
earned badges to find it.

**Why it belongs in a navigation drawer on Home:**
The Home screen is the application's persistent root — it is the screen the
user returns to after every other action. A navigation drawer on Home is the
conventional placement for global actions that affect the session or navigate
to top-level destinations. The drawer panel provides dedicated space for
navigation items (Profile, About) with log-out separated at the bottom of the
drawer by a visual divider.

The spatial separation within the drawer communicates the distinction between
navigation (go somewhere) and session action (end the session). Log-out is
styled with `MaterialTheme.colorScheme.error` to reinforce its destructive
nature, consistent with the M3 design system's semantic use of the error colour.

**Drawer structure:**

```
┌─────────────────────────────┐
│  U!REKA                     │
│  PLAY4CHANGE                │
│  [user name]                │
│  ─────────────────────────  │
│  👤  Profile                │
│  ℹ   About                  │
│                             │
│  [space — pushes to bottom] │
│                             │
│  ─────────────────────────  │
│  ⬡  Log out    ← error red  │
└─────────────────────────────┘
```

The `Spacer(Modifier.weight(1f))` between navigation items and log-out ensures
the log-out action is always at the bottom of the drawer regardless of how many
navigation items are added in future — the layout is self-organising, not
hardcoded by pixel offsets.

## Consequences for Existing Screens

Every screen required a review pass to remove padding and layout compensations
that were working around the previous `BaseView` gap.

**Screens updated:**
- `SplashScreen` — removed manual top padding, uses `contentAlignment = Alignment.Center`
- `LoginScreen` — removed hardcoded `Spacing.xxl + Spacing.xl` top padding; now uses
  a single `Spacer(Spacing.xxl)` for intentional breathing room below the status bar
- `HomeScreen` — restructured to pass `topBar`, `drawerState`, and `drawerContent`
  to `BaseView`; log-out logic moved from ProfileComponent to HomeComponent
- `ProfileScreen` — removed `bottomBar` with sign-out button; removed all
  sign-out events, effects, and state fields; the screen is now a pure display screen
- `TaskScreen` — passes `topBar` to `BaseView`; result overlay uses a `Box`
  inside the content lambda, not a wrapping Scaffold
- `AboutScreen`, `ExploreScreen` — each passes a `topBar` with a back arrow;
  no other structural changes required

**What was removed from ProfileComponent:**
`RequestSignOut`, `ConfirmSignOut`, `DismissSignOut` events, `SignedOut` effect,
and `showSignOutDialog` state field. The Profile feature is now entirely
concerned with displaying user data. The session lifecycle concern — log-out —
lives in `HomeComponent` where it belongs.

## Alternatives Considered

### Per-screen Scaffold (status quo before this ADR)

Each screen manages its own `Scaffold` and inset handling. Rejected because:
any systemic layout change requires touching every screen. The gap defect
demonstrated that this approach cannot be fixed without a coordinated change —
individual fixes in individual screens do not address the root cause.

### `systemBarsPadding()` modifier on individual screens

Apply `Modifier.systemBarsPadding()` to each screen's root composable instead
of using a Scaffold. This resolves the gap but does not provide `topBar` or
`bottomBar` slots. Screens that need a fixed bottom action (task submit button,
for example) would need to implement their own fixed-position overlays, which
reintroduces the same fragmentation problem in a different form.

### Separate `BaseViewWithDrawer` composable

Create a second composable for screens that need the drawer. Rejected because
it splits the `BaseView` contract into two types that must be kept in sync.
Every change to the Scaffold behaviour (insets, theming, error dialog) must
be applied twice. The optional parameter approach achieves the same result with
a single composable.

### Floating action button for log-out

Place log-out as a `FloatingActionButton` in the Scaffold's FAB slot on the
Profile screen. Rejected because the FAB is a primary action affordance in
M3 — it is used for the most frequent and constructive action on a screen, not
for a destructive session-termination action. Using it for log-out would
violate the semantic contract of the component.
