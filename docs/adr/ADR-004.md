# ADR-004 ‚Äî Internationalisation Strategy: Hybrid Server + Client

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind

---

## Context

The application targets multiple languages from day one ‚Äî English and Portuguese
as minimum objectives, French and German as extended objectives. Two categories
of translatable content exist with fundamentally different characteristics:

1. **API error messages** ‚Äî dynamic, server-originated, same meaning across all clients
2. **UI copy** ‚Äî static, client-rendered, platform-native (Android strings, iOS .strings)

Choosing a single translation strategy for both would be either over-engineered
or under-engineered depending on which direction you pick.

---

## Decision

**Hybrid approach ‚Äî server-side for errors, client-side for UI.**

| Content | Where translated | Mechanism |
|---------|-----------------|-----------|
| API error messages | Server | Spring `MessageSource` + `Accept-Language` header |
| UI labels, buttons, titles | Client | KMP `expect/actual` + platform-native resources |

---

## Server-Side Translation (API Errors)

**How it works:**
1. Client sends `Accept-Language: pt` header with every request
2. Spring `AcceptHeaderLocaleResolver` extracts locale automatically
3. `AppError` carries `messageKey` (e.g. `error.task.not_found`) and `params`
4. `MessageResolver` resolves `messageKey + params + locale` ‚Üí translated string
5. `GlobalExceptionHandler` returns structured `ErrorResponse`:

```json
{
  "status": 404,
  "message": "Tarefa n√£o encontrada.",
  "code": "error.task.not_found",
  "timestamp": "2026-03-09T10:00:00Z"
}
```

Message bundles in `src/main/resources/i18n/`:
- `messages.properties` ‚Äî English (default)
- `messages_pt.properties` ‚Äî Portuguese
- `messages_fr.properties` ‚Äî French (extended objective)

**Why server-side for errors:**
- Centralised ‚Äî one source of truth for Android, iOS, and any future web client
- Adding a new language requires one `.properties` file, zero app updates
- No client logic needed for error string resolution
- `setUseCodeAsDefaultMessage(false)` ‚Äî fails loudly in dev if a key is missing

---

## Client-Side Translation (UI Copy)

**How it works:**
- Android: `res/values/strings.xml`, `res/values-pt/strings.xml`
- iOS: `Localizable.strings`, `Localizable.strings (pt)`
- Bridged in KMP `:composeApp` via `expect/actual`

**Why client-side for UI:**
- Works fully offline ‚Äî zero network calls for "Submit" or "Daily Tasks"
- Uses the standard approach each platform already provides natively
- Better perceived performance ‚Äî no round trip for static content

---

## Tradeoff Summary

| Concern | Server-side | Client-side |
|---------|-------------|-------------|
| Error messages | ‚úÖ Centralised, consistent | ‚ùå Duplicated per platform |
| UI labels | ‚ùå Needs network, overkill | ‚úÖ Offline, native, fast |
| New language (errors) | Add `.properties` file only | ‚Äî |
| New language (UI) | ‚Äî | Requires app update |

---

## Consequences

- Every new `AppError` subclass must have a corresponding key in all `.properties` files
- New supported language requires coordinated update on both server and client
- The `code` field in `ErrorResponse` is useful for client analytics even when message is translated
