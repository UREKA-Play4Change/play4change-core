# ADR-012 — Topic Content Pipeline: Async Generation, File Storage, and Content Extraction

**Status:** Accepted  
**Date:** April 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-003 (Batch Generation), ADR-006 (AI Module Boundary), ADR-007 (AI Pipeline), ADR-009 (Deployment Strategy)

---

## Context

Phase 3 introduced the admin-facing content pipeline — the flow by which an
administrator creates a topic, provides source material (a URL or a PDF file),
and the system generates a full set of learning tasks from that material via
Mistral AI.

Three independent architectural questions had to be resolved before implementation:

1. **Async design** — task generation via Mistral takes 10–60 seconds. The HTTP
   request cannot wait. What mechanism drives the background work, how is failure
   guaranteed to be captured, and how is the system protected from jobs that
   never finish?

2. **File storage** — when an admin uploads a PDF, where does the file go? The
   extracted text must be stored for re-generation, but the original binary also
   has value as an audit trail and re-extraction source.

3. **Content extraction** — two source types exist: a URL (HTML page) and a PDF
   file. Both must yield clean plain text for the AI generation prompt. What
   libraries are appropriate and how are they structured behind the port boundary?

Each question is documented separately below because the reasoning is distinct,
even though all three were implemented together.

---

## Decision Summary

| Concern | Decision |
|---------|----------|
| Async mechanism | Spring `@Async` with `ThreadPoolTaskExecutor` + `withTimeoutOrNull` |
| Failure guarantee | Top-level try/catch → `FAILED` status; no scheduler needed |
| File storage | MinIO (self-hosted S3-compatible) via AWS SDK v2 |
| PDF extraction | Apache PDFBox 3.x |
| URL extraction | Jsoup |
| Port boundary | `FileStoragePort` + `ContentExtractorPort` — both in `application/port/` |

## Decision 1 — Async Task Generation

### The problem

`TaskGenerationPort.generateTasks()` calls Mistral AI and can take anywhere from
10 to 60 seconds depending on task count and model load. The admin's HTTP request
to `POST /admin/topics` cannot block for that duration — the connection would
time out on mobile networks, and the user experience of waiting a minute for a
response is unacceptable.

The topic must be created immediately (status = `GENERATING`) and the generation
must proceed in the background. When it finishes, the status transitions to
`ACTIVE`. When it fails, it must transition to `FAILED` — never left in
`GENERATING` permanently.

### Mechanism: Spring `@Async` with a named thread pool

```kotlin
// AsyncConfig.kt
@Configuration
@EnableAsync
class AsyncConfig {
    @Bean("generationExecutor")
    fun generationExecutor(): ThreadPoolTaskExecutor =
        ThreadPoolTaskExecutor().apply {
            corePoolSize   = 3
            maxPoolSize    = 3
            queueCapacity  = 25
            setThreadNamePrefix("gen-")
            initialize()
        }
}

// TaskGenerationOrchestrator.kt
@Async("generationExecutor")
fun generateAsync(topicId: String) {
    try {
        topicRepository.updateStatus(topicId, GENERATING)
        // ... build module, call AI, persist results ...
        topicRepository.updateStatus(topicId, ACTIVE)
    } catch (ex: Exception) {
        log.error("Task generation failed for topic {}: {}", topicId, ex.message)
        topicRepository.updateStatus(topicId, FAILED)
    }
}
```

The `@Async` annotation delegates execution to the `generationExecutor` thread
pool. The calling use case (`TopicManagementService`) returns immediately after
calling `generateAsync()` — the HTTP response is sent before generation begins.

### Why `@Async` over alternatives

Three mechanisms were evaluated:

**`@Async` (chosen)** — simple, Spring-native, integrates with the existing DI
container. The orchestrator is a plain Spring bean. No additional infrastructure.
Failure handling is a try/catch. The thread pool is a named bean that can be
monitored and tuned via configuration.

**`ApplicationEventPublisher`** — publishes a `TopicCreatedEvent` that a listener
handles asynchronously. More decoupled, but introduces an event type and a
listener class for what is a single caller / single handler relationship. The
added abstraction provides no benefit here — there is exactly one thing that
happens when a topic is created (generate tasks). Events shine when multiple
independent subscribers react to the same event. That is not the case here.

**`ThreadPoolTaskExecutor` directly** — calling `executor.submit { ... }` from
the use case manually. Works but bypasses Spring's `@Async` infrastructure
(AOP proxy, exception propagation, executor configuration). More boilerplate for
no gain over `@Async`.

### Timeout enforcement: `withTimeoutOrNull`

`TaskGenerationPort.generateTasks()` is a `suspend fun` (Kotlin coroutine). The
`@Async` method bridges to it via `runBlocking`. `withTimeoutOrNull` wraps the
coroutine call with the configured timeout:

```kotlin
val result = runBlocking {
    withTimeoutOrNull(timeoutMs) {
        taskGenerationPort.generateTasks(request)
    }
}

if (result == null) {
    topicRepository.updateStatus(topicId, FAILED)
    log.warn("Generation timed out for topic {}", topicId)
    return
}
```

If Mistral does not respond within the timeout window (`timeoutMs` = configured
Mistral timeout in milliseconds), `withTimeoutOrNull` returns `null`. The
orchestrator catches this and transitions the topic to `FAILED`. The coroutine
is cancelled — no threads are left waiting.

This is strictly better than a separate `@Scheduled` cleanup job that periodically
scans for stale `GENERATING` topics. The cleanup approach has a race window
between when the job gets stuck and when the scheduler runs. `withTimeoutOrNull`
has zero race window — the timeout fires in the same thread that owns the
generation job.

### `FAILED` status guarantee

Every path through `generateAsync` ends in either `ACTIVE` or `FAILED`:

```
Start
  │
  ├── updateStatus(GENERATING)
  │       │
  │       ├── withTimeoutOrNull → null  →  FAILED  (timeout)
  │       │
  │       ├── Either.Left(error)        →  FAILED  (AI error)
  │       │
  │       ├── Exception thrown          →  FAILED  (catch block)
  │       │
  │       └── Either.Right(tasks)
  │               │
  │               └── saveAll(tasks) → ACTIVE
  │
  └── (no path exits without a terminal status)
```

The top-level `try/catch` is the last-resort guarantee. Even if a bug in the
orchestrator throws an unexpected exception after `updateStatus(GENERATING)`,
the catch block sets `FAILED`. The topic is never stranded.

### Thread pool sizing

`corePoolSize = maxPoolSize = 3`. This means at most 3 topics can generate
simultaneously. The queue holds 25 more pending jobs. Rationale: Mistral has
a rate limit. Running more than 3 concurrent generation requests against a shared
API key increases the probability of hitting the rate limit and wasting all
concurrent attempts. 3 is a conservative choice that can be increased by
changing `ai.generation.pool-size` in `application.yml` without code changes.

## Decision 2 — File Storage: MinIO

### The problem

When an admin uploads a PDF, the file must be stored beyond the request lifecycle.
Three concerns drive the storage decision:

1. **Re-generation** — the admin can request task regeneration from stored
   content without re-uploading. The original source must be retrievable.
2. **Audit trail** — the raw PDF is the ground truth for what content was used
   to generate tasks. Storing only the extracted text loses provenance.
3. **Scale** — storing binary files in the PostgreSQL database is an
   anti-pattern that degrades database performance at any meaningful scale.

### Alternatives considered

**Store only extracted text (discard PDF)** — simplest approach, zero file
storage infrastructure needed. Rejected because it makes re-generation from
the original source impossible. If PDFBox produces poor extraction from a
specific PDF (common with scanned documents), there is no way to retry with a
different extraction strategy without asking the admin to re-upload.

**Store PDF as base64 in the database** — avoids external storage. Rejected
because binary blobs in a relational database degrade query performance, inflate
backup sizes, and prevent efficient range queries. PostgreSQL is not an object
store.

**Store PDF on the container filesystem** — no external dependency, zero config.
Rejected because the file is lost on every container redeploy, which is
unacceptable for a system that must re-generate tasks from stored content.
Consistent with ADR-009's philosophy: the deployment artefact is `docker compose
up -d` — the state of the filesystem between deploys is not preserved.

**AWS S3** — production-correct, globally available, durable. Rejected at this
stage because it introduces an AWS account dependency, egress costs, and IAM
configuration that add operational overhead inconsistent with the Docker Compose
deployment model. The migration path to real S3 is a one-line config change (see
below).

**MinIO (chosen)** — MinIO is an S3-compatible object store that runs as a
Docker container. The AWS SDK for Java connects to it using the same API as
real S3 — the only difference is the endpoint URL. This means:

- Zero external dependency: MinIO runs in `docker-compose.yml` alongside the
  application, database, and observability stack
- Zero cost beyond the VPS disk
- Zero code change to migrate to real S3 or Cloudflare R2 — change
  `MINIO_ENDPOINT`, `MINIO_ACCESS_KEY`, and `MINIO_SECRET_KEY` in `.env`

```yaml
# docker-compose.yml
minio:
  image: minio/minio:latest
  environment:
    MINIO_ROOT_USER: ${MINIO_ROOT_USER:-minioadmin}
    MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD:-minioadmin}
  ports:
    - "9000:9000"   # S3 API
    - "9001:9001"   # Web console
  volumes:
    - minio_data:/data
  command: server /data --console-address ":9001"
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:9000/minio/health/live"]
```

### `FileStoragePort` — the abstraction boundary

MinIO is an infrastructure detail. The application layer never imports
`software.amazon.awssdk`. The port interface shields all use cases from the
specific storage provider:

```kotlin
// application/port/FileStoragePort.kt
interface FileStoragePort {
    fun upload(bucket: String, key: String, bytes: ByteArray, contentType: String): String
    fun download(bucket: String, key: String): ByteArray
    fun delete(bucket: String, key: String)
}
```

`MinioFileStorageAdapter` in `infrastructure/storage/` implements this interface
using `S3Client` from the AWS SDK v2. Swapping to real S3 means: change the
`S3Client` endpoint configuration. The interface, use cases, and domain model
are untouched.

### What is stored

When an admin creates a PDF topic:
1. PDF binary → uploaded to MinIO, key = `topics/{topicId}/source.pdf`
2. `content_source_ref` column = the MinIO object URL
3. `raw_extracted_text` column = PDFBox-extracted plain text

When an admin creates a URL topic:
1. No binary upload — the URL itself is the source reference
2. `content_source_ref` column = the URL
3. `raw_extracted_text` column = Jsoup-extracted plain text

Both cases store the extracted text. The regenerate endpoint uses stored text
directly, avoiding redundant extraction. If text is missing (edge case), it
re-fetches from `content_source_ref`.

## Decision 3 — Content Extraction: PDFBox + Jsoup

### The problem

Two source types produce fundamentally different inputs:

- **URL** — an HTTP response containing HTML markup, navigation, ads, and
  boilerplate. The generation prompt needs only the article/content text.
- **PDF** — a binary document. Text must be extracted page by page.

Both must yield clean, prompt-ready plain text.

### PDF extraction: Apache PDFBox 3.x

PDFBox is the standard JVM library for PDF text extraction. It operates entirely
on the JVM — no native binaries, no system dependencies, no Docker layer changes.

```kotlin
override fun extractFromPdf(pdfBytes: ByteArray): String {
    return Loader.loadPDF(pdfBytes).use { doc ->
        val stripper = PDFTextStripper()
        val text = stripper.getText(doc).trim()
        log.debug("Extracted {} chars from PDF ({} pages)", text.length, doc.getNumberOfPages())
        text
    }
}
```

Note: PDFBox 3.x changed the entry point from `PDDocument.load()` to
`Loader.loadPDF()` — a breaking API change from 2.x. This was discovered during
compilation and corrected immediately. The version pinned in `build.gradle.kts`
is `3.0.1`.

**Limitation acknowledged:** PDFBox extracts text from text-based PDFs. Scanned
PDFs (images of text) yield empty or garbled output. OCR (e.g. Tesseract) would
be required for scanned documents. This is a known limitation — admins are
expected to upload text-based PDFs. OCR is a future enhancement, not a current
requirement.

### URL extraction: Jsoup

Jsoup fetches an HTML page and extracts readable text by stripping markup,
navigation, scripts, and style elements:

```kotlin
override fun extractFromUrl(url: String): String {
    val doc = Jsoup.connect(url)
        .timeout(10_000)
        .get()
    return doc.body().text().trim()
}
```

The 10-second timeout prevents a slow external URL from blocking the generation
thread indefinitely. A timeout produces an extraction failure which propagates
as `Either.Left(AppError)` from the use case, leaving the topic in `DRAFT`
status (never transitions to `GENERATING`).

**Why not a custom HTTP client + HTML parser?** Jsoup is the de-facto standard
for JVM HTML parsing. Its `.text()` method handles entity decoding, whitespace
normalisation, and tag stripping in one call. Building equivalent behaviour from
`RestTemplate` + regex would be more code with worse correctness.

### `ContentExtractorPort` — the abstraction boundary

Like `FileStoragePort`, the extraction strategy is hidden behind a port:

```kotlin
// application/port/ContentExtractorPort.kt
interface ContentExtractorPort {
    fun extractFromUrl(url: String): String
    fun extractFromPdf(pdfBytes: ByteArray): String
}
```

`ContentExtractorAdapter` in `infrastructure/content/` implements both methods.
The use case calls the port — it has no knowledge of PDFBox or Jsoup. Swapping
to a different extraction library (Apache Tika for multi-format support, for
example) requires replacing one class, not modifying any use case.

## The Complete Admin Content Flow

The three decisions above compose into a single coherent flow:

```
POST /admin/topics (multipart: metadata + optional PDF file)
  │
  ├── [PDF source]
  │     ContentExtractorPort.extractFromPdf(bytes) → rawText
  │     FileStoragePort.upload(bucket, key, bytes) → s3Url
  │     Topic { contentSourceType=PDF, contentSourceRef=s3Url, rawText }
  │
  ├── [URL source]
  │     ContentExtractorPort.extractFromUrl(url) → rawText
  │     Topic { contentSourceType=URL, contentSourceRef=url, rawText }
  │
  ├── Save Topic (status = DRAFT)
  │
  ├── Return 202 Accepted { topicId, status = GENERATING }
  │         ← HTTP response sent here, admin is not waiting
  │
  └── @Async: TaskGenerationOrchestrator.generateAsync(topicId)
        │
        ├── updateStatus(GENERATING)
        ├── Create TopicModule (orderIndex=0, objective=description)
        ├── Build GenerationRequest from rawText
        ├── withTimeoutOrNull(60s) { taskGenerationPort.generateTasks(request) }
        │       ├── null    → updateStatus(FAILED)
        │       ├── Left    → updateStatus(FAILED)
        │       └── Right   → saveAll(TaskTemplates) → updateStatus(ACTIVE)
        └── catch(Exception) → updateStatus(FAILED)

GET /admin/topics/{id}/status → { status: GENERATING | ACTIVE | FAILED }
  (admin polls until ACTIVE or handles FAILED)
```

## Regeneration Flow

When an admin calls `POST /admin/topics/{id}/regenerate`:

1. All existing `TaskTemplate` records for the topic are marked `isCurrent = false`
2. The existing `TopicModule` is deleted
3. `generateAsync(topicId)` is called — same flow as initial creation
4. The stored `rawExtractedText` is used directly (no re-fetch, no re-upload)
5. If `rawExtractedText` is null (edge case), the orchestrator re-fetches from
   `content_source_ref` (re-downloads from MinIO for PDF, re-fetches URL for URL)

Old task templates are soft-deleted (marked not current) rather than hard-deleted.
This preserves the audit trail of previous generation attempts and allows
comparison between versions.

---

## Task Versioning

When an admin edits an existing task template via
`PUT /admin/topics/{topicId}/tasks/{taskTemplateId}`:

```
Existing TaskTemplate (version=1, isCurrent=true)
  │
  └── New TaskTemplate created:
        version = 2
        isCurrent = true
        supersededBy = null

Old TaskTemplate updated:
  isCurrent = false
  supersededBy = new.id
```

Users with active `TaskAssignment` records referencing version 1 keep their
version — mid-task edits do not affect in-progress learners. New assignments
always receive the `isCurrent = true` version. This is enforced at the
repository level by querying `WHERE is_current = true AND module_id = ?`.

## Alternatives Considered (Summary)

| Concern | Rejected alternative | Reason |
|---------|---------------------|--------|
| Async | `ApplicationEventPublisher` | Over-abstracted for single subscriber |
| Async | Manual `executor.submit` | Bypasses Spring AOP, more boilerplate |
| Timeout | `@Scheduled` cleanup job | Race window between stuck job and scheduler run |
| File storage | Text only, discard PDF | Prevents re-generation from original source |
| File storage | Base64 in PostgreSQL | Degrades DB performance, inflates backups |
| File storage | Container filesystem | Lost on every redeploy |
| File storage | AWS S3 | External dependency, IAM overhead at current scale |
| PDF extraction | Apache Tika | Full multi-format support not needed; PDFBox is simpler for PDF-only |
| URL extraction | Custom HTTP + regex | More code, worse correctness than Jsoup |

---

## Consequences

- Task generation is fully non-blocking. Admin POST returns in < 100ms regardless
  of Mistral response time.
- Topics never get stuck in `GENERATING` — every code path ends in `ACTIVE` or
  `FAILED`.
- File storage is provider-agnostic. Migrating from MinIO to AWS S3 or
  Cloudflare R2 is a three-line change in `.env` — no application code changes.
- PDF extraction has a known limitation for scanned documents. This is documented
  here and communicated to admins via API validation error messaging.
- Task template versioning preserves the full history of content changes. An
  active learner's task is never silently changed mid-session.
- The regenerate endpoint allows admins to improve task quality without
  re-uploading source material — the stored raw text is the persistent source of
  truth for generation.
