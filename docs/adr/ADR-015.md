# ADR-015: Observability Strategy — Custom Metrics, Prometheus, and Grafana

**Status:** Accepted  
**Date:** 2026-04-05  
**Phase:** 6 — Observability  
**Author:** Play4Change Engineering

---

## Context

Play4Change is a production-targeted adaptive learning platform. After five phases of implementation, the core backend is complete: auth, topic management, async task generation, the full learning flow, and peer review. What was missing was visibility into how the system behaves at runtime.

Without instrumentation, the following questions have no answer:

- Is Mistral taking 2 seconds or 45 seconds to generate tasks? Is it timing out silently?
- Are users getting adaptive branches (meaning they are struggling), and what kind of errors are most common?
- Is the Redis cache actually reducing database load, or is it getting cache-missed on every request?
- How many enrollments happened today? What fraction of task submissions are correct on the first attempt?
- Is the JVM running out of heap? Are certain endpoints slow under load?

A system you cannot observe is a system you cannot improve. Observability is not a feature — it is the foundation of responsible production operation.

---

## Decision Drivers

- **Low instrumentation cost:** Adding metrics should not require changes to architecture or interfaces — it should be additive.
- **No vendor lock-in:** Metrics must be portable. Switching from self-hosted Prometheus to a managed service (Datadog, Grafana Cloud) should be a config change, not a code change.
- **Domain-specific signal:** JVM and HTTP metrics alone are not enough. The metrics that matter for an adaptive learning platform are domain events: struggles opened, tasks submitted, peer reviews completed, cache hit ratios.
- **Production-grade from day one:** The observability stack (Prometheus + Grafana) already runs in Docker Compose. Phase 6 should complete it, not stub it.

## The Observability Stack

### Mental Model

The pipeline has three stages:

```
Application Code
      │
      │  registry.counter("...").increment()
      │  Timer.start(registry) / sample.stop(timer)
      ▼
MeterRegistry (Micrometer)
      │
      │  /actuator/prometheus  (HTTP endpoint, scraped every 15s)
      ▼
Prometheus
      │
      │  PromQL queries
      ▼
Grafana Dashboard
```

**Micrometer** is the instrumentation façade — it sits between application code and any specific metrics backend. The application calls `registry.counter(...)` or `registry.timer(...)`. Micrometer translates these into the format that the configured backend understands. In this case, the backend is Prometheus, so Micrometer formats metrics as Prometheus text exposition format at `/actuator/prometheus`.

**Prometheus** scrapes that endpoint every 15 seconds and stores the time-series data. It does no alerting or visualisation — it is purely a time-series database.

**Grafana** queries Prometheus using PromQL and renders the results as panels. It has no direct connection to the application.

This separation means: if Play4Change later moves to Datadog, only the Micrometer registry dependency and one config line change. The application instrumentation code is identical.

## Decision 1: Micrometer API with Direct MeterRegistry Injection

### The Problem

There are three common patterns for adding metrics in Spring Boot applications:

**Pattern A — `@Timed` annotation (AOP-based)**  
Add `@Timed("my.timer")` to a method. Spring AOP wraps it automatically.

**Pattern B — Centralized MetricsService**  
Create one class that wraps MeterRegistry and expose it as a service. All other services inject MetricsService instead of MeterRegistry directly.

**Pattern C — Direct MeterRegistry injection**  
Each service that needs metrics injects `MeterRegistry` from Spring's IoC container. Metrics are recorded inline at the exact point they are meaningful.

### Alternatives Considered

| Pattern | Pros | Cons |
|---|---|---|
| `@Timed` (AOP) | Zero boilerplate, works for simple method timing | Cannot set dynamic tags at runtime; cannot record domain-specific events (only method duration); does not work for counters |
| Centralized MetricsService | Single point for all metric names; easy to audit | Violates Clean Architecture — infrastructure concern leaks into application layer; creates a god class; adds indirection with no benefit at this scale |
| Direct MeterRegistry injection | Tags are computed from actual runtime values; counters and timers are recorded at the exact semantically correct point; consistent with how Spring Boot ships Micrometer | Slightly more verbose |

### Decision

**Direct `MeterRegistry` injection into each service that needs it.**

`@Timed` is excluded because Play4Change's most important metrics are domain events (struggle sessions opened, peer reviews finalized), not method durations. These cannot be expressed as AOP method annotations. The centralized MetricsService was rejected because it would violate the Clean Architecture boundary — application services should not depend on an infrastructure wrapper class.

### Implementation

Six services were instrumented:

| Service | Metric | Type |
|---|---|---|
| `RedisCacheAdapter` | `cache_requests_total`, `cache_hits_total` | Counter × 2 |
| `EnrollmentService` | `topic_enrollments_total` | Counter |
| `TaskService` | `task_submissions_total` | Counter |
| `PeerReviewService` | `peer_reviews_completed_total` | Counter |
| `HandleStruggleService` | `struggle_sessions_total` | Counter |
| `TaskGenerationOrchestrator` | `task_generation_duration_seconds` | Timer |

All metrics automatically carry an `application="play4change-server"` tag, set globally via:

```yaml
management:
  metrics:
    tags:
      application: ${spring.application.name}
```

This means every metric can be filtered by application in Grafana, making the dashboard portable to multi-service deployments.

## Decision 2: Six Domain-Specific Metrics (Design Rationale)

### `task_generation_duration_seconds` — Timer with status tag

Mistral AI task generation runs asynchronously. Without timing it, there is no way to know whether the P95 generation time is 5 seconds or 5 minutes, whether timeouts are rare or frequent, or whether a deployment introduced a regression.

The Timer is started **inside** the `@Async` method body, not at the call site:

```kotlin
@Async("generationExecutor")
fun generateAsync(topicId: String) {
    val sample = Timer.start(registry)   // ← executor thread, not caller thread
    try {
        // ... work ...
        sample.stop(registry.timer("task_generation_duration_seconds", "status", "success"))
    } catch (ex: Exception) {
        sample.stop(registry.timer("task_generation_duration_seconds", "status", "failed"))
    }
}
```

Starting the sample at the call site would measure the time from dispatch to the executor thread, including queue wait time. That would conflate "the executor was busy" with "Mistral was slow" — two very different problems with very different remedies. Starting inside the async body measures only the actual generation work.

All four exit paths stop the sample with the appropriate `status` tag: `success`, `failed`, or `timeout`. This guarantees no sample is ever left running, which would cause a memory leak in long-running services.

The Prometheus histogram produced by Micrometer's Timer enables percentile queries:

```promql
histogram_quantile(0.95, rate(task_generation_duration_seconds_bucket[5m]))
```

This yields the P95 generation time — the time below which 95% of generation jobs complete — which is a far more useful signal than average duration for identifying tail latency.

### `struggle_sessions_total` — Counter with error_pattern tag

Each struggle session carries an `ErrorPattern` (WRONG_CONCEPT, PARTIAL_UNDERSTANDING, READING_ERROR, TIME_PRESSURE) computed deterministically at trigger time. Counting sessions by pattern answers: what kind of misconceptions are most common across the learner population?

If READING_ERROR is dominant, the task wording may need revision. If TIME_PRESSURE is dominant, the daily task window may be too tight. This is actionable signal that cannot be obtained from logs alone.

Tags use snake_case string literals, not `enum.name`, to keep the Prometheus label values stable even if enum names are renamed during refactoring.

### `cache_requests_total` + `cache_hits_total` — Two Counters, ratio in Grafana

Cache hit ratio is computed in Grafana from two counters rather than as a single gauge:

```promql
rate(cache_hits_total[5m]) / rate(cache_requests_total[5m])
```

This approach is preferred over a pre-computed ratio gauge because:
- Gauges for ratios can produce misleading spikes when the denominator is near zero
- Storing both raw values retains the full signal — you can independently chart total request rate and hit count, not just the ratio
- Prometheus best practices recommend counters over derived gauges for ratio computation

The `key_prefix` tag is extracted from the cache key by taking the segment before the first colon:

```kotlin
val keyPrefix = key.substringBefore(":", missingDelimiterValue = key)
// "task:abc:123"  → "task"
// "enrollment:u1" → "enrollment"
// "topic:xyz"     → "topic"
// "template:id:1" → "template"
```

This allows the dashboard to break down cache performance by data type — it may be that topic cache is hitting at 95% while task cache is only at 40%, indicating the task TTL should be extended.

### `peer_reviews_completed_total` — Counter with finalized tag

Every verdict submission increments this counter. The `finalized` tag (`true`/`false`) indicates whether that verdict was the third — the one that triggered majority-vote finalization. This allows the dashboard to distinguish raw verdict throughput from actual submission closures.

### `topic_enrollments_total` and `task_submissions_total`

These are operational health metrics. A sudden drop in enrollments could indicate a broken enroll endpoint or a topic generation failure leaving no ACTIVE topics to enroll in. A drop in task submissions indicates users are not progressing through their daily tasks.

The `task_submissions_total` counter carries `outcome` (correct, incorrect, late, pending_review) and `task_type` (multiple_choice, todo_action) tags, enabling a pie chart breakdown of submission outcomes — a direct signal of learner performance across the platform.

## Decision 3: Grafana Dashboard with Prometheus Datasource (Self-Hosted)

### The Problem

The Grafana service was already running in Docker Compose with a provisioning mount for datasources and dashboards. However:

1. The file `infra/grafana/provisioning/datasources/prometheus.yml` contained a **Prometheus scrape config** — a file intended for the Prometheus process, not for Grafana. This file is syntactically valid YAML but semantically wrong: Grafana ignores it and falls back to manual datasource configuration, meaning the Prometheus data source was never wired automatically.

2. The dashboard stub (`infra/grafana/dashboard.json`) was in the wrong directory. Grafana's dashboard provisioner was configured to read from `infra/grafana/dashboards/` (mapped to `/etc/grafana/dashboards`), but the stub was at `infra/grafana/dashboard.json` — one level above. It was never loaded.

Both issues were fixed as part of Phase 6.

### Datasource Fix

The correct Grafana datasource provisioning file format is:

```yaml
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    url: http://prometheus:9090
    isDefault: true
    access: proxy
```

This is a Grafana-specific format (not a Prometheus config) that Grafana reads at startup to auto-wire the datasource. With this file in place, every new Grafana instance provisioned from the Docker Compose stack automatically has Prometheus available — no manual "Add Data Source" step required. This is essential for reproducible development environments and CI/CD.

### Dashboard: 9 Panels

The dashboard is stored as a provisioned JSON file at `infra/grafana/dashboards/play4change-dashboard.json`. Grafana loads it automatically on startup. An operator with no Grafana experience can run `docker compose up` and immediately see the dashboard.

| Panel | PromQL | Type |
|---|---|---|
| Task Generation P95 Duration | `histogram_quantile(0.95, rate(task_generation_duration_seconds_bucket[5m]))` | Time series |
| Task Generation Success Rate | `rate(...{status="success"}[5m]) / rate(...[5m])` | Stat |
| Struggle Sessions by Error Pattern | `increase(struggle_sessions_total[1h])` | Bar chart |
| Cache Hit Ratio | `rate(cache_hits_total[5m]) / rate(cache_requests_total[5m])` | Gauge |
| Enrollments Over Time | `increase(topic_enrollments_total[1h])` | Time series |
| Task Submission Outcomes | `increase(task_submissions_total[1h])` | Pie chart |
| Peer Review Verdict Rate | `rate(peer_reviews_completed_total[5m])` | Time series |
| JVM Heap Usage | `jvm_memory_used_bytes{area="heap"}` | Time series |
| HTTP Request Rate | `rate(http_server_requests_seconds_count[5m])` | Time series |

The last two panels require no custom instrumentation — they come free from Spring Boot Actuator's auto-configured JVM and HTTP metrics. Including them alongside domain metrics gives a complete picture: you can correlate a spike in struggle sessions with a spike in JVM heap usage or a slowdown in HTTP response time.

### Alternatives Considered

**Grafana Cloud (managed):** Eliminates the self-hosted Grafana container. No operational cost for running Grafana. Rejected because it introduces an external network dependency into a local development stack — the entire platform should run with `docker compose up` and no internet access required. Migration to Grafana Cloud in production is a one-line datasource URL change.

**Prometheus Alertmanager:** Not included in Phase 6. Alerting rules are a natural Phase 7 addition — they require knowing the normal operating ranges first, which requires data from running the dashboard for a period of time. Adding alerts before having baseline data produces noisy false positives.

**Datadog / New Relic / Dynatrace:** Managed APM platforms. All rejected for the same reason: they introduce per-seat or per-host billing and vendor lock-in. Micrometer's registry abstraction means the application instrumentation is identical — switching is a dependency swap, not a code rewrite.

## Consequences

### Positive

- **Complete production readiness:** The system now has end-to-end visibility from domain events (struggle sessions) down to JVM internals (heap usage).
- **Zero architectural impact:** All six metric additions are additive — no interfaces changed, no use cases modified, no domain logic touched.
- **Reproducible dashboards:** Grafana and Prometheus configuration is fully provisioned from files in the repository. `git clone` + `docker compose up` gives a running observability stack with no manual setup.
- **Micrometer portability:** The `MeterRegistry` abstraction ensures the instrumentation code is backend-agnostic. Prometheus can be swapped for any Micrometer-supported backend (Datadog, CloudWatch, InfluxDB) with a dependency swap and config change.

### Negative / Trade-offs

- **No alerting yet:** The dashboard surfaces problems visually but does not page anyone when the task generation failure rate exceeds a threshold. Alertmanager integration is deferred.
- **No distributed tracing:** Micrometer metrics measure rates and durations but do not trace individual requests end-to-end. OpenTelemetry integration would be required for request-level tracing across the async generation pipeline.
- **Cardinality risk on cache keys:** The `key_prefix` tag was specifically designed to avoid high cardinality (using only 4 values: task, topic, enrollment, template). Using the full cache key as a tag would create one time-series per user per topic, which would exhaust Prometheus memory at scale.

## Notes

**Why `Timer.start(registry)` not `Timer.Sample.start(registry)`:** In Micrometer 1.11+ (shipped with Spring Boot 3), `Timer.Sample.start()` was deprecated in favour of the static `Timer.start(registry)` method. The builder chain `Timer.builder("name").tag(...).register(registry)` used in `sample.stop()` was also replaced with the more concise `registry.timer("name", "tag", "value")` overload, which avoids a type inference issue in the Kotlin compiler when the builder chain is passed as a lambda argument.

**The `@Value` annotation warnings:** Two pre-existing warnings about `@Value` annotation targets appear on `TaskGenerationOrchestrator` and `HandleStruggleService`. These were present before Phase 6 and are not introduced by the observability changes. They do not affect runtime behaviour.

**Dashboard UID:** The Grafana dashboard JSON uses `"uid": "prometheus"` for the datasource reference rather than the datasource name string `"Prometheus"`. The UID form is more robust: it resolves correctly even if the datasource display name is changed in the Grafana UI, and it is the form Grafana uses internally when exporting dashboards.
