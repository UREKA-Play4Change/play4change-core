# ADR-016 ‚Äî Authentication Hardening: Security Audit and Remediation

**Status:** Accepted
**Date:** April 2026
**Author:** Radesh Govind
**Relates to:** ADR-011 (Passwordless Authentication: Magic Link + OAuth 2.0)

---

## Context

An internal security audit of the Identity bounded context (documented in ADR-011)
identified 11 gaps between the original design intent and the actual implementation.
The gaps were discovered by reading every file in the auth package against the
ADR-011 specification line by line.

The gap categories were:

- **Security vulnerabilities** ‚Äî raw magic-link tokens persisted in the database;
  Google OAuth audience validation silently skipped when `GOOGLE_CLIENT_ID` was blank
- **Correctness bugs** ‚Äî magic-link claim check not atomic (TOCTOU race condition);
  refresh token rotation discarding the email claim from the new access token;
  logout not revoking the full token family
- **Error handling inconsistencies** ‚Äî auth exceptions, 403 responses, and malformed
  body rejections returning different error shapes instead of the unified `MessageResponse`
- **Operations gaps** ‚Äî no documented or automated path to create the first admin user

The audit was structured in two phases:

1. **Phase 1 (read-only)** ‚Äî produced a numbered gap table (G1‚ÄìG11) with severity
   and file-level location for each gap. No code was changed.
2. **Phases 2A‚Äì2C (fix batches)** ‚Äî gaps grouped by risk:
   - **Batch A** ‚Äî security fixes (G1, G3; G4 deferred)
   - **Batch B** ‚Äî correctness fixes (G2, G5, G6, G7)
   - **Batch C** ‚Äî error handling and operations (G8, G9, G10, G11)

---

## Decision

### Security Fixes (G1, G3; G4 deferred)

**G1 ‚Äî Raw magic-link tokens stored in plaintext in the database**

The original `MagicLinkService` persisted the raw 32-byte hex token directly in
`magic_link_tokens.token`. A database compromise would expose all pending tokens,
allowing an attacker to authenticate as any user whose link had not yet been clicked.

*Change:* `MagicLinkService.kt` now hashes the generated token with SHA-256 before
passing it to the repository. Only `SHA-256(token)` is stored. The raw token is
transmitted in the email link and in the single `GET /auth/verify` request ‚Äî it is
never persisted. `MagicLinkTokenJpaAdapter.kt` was updated to look up by hash on
verify. The `magic_link_tokens.token` column now semantically stores a hash, matching
the intent stated in ADR-011 but not previously implemented.

*Why SHA-256:* Magic-link tokens are 32 bytes of cryptographically random data ‚Äî
they are not user-chosen passwords. SHA-256 is a one-way function sufficient to
protect high-entropy random values. bcrypt's deliberately slow key derivation provides
no benefit here (see Alternatives Considered).

---

**G3 ‚Äî Google OAuth audience validation silently skipped when `GOOGLE_CLIENT_ID` is blank**

The original `GoogleOAuthAdapter` read the audience from configuration and compared
it to the `aud` claim in the Google ID token. When `GOOGLE_CLIENT_ID` was empty
(the default for local dev), the adapter skipped the audience check entirely ‚Äî any
Google ID token was accepted regardless of its intended audience.

*Change:* `GoogleOAuthAdapter.kt` now throws `IllegalStateException` at startup if
`GOOGLE_CLIENT_ID` is blank, and rejects tokens at verification time if the `aud`
claim does not match the configured client ID. An empty client ID is treated as a
misconfiguration, not a "skip validation" signal.

*Why this matters:* An ID token issued to a different application (a different `aud`)
must not be accepted by Play4Change. A malicious client could obtain a Google ID token
for any other application and submit it ‚Äî without audience validation, Play4Change
would accept it and issue a session for the corresponding Google account.

---

**G4 ‚Äî Facebook OAuth token verification not hardened (deferred)**

Facebook access tokens submitted to `POST /auth/oauth` with `provider=FACEBOOK`
are currently verified only by calling `GET https://graph.facebook.com/me`. This
does not prove the token was issued for Play4Change's Facebook App ‚Äî any valid
Facebook access token from any app would be accepted.

*Change:* Deferred. Hardening requires `FACEBOOK_APP_ID` and `FACEBOOK_APP_SECRET`
to perform App Token verification (`GET /debug_token`). These environment variables
are not yet configured. The Facebook OAuth endpoint remains functional with its
current unverified behaviour until the Meta service integration is completed.
`FACEBOOK_APP_ID` and `FACEBOOK_APP_SECRET` are documented in environment variable
tables as not-yet-required.

### Correctness Fixes (G2, G5, G6, G7)

**G2 ‚Äî Magic-link token claim check was not atomic (TOCTOU race condition)**

The original flow in `MagicLinkTokenJpaAdapter` performed two separate database
operations: `SELECT` (check `used = false`) followed by `UPDATE SET used = true`.
Under concurrent requests (or a race between two identical verify calls), both could
pass the `SELECT` check before either issued the `UPDATE`, allowing a single-use
token to authenticate two sessions.

*Change:* `MagicLinkTokenJpaRepository.kt` now uses an atomic SQL `UPDATE ‚Ä¶ SET
used = true WHERE token_hash = :hash AND used = false RETURNING *` CTE. The token
is claimed and returned in a single statement. If the update affects 0 rows (already
used or not found), the adapter treats the token as invalid. No separate `SELECT`
is required.

*Why atomic CTE over `@Transactional` + `SELECT FOR UPDATE`:* `SELECT FOR UPDATE`
requires an explicit transaction wrapping both statements, a wider `@Transactional`
scope on the service method, and adds lock contention under any concurrent load. The
single-statement CTE eliminates the need for application-level locking entirely and
reduces the claim path to one round-trip (see Alternatives Considered).

---

**G5 ‚Äî Refresh token rotation discarded the email claim**

When `POST /auth/refresh` was called, `TokenService` issued a new access token
using only the data available in the `refresh_tokens` row: `userId` and `role`. The
`email` claim ‚Äî present in the original access token ‚Äî was not propagated because
it was not stored on the refresh token record.

*Change:* `TokenService.kt` now loads the `User` entity by `userId` (primary key)
from `UserRepository` during token refresh. The email is read from the live user
record and embedded in the new access token. The `refresh_tokens` table is not
changed ‚Äî no email column was added.

*Why loading by PK instead of adding a column:* The email field exists in one
authoritative place: `users.email`. Duplicating it in `refresh_tokens` creates two
places to update if an email changes and violates single-source-of-truth. A PK
lookup is one indexed read and adds negligible latency (see Alternatives Considered).

---

**G6 ‚Äî Logout only invalidated the submitted token, not the entire family**

`DELETE /auth/logout` called `revokeToken(tokenHash)` which set `used = true` on
the single submitted refresh token. Other tokens in the same family (issued on the
same login event, e.g., from other devices) remained valid.

*Change:* `TokenService.kt` logout now calls `revokeFamily(familyId)` instead.
The `familyId` is read from the submitted token's record. All tokens sharing that
`family_id` are invalidated in a single `UPDATE`. This matches the session semantics
implied by ADR-011 ("family-based revocation") which was documented but not applied
to the logout path.

*Why the whole family:* Logout is a deliberate session termination. If a user logs
out on one device, tokens derived from the same login on other devices should also
be revoked. This is the expected behaviour for "sign out everywhere" and is consistent
with the reuse-detection revocation already implemented for the theft-detection path.

---

**G7 ‚Äî Public routes returned 401 when a stale Bearer token was present**

`JwtAuthFilter` parsed the `Authorization: Bearer` header unconditionally on every
request, including public routes (`/auth/magic-link`, `/auth/verify`, `/auth/oauth`).
A client presenting an expired access token while trying to re-authenticate would
receive a 401 from the filter before the controller was reached, preventing re-login
without first clearing local token storage.

*Change:* `JwtAuthFilter.kt` catches token parse/validation exceptions on public
routes and continues the filter chain without setting a `SecurityContext`. If parsing
fails on a public route, the request proceeds to the controller (which requires no
authentication). The 401 is only returned when a protected route receives an invalid
token.

*Why this matters:* The mobile client may present a cached (expired) access token
on a magic-link verify or refresh call. Blocking that call with 401 creates a
dead-lock: re-auth requires calling a public endpoint, but the public endpoint rejects
the stale token before the request reaches the handler.

### Error Handling and Operations (G8, G9, G10, G11)

**G8 ‚Äî Authentication exceptions returned raw Spring error bodies**

`AuthenticationException` subtypes thrown from `JwtAuthFilter` and `MagicLinkService`
were handled by Spring's default error mechanism, producing responses with varying
shapes (`timestamp`, `status`, `error`, `path` fields). The frontend had no single
shape to parse.

*Change:* `AuthExceptionHandler.kt` was extended with `@ExceptionHandler` methods
for all auth-related exceptions, returning `ResponseEntity<MessageResponse>` with a
consistent `{ "message": "..." }` body and the appropriate HTTP status code.

---

**G9 ‚Äî 403 Forbidden response did not use `MessageResponse`**

Spring Security's `AccessDeniedException` handler returned a default error body
rather than the `MessageResponse` shape used by all other error responses.

*Change:* `AuthExceptionHandler.kt` adds an `AccessDeniedException` handler that
returns `403 Forbidden { "message": "Insufficient role" }`. The role is embedded in
the JWT ‚Äî the rejection occurs without any database lookup.

---

**G10 ‚Äî 400 for malformed JSON body did not use `MessageResponse`**

`HttpMessageNotReadableException` (thrown when the request body is not valid JSON)
produced Spring's default error body.

*Change:* `AuthExceptionHandler.kt` adds a `HttpMessageNotReadableException` handler
that returns `400 Bad Request { "message": "Malformed request body" }`.

---

**G11 ‚Äî No documented or automated path to create the first admin user**

There was no seeded admin account and no documented procedure for promoting the first
user to `ADMIN`. The `admin_topic.http` demo file contained a typo in the promotion
SQL (`admin@ example.com` with an erroneous space).

*Change:* `V10__seed_initial_admin.sql` was added as a Flyway migration containing
the canonical promotion SQL in a comment and a `SELECT 1` placeholder (Flyway requires
at least one executable statement). `README.md`, `demo/HOW_TO_RUN.md`, and
`demo/admin_topic.http` were updated with the correct SQL, the re-login requirement,
and the typo fix.

## Consequences

- **24 unit tests pass** ‚Äî up from 22. `TokenServiceTest` added 2 tests covering
  full family revocation on logout and no-op behaviour on unknown token hash.
- **All auth error responses share `MessageResponse` shape** ‚Äî `{ "message": "..." }`
  is the single error contract for all auth endpoints. Frontend error handling is
  now uniform.
- **Magic-link tokens are safe against DB compromise** ‚Äî only SHA-256 hashes are
  stored. Raw tokens travel only in email links and in the single verify request.
- **OAuth audience validation is mandatory** ‚Äî a blank or missing `GOOGLE_CLIENT_ID`
  is a hard configuration error, not a silent skip.
- **Refresh rotation preserves all JWT claims** ‚Äî the email claim is present in
  every access token, whether issued on initial login or on refresh.
- **Logout revokes the entire token family** ‚Äî a single logout terminates all sessions
  derived from the same login event across all devices.
- **Public routes do not 401 on stale Bearer tokens** ‚Äî clients can re-authenticate
  regardless of what is in their local token cache.
- **Facebook OAuth hardening is deferred** ‚Äî pending Meta service availability.
  `FACEBOOK_APP_ID` and `FACEBOOK_APP_SECRET` are documented but not yet required.
- **No breaking API changes** ‚Äî all endpoints maintain the same request and response
  contracts. The fixes are entirely internal to the server implementation.

## Alternatives Considered

### G1 ‚Äî bcrypt instead of SHA-256 for magic-link token storage

bcrypt is the standard algorithm for hashing user-chosen passwords. The alternative
was to apply it here for defence-in-depth.

**Rejected.** Magic-link tokens are 32 bytes of cryptographically random data ‚Äî they
are not user-chosen passwords. The entire motivation for bcrypt's slow-by-design
property is to make offline dictionary attacks against human-chosen passwords
economically infeasible. A 32-byte random token has ~256 bits of entropy ‚Äî there is
no dictionary to attack. SHA-256 is computationally instant and provides the same
one-way protection for values that cannot be guessed. Using bcrypt here would add
60‚Äì400ms of latency per verify request with zero security benefit.

---

### G2 ‚Äî `SELECT FOR UPDATE` instead of atomic CTE for token claim

The conventional Spring/JPA approach to preventing the TOCTOU race condition is to
use `@Transactional` on the service method plus a `SELECT FOR UPDATE` query to hold
a row lock between the read and the update.

**Rejected.** `SELECT FOR UPDATE` requires a wider `@Transactional` scope on the
service method (locking across the service boundary rather than inside the repository).
Under connection pool pressure ‚Äî when many threads are waiting for a connection ‚Äî
holding a row lock while waiting for a connection is a deadlock risk. The atomic CTE
reduces the entire claim operation to a single SQL statement: one round-trip, no
explicit locking, no application-level transaction scope required. The result is
simpler code with better concurrency properties.

---

### G5 ‚Äî Adding an `email` column to `refresh_tokens` instead of loading the user by PK

The email could be stored redundantly in the `refresh_tokens` table at issuance time
so that refresh rotation does not require a separate database read.

**Rejected.** This violates single-source-of-truth for user email. If a user changes
their email (a capability that may be added in a future phase), the `refresh_tokens`
rows would contain stale email values, and all active sessions would issue access
tokens with the old email until each token was individually rotated out. Loading
`users` by primary key on refresh is one indexed lookup against an in-memory-cached
table and keeps the `refresh_tokens` schema narrow and coherent.
