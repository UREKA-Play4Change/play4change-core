# ADR-017 — Play4Change Admin Web: React SPA with Hexagonal Architecture

**Status:** Accepted
**Date:** April 2026                                                                                              
**Author:** Radesh Govind                                                                                       
**Relates to:** N/A (first ADR for this sub-project)
                                                                                                                    
---                                                                                                               

## Context

Play4Change requires an admin web interface that allows platform administrators to create,                        
monitor, and regenerate AI-generated learning topics. The backend REST API already exists
and exposes endpoints under `/admin/*` and `/auth/*`. The admin UI must authenticate via                          
magic link (passwordless email) and optionally OAuth, display topic status through their                          
AI-generation lifecycle (PENDING → GENERATING → ACTIVE / FAILED), and provide a public                            
marketing landing page for prospective learners.

The project is a small team effort with an aggressive build timeline. The frontend must                           
be independently deployable, support a dev-vs-production mode switch without code changes                         
(for local development without a live API), and be maintainable by engineers who may join                         
later with varying React experience. Deployment target is Cloudflare Pages (static CDN),                          
ruling out server-side rendering frameworks that require a Node runtime at the edge.
                                                                                                                    
---                                                                                                               

## Decision

A React 18 single-page application built with Vite, structured in four explicit layers                            
(Domain → Application → Infrastructure → UI) following the Ports and Adapters                                   
(Hexagonal) pattern, deployed as a static bundle to Cloudflare Pages.
                                                                                                                    
---

## Architecture

### Layer structure

src/
domain/         — TypeScript interfaces only; no runtime code
models/       — Topic, Auth, Stats, Enrollment types                                                          
ports/        — IAuthService, ITopicService, IStatsService contracts                                          
application/    — Business orchestration; depends only on domain                                                
hooks/        — useAuth, useTopics, useStats (TanStack Query wrappers)                                        
infrastructure/ — Side-effectful implementations                                                                
api/          — Real Axios adapters (authAdapter, topicAdapter, statsAdapter)                                 
mock/         — In-memory mock adapters (switched by VITE_USE_MOCK)                                           
di/           — container.ts: single factory that picks mock or real at startup                               
ui/             — React components; depends only on application hooks                                           
pages/        — Route-level components                                                                        
layouts/      — AdminLayout (sidebar), PublicLayout (nav+footer)                                              
components/   — Shared primitives (ProtectedRoute, TopicCard, StatCard, …)                                    
hooks/        — UI-only hooks (useIntersectionObserver)

### Routing

React Router v7 with `BrowserRouter`. Routes are declared in `src/App.tsx` using                                  
`React.lazy()` for code-splitting on every page. Route constants live in
`src/lib/constants.ts` (`ROUTES` object) so paths are never hard-coded in components.

Protected admin routes are wrapped in `<ProtectedRoute>` (calls `GET /admin/me`;                                  
redirects to `/admin/login` on error or missing user) and rendered inside                                         
`<AdminLayout>` (sidebar navigation, mobile overlay drawer).

### Auth flow

1. User submits email → POST `/auth/magic-link`
2. User clicks email link → `AuthVerifyPage` calls GET `/auth/verify?token=…`
3. Response `{accessToken, refreshToken}` written via `setTokens()`:
    - Access token → `sessionStorage`
    - Refresh token → `document.cookie` with `SameSite=Strict; Secure; max-age=604800`
4. `BroadcastChannel('auth')` notifies other open tabs of successful login.
5. On 401 from any protected endpoint: `apiClient.ts` interceptor calls POST                                      
   `/auth/refresh`, queues in-flight requests, replays on success. On refresh failure:                            
   dispatches `auth:session-expired` DOM event, which `App.tsx` catches to clear                                  
   QueryClient and redirect to `/admin/login?reason=session-expired`.

### API communication

Single Axios instance in `src/infrastructure/api/apiClient.ts`. Request interceptor                               
attaches `Authorization: Bearer {token}` to every non-`/auth/` request. Error messages
are normalised by `src/lib/errorMessage.ts` which probes `data.message`, `data.error`,                            
`data.detail` in order.

### State management

No client-side global state store. TanStack Query v5 manages all server state with                                
query keys `['currentUser']`, `['topics', status]`, `['topic', id]`,
`['platformStats']`. Mutations (login, create topic, logout) invalidate or reset                                  
relevant queries on settle. Local component state (form fields, UI toggles) uses                                  
`useState`/`useReducer` as appropriate.

### Mock / real adapter switching

`VITE_USE_MOCK=true` in `.env.development` causes `container.ts` to instantiate mock                              
adapters that return deterministic in-memory data with artificial latency. Flipping the
flag to `false` and supplying a real `VITE_API_BASE_URL` switches to HTTP adapters with                           
no code changes.

## Why this, not something else

| Choice | Alternative considered | Why this was chosen |                                                         
  |---|---|---|                                                                                                   
| React + Vite | Next.js | Cloudflare Pages serves static files; no Node runtime available. Next.js SSR/SSG would 
require either Workers (extra complexity) or full static export (loses most Next.js value). Vite produces a plain
static bundle with zero server dependency. |
| React Router v7 | TanStack Router | React Router is the established standard; team already familiar. TanStack   
Router offers stricter type-safety on params but adds API surface to learn with no concrete benefit at this scale.
| TanStack Query | Redux Toolkit Query / SWR | TanStack Query v5 provides mutation state, optimistic updates,     
devtools, and background refetch in a single package with no boilerplate store setup. RTK Query would require     
Redux infrastructure; SWR lacks first-class mutation handling. |
| Axios | Fetch API | Axios provides request/response interceptors as a first-class feature, which is essential   
for the token-refresh queue pattern. The native Fetch API requires manual wrapper code to achieve equivalent      
behaviour. |
| sessionStorage for access token | localStorage | sessionStorage is scoped to a single tab and cleared on close,
reducing the window of exposure if XSS occurs. localStorage persists across sessions and all tabs, making stolen  
tokens longer-lived. |
| Cookie for refresh token | sessionStorage / localStorage | Cookie with `SameSite=Strict; Secure` is not readable
by JavaScript if `HttpOnly` is set server-side, providing the strongest XSS protection for the long-lived        
credential. |
| Hexagonal (Ports & Adapters) | Flat component → fetch | Clean substitutability between mock and real adapters   
without conditional logic scattered through components. Also makes the domain model explicit and independently    
testable. |
| Tailwind CSS v4 | CSS Modules / styled-components | Utility-first CSS eliminates context switching between files
for styling. Tailwind v4 ships as a Vite plugin with no PostCSS config. No runtime JS overhead (no CSS-in-JS). |
| Vitest | Jest | Vitest is native to the Vite build pipeline; shares the same config, transformation, and alias
resolution. Jest requires separate Babel/ts-jest config and does not understand Vite's `@` alias natively. |      
| i18next | react-intl / no i18n | Play4Change targets European learners across 9 languages. i18next with       
`i18next-browser-languagedetector` provides automatic locale detection, JSON translation files per language, and a
mature React binding. |                                                                                        
| Cloudflare Pages | Vercel / Netlify | Cloudflare Pages offers global CDN distribution, free tier, and is already
used for the project's infrastructure. No per-seat pricing at the CDN layer. |

## Security Considerations

**Access token in `sessionStorage`:** Readable by any JavaScript at the same origin.                              
If a stored XSS payload executes, it can exfiltrate the access token. Mitigated by the
short token lifetime and tab-scoped storage, but not fully eliminated without a BFF                               
(Backend For Frontend) pattern that keeps the token server-side.

**Refresh token in cookie:** The frontend sets the cookie via `document.cookie` with                              
`SameSite=Strict; Secure`. Whether `HttpOnly` is enforced depends entirely on the backend                       
response header — the frontend cannot read an `HttpOnly` cookie, so it stores the refresh                         
token in the cookie itself only if the backend echoes it back in the body. This coupling                          
should be reviewed: if the backend issues `HttpOnly` cookies directly, the frontend                               
should remove its `document.cookie` write for the refresh token.

**No Content-Security-Policy:** Neither `index.html` nor the Cloudflare Pages deployment                          
config defines a CSP. This means a successful XSS can load arbitrary scripts. A                                 
`Content-Security-Policy` header restricting `script-src` to `'self'` should be added                             
via Cloudflare Pages `_headers` file.

**`i18next` `escaping: false`:** Translation interpolation is not HTML-escaped. All                               
current translation values are static developer-controlled strings, so this is safe                             
today. If user-generated content is ever placed into a translation variable, it must                              
be sanitised before insertion or `escaping` must be re-enabled.

**No `dangerouslySetInnerHTML`:** A search of the codebase finds no usage. XSS from                               
direct HTML injection is not a current risk.

**CORS:** Delegated entirely to the backend API. The frontend has no influence over                               
cross-origin policy.

**Magic-link token in URL:** The token appears in `?token=…` on the verify page. It                               
lands in browser history and server access logs. Backend should expire tokens after
first use with a short TTL (recommended: 15 minutes, single-use).

**No rate limiting on magic-link form:** The frontend submits POST `/auth/magic-link`                             
on every form submission with no debounce or cooldown. Backend must enforce rate                                  
limiting per email address.

## Consequences

**Easier because of these choices:**
- Swapping mock ↔ real adapters for development or testing requires one env-var change.
- Domain models and port contracts are the single source of truth; backend contract                               
  changes surface as TypeScript compile errors throughout the codebase.
- TanStack Query handles loading, error, and stale states uniformly; no manual                                    
  `isLoading` flags or `try/catch` in components.
- Vitest and Vite share config; no separate test runner configuration to maintain.
- Cloudflare Pages deploy is a single `wrangler pages deploy dist` command already                                
  wired into CI.

**Harder because of these choices:**
- `sessionStorage` for access tokens means the user is logged out when they open a                              
  new tab — mitigated by `BroadcastChannel` propagation of login events, but a new                                
  blank tab opened before login will require a re-auth dance.
- No SSR means the public landing page is not indexable by search engines without a                               
  separate prerendering step or a service-worker-based solution.
- Hexagonal layering adds indirection (domain → port → adapter → hook → component);                               
  simple changes touch more files than a flat architecture would.

**Must be revisited at scale or in production:**
- Add `_headers` file for Cloudflare Pages to emit `Content-Security-Policy`,                                     
  `X-Frame-Options`, `X-Content-Type-Options`, and `Referential-Policy` headers.
- Clarify `HttpOnly` cookie ownership for the refresh token (frontend cookie write vs.                            
  backend `Set-Cookie`).
- Complete OAuth flows for Google and Facebook (currently placeholder buttons).
- Add topic edit functionality (no route or form exists for updating topic metadata).
- Add E2E tests (Playwright) covering the magic-link auth flow and topic creation path.
- Enable `i18next` interpolation escaping or audit all interpolated values before any                             
  user-generated content reaches translation strings.
- Remove `EnrollmentSummary` from `src/domain/models/Enrollment.ts` or implement the                              
  full feature.
