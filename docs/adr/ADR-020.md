## Context

Phase 04 requires replacing all mock repositories in the KMP mobile client with real HTTP
implementations. Before any repository can make authenticated requests, three foundational
concerns must be resolved:

1. **Token storage:** The server issues short-lived access tokens (15 min) and long-lived
   refresh tokens (7 days). The refresh token must survive process death and persist across
   app restarts. Storing it in plain SharedPreferences or UserDefaults would expose it to
   any app with READ_EXTERNAL_STORAGE or to a jailbroken device filesystem read.

2. **Token refresh transparency:** Every API call that receives a 401 must silently refresh
   the token pair and retry the original request. Propagating 401s to the UI layer would
   force every feature screen to implement its own refresh logic â€” a maintenance and
   correctness risk.

3. **HTTP engine configuration:** The app targets Android and iOS from a shared KMP module.
   The Ktor `HttpClient` must be configured once in `commonMain` but backed by a
   platform-appropriate engine (OkHttp on Android, Darwin on iOS). Tests must be able to
   inject a `MockEngine` without touching production code.

---

## Decision

**Store tokens in EncryptedSharedPreferences (Android) and Keychain (iOS); configure a
single Ktor `HttpClient` in `commonMain` with the `Auth` bearer plugin handling the
401â†’refreshâ†’retry cycle transparently; inject the platform engine via `expect/actual` and
allow test overrides via an optional `engine` parameter.**

## Why This, Not Something Else

### Token storage

| Alternative | Why rejected |
|-------------|-------------|
| Plain `SharedPreferences` / `UserDefaults` | Values are stored as plaintext on the filesystem. Any process with sufficient permissions can read them. Violates OWASP A02. |
| In-memory only (no persistence) | Refresh token is lost on process death. Every app restart requires re-authentication via magic link or OAuth â€” unacceptable UX. |
| `multiplatform-settings` library (encrypted mode) | Wraps EncryptedSharedPreferences on Android but uses `NSUserDefaults` on iOS by default (not Keychain). Not suitable for secrets without additional platform bridges. |
| Room/SQLCipher database | Correct security properties but overkill for two string values. Adds a dependency and migration surface for a two-field store. |

### Token refresh

| Alternative | Why rejected |
|-------------|-------------|
| Propagate 401 to each ViewModel and handle there | Every feature must implement refresh logic. Race conditions when multiple requests fail simultaneously. Each screen must know about token state â€” breaks separation of concerns. |
| Periodic proactive refresh (timer-based) | Adds background work, battery impact, and complexity. Does not eliminate 401s (tokens can be revoked externally). |
| Manual refresh call before each request | Requires clock synchronization with server token expiry. JWT `exp` claim parsing on client introduces dependency on JWT library just for token inspection. |

### HTTP engine abstraction

| Alternative | Why rejected |
|-------------|-------------|
| Separate `HttpClient` per platform module | Duplicates `ContentNegotiation`, `Auth`, timeout, and `defaultRequest` configuration. Drift between platforms is a correctness risk. |
| OkHttp for both Android and iOS (via JVM interop) | Kotlin/Native does not run on the JVM. OkHttp is JVM-only. |
| `HttpClient()` with no engine argument (auto-resolve) | Works in some Gradle configurations but is fragile and not recommended for KMP modules with multiple targets. Explicit engine declaration makes the dependency graph clear. |

## Consequences

**Easier:**
- All authenticated HTTP calls get silent token refresh with no per-feature code.
- Tests inject `MockEngine` without changing production configuration.
- Platform token storage is encapsulated behind `TokenStorage` â€” feature code never calls
  platform APIs directly.
- `SessionEventBus` provides a single observable stream for session expiry across all screens.

**Harder:**
- The `EncryptedSharedPreferences` MasterKey creation is synchronous and must not be called
  on the main thread at high frequency (lazy initialization mitigates this).
- iOS Keychain access requires `@OptIn(ExperimentalForeignApi::class)` due to Kotlin/Native
  C interop â€” this must be maintained as Kotlin/Native matures.
- The `refreshTokens` lambda in Ktor's `Auth` plugin makes a direct HTTP call to
  `/auth/refresh`. This call bypasses `AuthRepository`, so the refresh endpoint wire format
  (request body: `{"refreshToken":"..."}`, response: `{"accessToken","refreshToken","expiresIn"}`)
  is duplicated in `HttpClientFactory`. If the server changes this contract, both places must
  be updated.

**Must revisit when:**
- `androidx.security:security-crypto` reaches stable 1.1.x (switch from `MasterKeys` legacy
  API to `MasterKey.Builder` for cleaner key lifecycle management).
- Kotlin/Native stabilises its Security framework interop (remove `ExperimentalForeignApi`).
- The app adds certificate pinning (requires `OkHttp` `CertificatePinner` configuration in
  the Android actual engine factory).

## Security Considerations

**OWASP A02 â€” Cryptographic Failures:**
- Android: `EncryptedSharedPreferences` uses AES256-GCM for values and AES256-SIV for key
  names. The master key lives in the Android Keystore hardware-backed enclave and never
  leaves it.
- iOS: Keychain items are stored with `kSecAttrAccessibleAfterFirstUnlock`, which keeps them
  inaccessible until the first device unlock after boot. This is the correct setting for
  background-accessible secrets; `kSecAttrAccessibleAlways` would be weaker.
- Access tokens are NOT persisted to disk. They are loaded from storage on app launch via
  the `loadTokens` callback and held in Ktor's in-memory bearer token cache. Only the refresh
  token is stored persistently.

**OWASP A07 â€” Auth Failures:**
- Refresh token reuse detection is handled server-side (ADR-011). If a stolen refresh token
  is used, the server revokes the entire token family and the client receives a non-2xx on
  the refresh call, triggering `onSessionExpired` and storage clear.
- `markAsRefreshTokenRequest()` prevents recursive 401 handling on the refresh call itself.

**Residual risk:** The refresh token is stored on the device. If the device is compromised
at the OS level (root/jailbreak) after the first unlock, the token is extractable. This is
an accepted risk common to all mobile authentication schemes that survive process death.
Full mitigation requires hardware-bound keys (Android StrongBox / iOS Secure Enclave),
which is out of scope for this phase.
