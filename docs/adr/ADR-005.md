# ADR-005 — Adaptive Learning Path Architecture

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind

---

## Context

The application delivers a fixed learning path (like Candy Crush levels) where
every user follows the same sequence of tasks. However, learners have different
difficulties at different points. A purely fixed path abandons struggling users.
A fully personalised path is expensive and architecturally complex.

The challenge: how do you support personalised help for struggling users without
abandoning the predictability and cost efficiency of the fixed path?

---

## Decision

**Fixed path as the spine. Adaptive branches as temporary detours.**

```
Fixed Path:  [Task 1] → [Task 2] → [Task 3] → [Task 4] → [Task 5]
                                       ↓ struggle detected
Adaptive:               [SubTask A] → [SubTask B] → [SubTask C]
                                       ↓ objective achieved
Fixed Path resumes:                                    → [Task 4]
```

The user always returns to the fixed path once the struggle is resolved.
The branch is temporary scaffolding, not a permanent fork.

---

## Course Lifecycle

```
CREATED → SUBSCRIPTION_OPEN → SUBSCRIPTION_CLOSED → IN_PROGRESS → COMPLETED
```

1. **Admin creates course** — defines structure (modules, topics) and subscription window
2. **Subscription window** — users can join; after window closes, no new subscribers
3. **In progress** — all subscribers work through tasks simultaneously (1 per day)
4. **Completed** — last subscriber completes the final module; course archived to statistics

The course ends when the **last subscribed user** completes it — not on a fixed date.
This means the course lifetime is bounded by the slowest learner, which adaptive
branching directly helps by accelerating struggling users.

---

## Struggle Detection

A struggle event is recorded when:
- User fails the same task more than `N` times (configurable per course)
- User explicitly requests help
- User has not attempted a task within the expected window

`StruggleDetectionService` evaluates these signals and produces a `StruggleContext`:

```kotlin
data class StruggleContext(
    val userId: String,
    val taskId: String,
    val moduleId: String,
    val courseId: String,
    val attemptCount: Int,
    val errorPattern: String,    // categorised error type
    val domainContext: String,   // course subject for AI context
)
```

## Adaptive Branch Generation — Similarity-Based Reuse

Before generating new content, the system queries pgvector for similar past struggles.
This is the key cost-control mechanism — the adaptive generation library grows over
time, reducing the need for fresh LLM calls.

```
Query pgvector with struggle embedding
         │
         ├── similarity > 0.90  →  reuse existing branch entirely
         │                          (zero LLM cost)
         │
         ├── similarity > 0.65  →  merge existing + generate complement
         │                          (partial LLM cost)
         │
         └── similarity < 0.65  →  generate fresh branch
                                    store with embedding for future reuse
                                    (full LLM cost, amortised over future users)
```

**Why this threshold approach:**
- Single algorithm, three natural outcomes driven by data
- No special casing or conditional generation logic
- Thresholds are configurable — tunable with real usage data over time
- Cost decreases as pattern library grows — the system gets cheaper to run

**Why pgvector for similarity:**
- Already in the stack for batch deduplication
- Vector similarity is the correct primitive for semantic struggle matching
- "User confused about concept X" and "User confused about concept X variant"
  should match — keyword search would not catch this

## Data Model (key entities)

```
courses
  id, title, subject_domain, subscription_opens_at, subscription_closes_at,
  status (CREATED | SUBSCRIPTION_OPEN | SUBSCRIPTION_CLOSED | IN_PROGRESS | COMPLETED)

course_modules
  id, course_id, order_index, title, topic, objective

task_templates (fixed path — AI generated from topics at course creation)
  id, module_id, order_index, title, description, hint, points_reward,
  embedding (vector)

user_course_enrollments
  id, user_id, course_id, enrolled_at, completed_at, current_task_template_id

user_task_progress
  id, user_id, task_template_id, status, attempt_count, completed_at

struggle_events
  id, user_id, task_template_id, error_pattern, attempt_count,
  embedding (vector), created_at

adaptive_branches
  id, struggle_event_id, status (ACTIVE | COMPLETED | ABANDONED),
  reused_from_branch_id (nullable — null means freshly generated)

adaptive_tasks
  id, branch_id, order_index, title, description, hint, points_reward
```

## At Scale Considerations

| Scale | Fixed Path | Adaptive Branches |
|-------|-----------|------------------|
| 1,000 users | Batch covers all | Low volume, fast |
| 100,000 users | Batch unchanged | pgvector reuse dominant |
| 1,000,000 users | Partitioned batch | Dedicated branch generation workers |

The fixed path cost curve is flat (batch scales with courses, not users).
The adaptive cost curve grows sublinearly as the pgvector library accumulates.

---

## Consequences

- `StruggleDetectionService` is a critical path component — must be fast
- pgvector is load-bearing for both deduplication AND similarity-based reuse
- Similarity thresholds (0.90, 0.65) are initial values — must be monitored
  via Prometheus and tuned with real data after launch
- Course completion depends on all users — adaptive branching directly impacts
  the course end date by helping slow learners accelerate
