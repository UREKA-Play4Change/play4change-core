# ADR-021 — Full-Stack Flow Audit: Auth, Enrollment, Task Lifecycle, and Struggle Detection

**Status:** Informational (audit record — not a decision)
**Date:** 2026-05-12
**Author:** Radesh Govind (A51620)
**Scope:** All flows from mobile client to server for Phase 01–04 implemented features
**Method:** Static code analysis (all layers) + agentic/manual-testing/* execution on
           iPhone 16e iOS 26.2 simulator against local Docker Compose stack

---

## Why This Exists

Phase 05 is starting. Before adding push notifications and background fetch on top of this
thing, someone needs to actually document what the hell is happening end-to-end — what goes
over the wire, where the handshakes are, what is hardcoded, what is broken by design, and
what we got right. This document is that record.

A future engineer reading this should be able to reconstruct the full system behaviour without
reading 40 source files. If they can't, this document failed.

## 1. Authentication Flow

### 1.1 Magic Link — Full Protocol Trace

```
[Mobile App]                    [Server]                    [Email Provider]
    |                               |                               |
    |-- POST /auth/magic-link ---->>|                               |
    |   body: { "email": "..." }    |                               |
    |                               | generate 32-byte raw token    |
    |                               | hash = SHA-256(raw)           |
    |                               | save MagicLinkToken{          |
    |                               |   token=hash,                 |
    |                               |   email=normalised,           |
    |                               |   expiresAt=now+15min,        |
    |                               |   used=false }                |
    |                               |-- sendMagicLink(email,  -->>  |
    |                               |   baseUrl/auth/verify?        |
    |                               |   token=rawToken)             |
    |<<-- 202 Accepted -------------|                               |
    |   body: { "message": "..." }  |                               |
    |                               |                               |
    | [user clicks email link]      |                               |
    |                               |                               |
    [Browser/Email client]          |                               |
    |-- GET /auth/verify?token= -->>|                               |
    |   token=rawToken              |                               |
    |<<-- 302 Found ----------------|                               |
    |   Location: play4change://    |                               |
    |   auth/verify?token=rawToken  |                               |
    |                               |                               |
    | [iOS intercepts deep link,    |                               |
    |  opens Play4Change app]       |                               |
    |                               |                               |
[Mobile App]                        |                               |
    |-- POST /auth/magic-link/ -->> |                               |
    |   verify                      |                               |
    |   body: { "token": rawToken } |                               |
    |                               | hash = SHA-256(rawToken)      |
    |                               | atomic CTE UPDATE:            |
    |                               |   SET used=true               |
    |                               |   WHERE token=hash            |
    |                               |     AND used=false            |
    |                               |     AND expiresAt > now()     |
    |                               | returns email or null         |
    |                               | if null → 401                 |
    |                               | upsert User                   |
    |                               | issue(userId, email, role)    |
    |                               |   → accessToken (JWT, 15min)  |
    |                               |   → rawRefresh (SHA-256 hash  |
    |                               |     stored, raw returned)     |
    |                               |   → familyId (UUID)           |
    |<<-- 200 OK ------------------|                               |
    |   { accessToken, refreshToken,|                               |
    |     expiresIn: 900 }          |                               |
    | tokenStorage.store(AT, RT)    |                               |
    | [Keychain on iOS]             |                               |
```

**What's good:**
- SHA-256 hash stored, raw token only in email link and POST body — correct
- Atomic CTE UPDATE closes TOCTOU race (ADR-016 G2)
- 15-minute TTL enforced at DB level, single-use enforced atomically
- GET /auth/verify is redirect-only, no token consumption there — fixed after B10

**What's broken / inconsistent:**

**BUG 1 — baseUrl points to the admin web, not the server.**
`application.yml`:
```yaml
app:
  base-url: ${APP_BASE_URL:http://localhost:5173}
```
`http://localhost:5173` is the React admin web dev server. The magic link email URL is
`http://localhost:5173/auth/verify?token=...`. That endpoint does not exist there.
On a real deployment, unless `APP_BASE_URL` is set to the server public URL
(e.g. `https://api.play4change.app`), every magic link sends users to the wrong host.
The GET /auth/verify endpoint is on the Spring server at 8080, not at 5173.
**This is a P1 for any real deployment.** In the local Docker stack it works only because
docker-compose sets APP_BASE_URL, but the default in application.yml is a foot-gun.

**BUG 2 — no input validation on POST /auth/magic-link.**
`AuthController.requestMagicLink` has no `@Valid` annotation and no `@field:Email` on
`MagicLinkRequest.email`. The email is normalised (`lowercase().trim()`) but otherwise
passed directly to the DB and email port. An empty string, a SQL fragment, or
`"not@valid"` all reach `magicLinkTokenRepository.save()`. Tracked as R16 in THREAT-LOG,
planned for Phase 07. In the meantime, Postgres column constraint is the only guard.

**BUG 3 — debug token paste path (iOS) described incorrectly in phase-04-recipe.md.**
Section 1 says "Extract the raw token from server logs." The current email adapter
(ResendEmailAdapter) does NOT log the token — it sends it to Resend and the token is gone
from the server side. The ConsoleEmailAdapter logs it but is not active when RESEND_API_KEY
is set. The recipe treats these as two paths of equal relevance. In practice on this dev
setup (Resend active), the only path is:
1. Check real inbox for the link
2. Extract `?token=<RAW>` from the URL in the email
3. Paste into the debug field in the app
The log-scraping path does not work with Resend. The recipe is misleading.

### 1.2 Token Storage — iOS

Service key: `com.ureka.play4change.tokens`
Keys: `access_token`, `refresh_token`
Keychain class: `kSecClassGenericPassword`
Accessibility: `kSecAttrAccessibleAfterFirstUnlock`
Required flag: `kSecUseDataProtectionKeychain = kCFBooleanTrue` (mandatory on iOS 26.2
              simulator — System Keychain disabled via feature flag)

Strategy: add-or-update (SecItemAdd, fall back to SecItemUpdate on errSecDuplicateItem).
Any other OSStatus throws `IllegalStateException`. This is correct and was the fix for B7.

**No memory-only fallback.** If the Keychain throws on save, the caller crashes. This is
intentional (fail fast > silently proceed with no tokens), but there is no error surfaced
to the UI. The Koin DI wires `KeychainTokenStorage` as a singleton. If the first save
fails, the user sees a crash, not an error screen.

### 1.3 HTTP Client — Bearer Token Plugin (Ktor 3.0)

```
Ktor Auth plugin: bearer {
    loadTokens { ... }        // reads from Keychain/EncPrefs
    refreshTokens { ... }     // calls POST /auth/refresh
    sendWithoutRequest { true } // ← documented trap
}
```

**`sendWithoutRequest { true }` does NOT proactively inject the Authorization header.**
In Ktor 3.0, this flag means "don't wait for a 401 to start the auth flow," but
`addRequestHeaders`/`loadTokens` is still not called before the first request in a new
session. The first authenticated request after login goes out without `Authorization`,
receives 401, and then `refreshTokens` is invoked.

This is exactly what caused B9 (Ktor 3.0 `sendWithoutRequest` behaviour). The fix
(`oldTokens?.refreshToken ?: tokenStorage.getRefreshToken()`) is in place. But the
underlying problem remains: **every first request after login costs an extra round-trip**.
User logs in, navigates to Home, fires 3 concurrent requests (profile + today-task +
roadmap), all three go out without auth, all three get 401, and then B11 kicks in:
concurrent refresh race. The Mutex fix is scheduled for Phase 05 Task 5.1 (pre-task).

**BUG 4 (B11) — no Mutex in refreshTokens.**
Three concurrent 401s → three concurrent `refreshTokens` invocations → first one succeeds,
rotates the refresh token → second one presents the rotated (now-used) token → server
detects reuse, revokes entire family → `sessionExpired()` fires → user is logged out.
This happens ~every time the access token expires while the home screen is loading.
With a 15-minute TTL, that's at least once per average usage session.
Fix is documented: wrap `refreshTokens` body in a `Mutex`, check if stored access token
already differs from `oldTokens.accessToken` (means another coroutine already refreshed).

### 1.4 Token Refresh Protocol

```
[Mobile App]                    [Server]
    |-- POST /auth/refresh ----->>|
    |   { "refreshToken": raw }   |
    |                             | hash = SHA-256(raw)
    |                             | find RefreshToken by hash
    |                             | if not found → 400
    |                             | if used=true:
    |                             |   revokeAllByFamilyId()  ← theft detection
    |                             |   throw SecurityException → 401
    |                             | if expired → 400
    |                             | markUsed(stored.id)
    |                             | generate new access + refresh tokens
    |                             | save new RefreshToken (same familyId)
    |<<-- 200 OK -----------------|
    |   { accessToken, refreshToken, expiresIn }
    | tokenStorage.store(AT, RT)  |
```

Refresh token rotation is implemented correctly. Family-based revocation on reuse is
correct. The only issue is the lack of Mutex on the client causing the reuse in the
first place (B11).

### 1.5 Logout

Client calls `POST /auth/logout` (not DELETE — `HttpAuthRepository.logout()` uses POST).
Server has both `@DeleteMapping("/logout")` and `@PostMapping("/logout")` — both call
`tokenUseCase.revoke(request.refreshToken)`. The DELETE variant is described in the
server code as "kept for demo/CLI clients." The client uses POST. This is fine, but the
two handlers doing the same thing is dead code debt.

**BUG 5 — logout clears storage before server call.**
`HttpAuthRepository.logout()`:
```kotlin
val refreshToken = tokenStorage.getRefreshToken()
tokenStorage.clear()   // ← tokens gone from Keychain
if (refreshToken != null) {
    runCatching { client.post("/auth/logout") { ... } }
}
```
If the server call fails (network down), the local tokens are cleared but the server's
refresh token is not revoked. The server token remains valid for up to 7 days. A stolen
device with a cached refresh token could still authenticate. Correct order: call server
first, then clear local storage only on success. The `runCatching` swallows all errors,
so logout always appears to succeed regardless.

## 2. Topic Creation

### 2.1 Protocol Trace

```
[Admin client]                  [Server]                    [Mistral AI]
    |-- POST /admin/topics --->> |                               |
    |   { title, description,    |                               |
    |     category, urls[],      |                               |
    |     durationDays,          |                               |
    |     difficulty, language,  |                               |
    |     taskCount }            |                               |
    |                            | SSRF check on URLs:           |
    |                            |   scheme must be https        |
    |                            |   no RFC1918/loopback         |
    |                            |   DNS resolution then IP check|
    |                            | create Topic{status=INGESTION}|
    |                            | save to DB                    |
    |<<-- 201 Created ------------|                               |
    |   { id, title,             |                               |
    |     currentPhase: INGESTION|                               |
    |     ... }                  |                               |
    |                            |                               |
    |    [async, generationExecutor pool, 3 threads, queue 25]   |
    |                            |                               |
    |                            | INGESTION:                    |
    |                            |   fetch URL(s) content        |
    |                            |   → ANALYSIS                  |
    |                            | ANALYSIS:                     |
    |                            |   classify content            |
    |                            |   → GENERATION                |
    |                            | GENERATION:                   |
    |                            |-- generateTasks(context) -->> |
    |                            |                               | Mistral call
    |                            |                               | model: mistral-small-latest
    |                            |                               | temp: 0.7
    |                            |                               | timeout: 60s
    |                            |<<- tasks JSON ----------------|
    |                            | AiOutputSanitiser:            |
    |                            |   jsoup strip HTML            |
    |                            | schema validate               |
    |                            | retry once on failure         |
    |                            | persist TaskTemplates         |
    |                            | generate TaskInstances        |
    |                            |   (5 per template by default) |
    |                            | → INDEXING                    |
    |                            | INDEXING:                     |
    |                            |   build pgvector embeddings   |
    |                            |   → ACTIVE                    |
    |                            | create MicroCompetence        |
    |                            | (badge prerequisite)          |
```

**What's good:**
- SSRF validation blocks non-HTTPS and RFC1918 (R06 partially mitigated)
- AI output sanitised with jsoup before DB persistence (R07 fixed)
- Schema validation with single retry on failure
- 5 task instances per template (dedup thresholds: 0.90 full-reuse, 0.65 partial-reuse)

**What's wrong / inconsistent:**

**BUG 6 — generation pool is too small for concurrent users.**
`pool-size: 3`, `queue-capacity: 25`. If 4 admins create topics simultaneously, the 4th
blocks. If 26 queue up, the 27th gets `RejectedExecutionException`. There is no HTTP-level
response for "the queue is full" — the request to POST /admin/topics returns 201 immediately
(the async task is fire-and-forget). The topic status would show INGESTION indefinitely.
This is not guarded. The watchdog catches stuck topics at `stuck-threshold-minutes: 5`.
But the 27th topic never gets a worker and the watchdog doesn't know to retry it because the
DB record exists (status=INGESTION) and the watchdog thinks it's being processed.

**HARDCODED — Mistral model.**
`model: mistral-small-latest` is in application.yml but hardcoded as a default. There is
no env var override for the model. If Mistral deprecates `mistral-small-latest` without
notice (they do this), redeployment requires a config change + rebuild. Should be:
```yaml
ai:
  mistral:
    model: ${MISTRAL_MODEL:mistral-small-latest}
```

**BUG 7 — MicroCompetence creation timing is ambiguous.**
`BadgeIssuanceService.issueBadge()` calls `badgeRepository.findMicroCompetenceByTopicId(topicId)`.
If `MicroCompetence` doesn't exist yet (topic just became ACTIVE and the INDEXING→ACTIVE
transition creates it), there's a race: `issueBadge` is called synchronously after
`submitAnswer`, which could happen milliseconds after the topic goes ACTIVE. If the
MicroCompetence hasn't been committed yet, `issueIfNotYetEarned` returns silently
(`val microCompetence = badgeRepository.findMicroCompetenceByTopicId(topicId) ?: return`).
Badge is never issued. The user has to wait for a second correct submission that will never
come (all tasks already done). This is unobservable to the user — they just don't get a badge.

## 3. Topic Subscription (Enrollment)

### 3.1 Protocol Trace

```
[Mobile App]                    [Server]
    |-- POST /topics/{id}/enroll >>|
    |   Authorization: Bearer AT  |
    |                             | find Topic by id
    |                             | guard: status == ACTIVE
    |                             | guard: !topic.isExpired()
    |                             | guard: no existing enrollment
    |                             | find TopicModule (first one)
    |                             | find TaskTemplates for module
    |                             | guard: templates not empty
    |                             | create Enrollment{
    |                             |   enrolledAt = now,
    |                             |   currentDayIndex = 0,
    |                             |   totalPointsEarned = 0,
    |                             |   streakDays = 0 }
    |                             | create TaskAssignment{
    |                             |   dayIndex 0 template (minByOrNull),
    |                             |   dueAt = now.plusHours(24),  ← BUG I03
    |                             |   shuffledOrder = SHA-256 seed }
    |<<-- 200 OK -----------------|
    |   { id, topicId, status:    |
    |     ACTIVE, currentDayIndex: 0, ... }
```

**BUG 8 (I03) — first assignment dueAt is wrong.**
`dueAt = now.plusHours(24)` in `EnrollmentService`. Every other assignment uses
`DayIndexCalculator.startOfTomorrow(timezone)` (correctly computes midnight in the user's
timezone). The first assignment is 24 hours from enrollment time, not midnight. A user
enrolling at 11 PM has until 11 PM the next day, not midnight. The late-submission
half-point penalty is incorrectly not applied in edge cases. Low severity but it's an
inconsistency that will bite someone during Phase 07 when they read the audit trail.

**BUG 9 — double-enrollment returns 409 Conflict.ConcurrentModification.**
`Conflict.ConcurrentModification` is the wrong error type for a duplicate enrollment.
This is a semantic error: `ConcurrentModification` implies a race condition, not a business
rule violation. The client receives a 409 with a misleading error code. Should be a
dedicated `AlreadyEnrolled` error that maps to 409 with a clear message. Not a crash-level
bug but confusing when debugging.

**BUG 10 — no enrollment exists for topics with taskCount=0.**
`EnrollmentService` guards `templates.isNotEmpty()` before creating the enrollment. But
`Topic.taskCount` is set at creation time and can be inconsistent with the actual number
of generated templates. If Mistral generates 0 valid tasks (all fail schema validation),
the topic goes ACTIVE with 0 templates. Any enrollment attempt then returns 400
"topic has no tasks yet" — correct guard, but the error message says "no tasks yet"
implying a temporary state, when in fact the topic is permanently broken. No admin endpoint
to mark a topic as FAILED after it reaches ACTIVE.

**HARDCODED — only the first TopicModule is used.**
`modules.firstOrNull()` in both `EnrollmentService` and `TaskService`. The data model
supports multiple modules per topic but the business logic always picks the first one.
If a topic ever gets a second module (e.g. from an admin correction), learners will
never see it. This is a silent feature limitation, not documented in the topic creation API
or the client.

## 4. Task Lifecycle (Today's Task → Submit)

### 4.1 GetTodayTask Protocol

```
[Mobile App]                    [Server]
    |-- GET /tasks/today ------>> |
    |   ?topicId=<id>             |
    |   Authorization: Bearer AT  |
    |   X-Timezone: Europe/Lisbon |
    |                             | find Enrollment(userId, topicId)
    |                             |
    |                             | [prod mode: timezone-based]
    |                             | dayIndex = enrolledAt → now, offset by timezone
    |                             | find assignment for dayIndex
    |                             | if found && PENDING → return it
    |                             | if found && SUBMITTED → 404 + X-Task-Available-At
    |                             |
    |                             | [no assignment for dayIndex yet]
    |                             | resolve user.preferredLanguage
    |                             | language gating check
    |                             | if language gen pending → 202
    |                             | select TaskInstance (shuffle-seed based)
    |                             | shuffle option order (SHA-256 seed)
    |                             | create TaskAssignment
    |<<-- 200 OK -----------------|
    |   { assignmentId,           |
    |     title, description,     |
    |     options[],              |  ← shuffled by optionOrder
    |     correctAnswer: null,    |  ← NEVER sent to client
    |     pointsReward,           |
    |     dueAt,                  |
    |     wrongAttemptCount: 0,   |
    |     taskTemplateId }        |
```

**What's good:**
- Options are shuffled per user per task (SHA-256 seed = userId + taskId + enrollmentId)
- Correct answer never sent in the response
- Language gating correctly returns 202 + headers when generation is pending
- B5 fixed: SUBMITTED tasks return 404, not 200

**BUG 11 — options in response are not re-mapped through shuffled order.**
The `optionOrder` list is `[2, 0, 3, 1]` (original indices in shuffled positions).
The server stores the template's canonical option list. The client-facing response must
re-order options so that `response.options[0]` = `template.options[optionOrder[0]]`.

Reading `TaskService.getTodayTask()`: options from the template are returned directly.
The `optionOrder` is stored in the assignment but the re-ordering of the option strings
for the response DTO needs to be verified — the response DTO must apply the shuffle before
serialising. If it doesn't, the client sees unshuffled options, the stored `optionOrder`
is mismatched, and `submitAnswer` will evaluate the wrong answer.

Checking the web controller layer response is critical here. **This needs to be verified
in the live test against the actual wire format.** If the options in the API response
are NOT shuffled but `selectedOption` is interpreted as an index into the shuffled order,
users always see canonical option 0 first and the "correct" answer changes per user.

**HARDCODED — task instance selection uses modulo.**
`taskInstanceRepository.findByTaskTemplateId(templateId)` returns all instances, then
`instances[Math.floorMod(seed, instances.size)]` selects one. The same user always gets
the same instance for the same task (deterministic). This is intentional for anti-cheat.
But the instance list is ordered by DB insertion order, not shuffled. The modulo over
seed selects a pseudo-random instance per user, which is correct. But there is no
documented guarantee that `findByTaskTemplateId` returns results in stable order across
restarts. If Postgres returns them in a different order (it can), the same user could
get a different instance after a DB restart. The seed selection should use ORDER BY id
to ensure stability. Currently there is no ORDER BY in the JPA query.

### 4.2 SubmitAnswer Protocol

```
[Mobile App]                    [Server]
    |-- POST /tasks/{id}/submit >>|
    |   { "selectedOption": 2 }   |
    |   Authorization: Bearer AT  |
    |                             | find TaskAssignment by id
    |                             | guard: assignment.userId == caller
    |                             | guard: status == PENDING
    |                             | guard: selectedOption >= 0
    |                             |
    |                             | find TaskTemplate
    |                             | originalIndex = optionOrder[selectedOption]
    |                             | canonicalAnswer = instance.correctAnswer ?? template.correctAnswer
    |                             | isCorrect = originalIndex == canonicalAnswer
    |                             | isLate = now > dueAt
    |                             | pointsAwarded = isCorrect
    |                             |   ? (isLate ? reward/2 : reward)
    |                             |   : 0
    |                             |
    |                             | if wrongAttemptCount == 0 && !isCorrect:
    |                             |   copy(wrongAttemptCount=1, keep PENDING)
    |                             | else if wrongAttemptCount == 1 && !isCorrect:
    |                             |   classifyError(), triggerAsync(struggle)
    |                             |   markSubmitted(isCorrect=false)
    |                             | if isCorrect:
    |                             |   markSubmitted(isCorrect=true)
    |                             |
    |                             | enrollment.addPoints(pts).incrementStreak()
    |                             |   OR enrollment.resetStreak()
    |                             | badgeIssuancePort.issueBadge(...)
    |                             |
    |<<-- 200 OK -----------------|
    |   { isCorrect, pointsAwarded,|
    |     totalPoints, streakDays, |
    |     struggleTriggered }      |
```

**What's good:**
- Ownership check (`assignment.userId == caller`) — prevents cross-user submission
- Idempotency guard (`status == PENDING`) — prevents double-submission
- Late penalty correctly halves points
- `struggleTriggered` flag in response is correct (client can react immediately)

**BUG 12 — streak reset is wrong on second wrong answer.**
When the second wrong answer is submitted: `assignment.wrongAttemptCount == 1 && !isCorrect`.
The code calls `markSubmitted(isCorrect=false)`. Then:
```kotlin
} else if (updatedAssignment.status != AssignmentStatus.PENDING) {
    enrollment.resetStreak()
}
```
`updatedAssignment.status` is `SUBMITTED` (not PENDING) so `resetStreak()` is called.
But the enrollment streak should only reset if the user abandons the learning path, not
after triggering struggle (the struggle is a remediation, not a failure). After resolving
the struggle, the original assignment is reset to PENDING and the user must answer it again.
If they get it right after struggle, they get points but no streak increment — the streak
was already reset. This means struggle resolution doesn't restore the streak. It just stays
at 0 until the next correct answer on a different task.

**BUG 13 — badge issuance check uses `correctCount >= topic.taskCount`.**
```kotlin
val correctCount = assignments.count { it.isCorrect == true && it.status != AssignmentStatus.PENDING }
if (correctCount >= topic.taskCount) { issueIfNotYetEarned(...) }
```
`topic.taskCount` is the number requested at topic creation. If Mistral generates fewer
tasks (say 1 instead of 2 due to a schema validation failure), `topic.taskCount = 2` but
there's only 1 template. A user who correctly answers that 1 task never reaches
`correctCount >= 2` and never gets a badge. No error. No feedback. Badge just never appears.
This should compare against the actual number of templates in the module, not the requested
count.

## 5. Topic Completion (Badge Issuance)

### 5.1 Protocol Trace

```
submitAnswer → badgeIssuancePort.issueBadge(userId, topicId, enrollmentId)
    |
    | find Topic → get topic.taskCount
    | find assignments for enrollmentId
    | count(isCorrect==true && status!=PENDING)
    | if count >= taskCount:
    |   find MicroCompetence for topicId
    |   if not found → SILENT RETURN (BUG 7 above)
    |   if Badge already exists for (userId, microCompetenceId) → SILENT RETURN
    |   saveBadge(Badge{userId, microCompetenceId, earnedAt=now()})
    |   log.info("Badge issued")
```

**HARDCODED — badge issuance is synchronous in the submit-answer transaction.**
`badgeIssuancePort.issueBadge()` is called inside `submitAnswer`, which is a Spring
`@Transactional` context (inherited from the JPA save calls). If badge issuance fails
(DB exception, MicroCompetence not found causing NPE somewhere), the entire submit
transaction could roll back. The submit response would never reach the client even though
the answer was logically correct.

Actually, looking more carefully: `BadgeIssuanceService.issueBadge()` uses early returns
and logs warnings rather than throwing. So it won't roll back the parent transaction.
But it runs in the same thread and adds latency to the submit response.
**The submit endpoint response time = answer evaluation + streak update + badge check.**
For a topic with many enrollments, `enrollmentRepository.findAssignmentsByEnrollmentId()`
fetches all assignments every time any task is submitted. On a topic with 30 tasks and
1000 enrolled users, every submit loads 30 assignments per user. No pagination, no caching.

### 5.2 Roadmap Endpoint

After badge issuance, the roadmap shows node status. The `RoadmapService` maps:
- `SUBMITTED + isCorrect=true` → `COMPLETED`
- `SUBMITTED + isCorrect=false` → ??? (no explicit case — falls to `else → PENDING`)

A user who submitted the wrong answer twice (triggering struggle) has an assignment with
`status=SUBMITTED, isCorrect=false`. Their roadmap node shows `PENDING` after the struggle
session is triggered. That's arguably correct (they need to retry), but visually confusing:
the node looks like it hasn't been attempted.

## 6. Struggle Detection and Resolution

### 6.1 Trigger Protocol

```
submitAnswer (2nd wrong attempt)
    |
    | ErrorPatternClassifier.classify(assignment, selectedOption, template)
    |   → priority chain:
    |     TIME_PRESSURE: wrongAttemptCount > threshold && submittedAt near dueAt
    |     PARTIAL_UNDERSTANDING: selectedOption consistently off-by-one from correct
    |     READING_ERROR: consecutive wrong with same selectedOption
    |     WRONG_CONCEPT: default
    |
    | handleStruggleService.triggerAsync(...)
    |   @Async("generationExecutor") ← same pool as topic generation!
    |   save StruggleSession{status=OPEN, adaptiveTasks=[]}
    |   increment metric: struggle_sessions_total
    |   build StruggleContext{...}
    |   runBlocking { withTimeoutOrNull(60s) { generateAdaptiveBranch(context) } }
    |   if timeout → save session.abandon()
    |   if AI error → save session.abandon()
    |   if success:
    |     map subtasks → AdaptiveTask[]{
    |       optionOrder = (0..n).toMutableList().also { it.shuffle() }  ← BUG 14
    |     }
    |     save sessionWithTasks
    |
    | [client gets struggleTriggered=true in submit response]
    | [client polls GET /struggle/enrollment/{id}]
```

**BUG 14 — adaptive task option shuffle uses `java.util.Random()` without a seed.**
```kotlin
val shuffledOrder = (options?.indices?.toMutableList() ?: mutableListOf())
    .also { it.shuffle() }
```
`.shuffle()` with no argument uses `java.util.Random()` — different random per call,
not per-user. The shuffle is not deterministic. Two clients requesting the same adaptive
task could see different option orderings if the session is regenerated. More critically,
**the shuffle is applied at generation time, not at serve time**. The `optionOrder` is
stored in the DB. But the options in the response DTO must still be re-ordered using
`optionOrder` (same issue as BUG 11 for regular tasks).

**BUG 15 — struggle uses the same thread pool as topic generation.**
`@Async("generationExecutor")` — `pool-size: 3`. If 3 topic generations are running and
a user triggers struggle, the struggle task is queued. The queue capacity is 25. This
means: concurrent struggle generation is throttled behind topic generation. A user who
triggers struggle during a period of heavy admin activity (many new topics) could wait
minutes for their adaptive tasks. No timeout visible to the user — they just see a
loading state on the struggle screen. A separate executor pool for struggle generation
would fix this. It's a configuration change, not a code change.

**BUG 16 — struggle `errorPattern` mapping loses information.**
```kotlin
ErrorPattern.TIME_PRESSURE -> com.ureka.play4change.model.ErrorPattern.UNKNOWN
```
`TIME_PRESSURE` maps to `UNKNOWN` when passed to the AI generation context. Mistral
generates an adaptive branch without knowing the user had a time pressure issue. The
remediation content is generic rather than time-management focused. This is a silent
degradation of the struggle remediation quality.

### 6.2 Resolution Protocol

```
[Mobile App]                    [Server]
    |-- POST /struggle/{sessionId}/tasks/{taskId}/submit >>|
    |   { "selectedOption": 1 }   |
    |                             | find StruggleSession
    |                             | find AdaptiveTask in session
    |                             | mark task completed
    |                             | if all tasks completed:
    |                             |   session.resolve()
    |                             |   find originalTaskAssignment
    |                             |   reset to PENDING{
    |                             |     wrongAttemptCount=0,
    |                             |     selectedOption=null,
    |                             |     isCorrect=null,
    |                             |     submittedAt=null }
    |<<-- 200 OK -----------------|
    |   { updated AdaptiveTask }  |
```

**What's good:**
- Original assignment is reset to PENDING after all adaptive tasks are done (I02 fixed)
- Session status transitions: OPEN → RESOLVED (or ABANDONED on AI failure)

**BUG 17 — struggle adaptive task submit accepts any selectedOption value.**
Looking at the pattern from `TaskService.submitAnswer`: there is a guard
`ensure(command.selectedOption >= 0)`. The adaptive task submit in `AdaptiveTaskService`
needs the same guard. If it doesn't validate the index range, a client could submit
`selectedOption: 999` and the evaluation would try `optionOrder[999]` → IndexOutOfBoundsException.

**BUG 18 — GET /struggle/enrollment/{id} returns the last session, not the active one.**
If a user has multiple struggle sessions (triggered on different tasks across multiple
enrollments), the endpoint returns... whatever the repository returns. The repository
`findOpenByEnrollmentId(enrollmentId)` should only return OPEN sessions, but if there are
multiple OPEN sessions (theoretically possible if the first was never resolved), the
client would see an arbitrary one.

## 7. Wire-Level Observations

### 7.1 Headers

**Sent by mobile client on every request:**
- `Authorization: Bearer <accessToken>` (via Ktor Auth plugin)
- `Content-Type: application/json`
- `X-Request-ID: <random UUID>` (set in `defaultRequest`)

**Missing from mobile client:**
- `X-Timezone` — not sent by the mobile client at all. The server accepts it via
  `@RequestHeader(value = "X-Timezone", required = false)` in the task controller.
  Without it, `DayIndexCalculator.compute()` uses UTC as the fallback timezone. Any user
  not in UTC gets the wrong day index. **This is a silent data bug.** A learner in
  Lisbon (UTC+1) enrolling at 11 PM UTC sees dayIndex=1 (tomorrow in UTC) but the
  server computes dayIndex=0 (today in UTC because it's 11 PM UTC, midnight in Lisbon).
  The mobile client has `CurrentHour.ios.kt` to get the local hour but never passes the
  timezone in the request header.

### 7.2 Error Response Contract

The server returns different error shapes depending on the layer that throws:
- Spring `@ExceptionHandler` → `{ "message": "..." }` or `{ "error": "...", "details": [...] }`
- Arrow Either left side → mapped in controllers, inconsistent field names
- Unhandled exceptions → Spring default error format `{ "timestamp", "status", "error", "path" }`

The mobile client in `HttpAuthRepository` matches on HTTP status codes only, not on error
body content. Any 4xx from the server looks the same to the client regardless of whether
it's an invalid token, an expired token, or a validation error. The client error handling
is: 401 → Unauthorized, 403 → Forbidden, 5xx → ServerError, else → Unknown. No
differentiation between, say, 400 BadRequest (wrong email format) and 400 (token expired).

## 9. Decisions Record (Tactical)

These are choices that were made that a future session might reverse. Documenting here
so they don't get re-litigated without context.

**D1 — sendWithoutRequest{true} is kept despite the extra round-trip.**
Removing it would require implementing a custom `addRequestHeaders` block. The current
behaviour (first request gets 401, then refresh + retry) is accepted because it only
happens once per app session, not per request. The real fix is B11 (Mutex).

**D2 — `@Async("generationExecutor")` for struggle — same pool as topic gen.**
This is a resource sharing choice made because a separate pool would require a second
`ThreadPoolTaskExecutor` bean. The trade-off: simpler config vs. potential starvation.
Acceptable for a POC with ≤5 concurrent admins. Phase 06 (admin web) might change this.

**D3 — badge issuance is synchronous in submit-answer.**
An async approach (like `@TransactionalEventListener`) would decouple badge issuance from
answer submission but would require Spring event infrastructure. The synchronous approach
is simpler and correct for current scale. Badge issuance latency is O(assignments in
enrollment) — acceptable for ≤30 tasks per topic.

### B.3 The Systematic Post-Auth 401 — Root Cause Analysis

**What happens:** Every successful `POST /auth/magic-link/verify` causes an immediate 401 on the
first subsequent request, forcing a token refresh before any real API call can succeed.

**Wire evidence:**
- Request body size for the 401 request: `body N 0`, total 281 bytes
- Same request retried with new token: 585 bytes (304 bytes additional = Authorization Bearer JWT)
- This confirms the 401 request had NO Authorization header

**Root cause:** The Ktor bearer plugin (`Auth { bearer {} }`) caches the result of `loadTokens()`
in memory. On cold start, `loadTokens()` is called and returns `null` (Keychain empty after
`SecItemDelete` × 2). This null is cached. When `verifyMagicLink` completes and writes tokens to
Keychain (SecItemAdd × 2), the Ktor plugin's in-memory cache is NOT invalidated. The next request
uses the cached `null` → no Bearer header → 401. Only after `refreshTokens()` fires (reads from
Keychain via the B9 fallback: `tokenStorage.getRefreshToken()`) does the plugin update its cache.

**This is not B11.** B11 is the Mutex issue where concurrent 401s cause multiple simultaneous
`refreshTokens()` calls → server sees reuse → family revocation. B.3 is a DIFFERENT issue:
even a single request immediately after auth gets 401, wasting the first issued token pair.

**Structural defect:** `HttpAuthRepository.verifyMagicLink()` stores tokens to `tokenStorage`
(Keychain) but has no way to notify the Ktor `BearerAuthProvider` to invalidate its cached null.
The only mechanism to update the plugin's cache is a successful `refreshTokens()` call. This means
the first-post-auth-401 is **architecturally guaranteed** with the current design.

**Impact assessment:**
- Performance: 1 extra round-trip (74ms verify + 16ms 401 + 15ms refresh = 31ms overhead per login)
- Security: the token pair from `verify` is immediately consumed by `refresh` (52ms lifespan)
  → effectively the verify token is single-use within milliseconds ✓ (not a security issue)
- UX: invisible to user, app handles it correctly
- Concurrency risk: if the home screen fires N requests simultaneously after login, ALL get 401
  simultaneously → N concurrent `refreshTokens()` calls → B11 activates → family revocation →
  user is force-logged-out on the FIRST login. This is the actual severity of B11.

### B.4 Keychain Storage — Confirmed Behavior

From the Security framework activity log:

| Operation | When | Result |
|-----------|------|--------|
| `SecItemDelete` × 2 | Cold start (400 on /profile) | Clears stale tokens |
| `SecItemAdd` × 2 | After `POST /auth/magic-link/verify` → 200 | Writes access + refresh |
| `SecItemCopyMatching` × 1 | During `refreshTokens()` callback | Reads refresh token |
| `SecItemAdd` → `SecItemUpdate` × 2 | After `POST /auth/refresh` → 200 | Add-or-update pattern |

The add-or-update pattern (SecItemAdd → errSecDuplicateItem → SecItemUpdate) works correctly.
`kSecUseDataProtectionKeychain = true` is set. Service key: `com.ureka.play4change.tokens`.

**FINDING B-02:** The `SecItemDelete` on cold start is triggered by the 400 response from `/profile`,
not by explicit logout. This means any spurious 400 (proxy error, server restart) on app launch
will clear the user's Keychain tokens and force re-authentication. The correct behaviour should
be: only clear tokens on explicit 401 with an `invalid_token` body, or on explicit logout.
