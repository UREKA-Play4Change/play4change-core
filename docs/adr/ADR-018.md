# ADR-018 — Security Toolchain Gap Analysis and Remediation Roadmap

**Status:** Accepted
**Date:** April 2026
**Author:** Radesh Govind
**Relates to:** ADR-009 (Infrastructure and Secrets Management), ADR-011 (Passwordless Authentication), ADR-016 (Authentication Hardening)

---

## Context

A systematic security posture review of the Play4Change project was conducted against
the following control categories: OWASP Top 10, XSS, CSRF, SAST, DAST, SCA, threat
modelling, Content Security Policy, security headers, deny-by-default access control,
SSRF mitigation, rate limiting, and secret scanning.

The review mapped every existing control to its OWASP category, identified which controls
were implemented, which were planned but not yet executed (Phase 07, PENDING), and which
were entirely absent from any phase plan.

The following was confirmed already in place:

- Stateless JWT with `SessionCreationPolicy.STATELESS` — CSRF is structurally eliminated
  and correctly documented as an accepted risk in THREAT-LOG.md (R01, ADR-011)
- SHA-256 hashing of all tokens before persistence (magic-link and refresh tokens)
- Magic-link single-use with atomic CTE claim (TOCTOU fixed, ADR-016 G2)
- CORS restricted to an environment variable allowlist, no wildcard
- Docker container runs as non-root (`appuser`)
- OWASP dependency-check Gradle plugin with CVE ≥ 7.0 as a build blocker, weekly CI cron
- jsoup `Safelist.none()` sanitisation of all AI-generated output before persistence
- SSRF partial mitigation: https-only scheme, RFC 1918 + loopback + link-local IP block,
  10s connect / 30s read timeouts (Phase 02, Task 2.4)
- STRIDE analysis tables in THREAT-LOG.md for Phases 01, 02, and 04
- Detekt static analysis wired to CI with a committed baseline

The following were confirmed planned but not yet executed (entry criteria: Phases 01–06 DONE):

- Rate limiting on `/auth/**` via Bucket4j + Redis (Phase 07, Task 7.1)
- Swagger UI gated behind ADMIN JWT in the `prod` profile (Phase 07, Task 7.2)
- `/actuator/prometheus` moved to internal port only (Phase 07, Task 7.3)
- `@Valid` + Bean Validation audit on all `@RequestBody` DTOs (Phase 07, Task 7.5)
- OWASP ZAP baseline DAST scan with findings documented in DAST-REPORT.md (Phase 07, Task 7.6)
- ZAP Critical/High finding remediation, including `X-Content-Type-Options` and
  `X-Frame-Options` headers via Nginx (Phase 07, Task 7.7)

The review identified six control categories that were entirely absent from any existing
phase plan and had no tracking entry in THREAT-LOG.md:

1. **OWASP Threat Dragon model** — STRIDE analysis exists as markdown tables but no
   machine-readable `.td` Data Flow Diagram exists. Trust boundaries are described
   in prose but not visualised or exportable as a threat list.

2. **Authenticated DAST scan** — the planned ZAP baseline scan (Phase 07) covers only
   unauthenticated endpoints. All learner and admin flows (`/enrollment/**`, `/tasks/**`,
   `/admin/**`, `/struggle/**`) are unreachable without a valid JWT and would not be
   tested by the baseline scan.

3. **Security-specific SAST** — Detekt enforces Kotlin code quality and style but does
   not detect security anti-patterns. Spring Boot and JWT-specific vulnerabilities
   (JWT `none`-algorithm acceptance, open redirect, insecure randomness, Spring Security
   misconfiguration) require a dedicated security linter such as SpotBugs with
   FindSecBugs or a Semgrep ruleset.

4. **Full HTTP security header set** — Phase 07 (Task 7.7) adds `X-Content-Type-Options`
   and `X-Frame-Options` via Nginx in response to ZAP findings. Three headers with
   material security impact were not planned: `Content-Security-Policy` for the admin
   React SPA, `Strict-Transport-Security` for HTTPS enforcement, and `Referrer-Policy`
   to prevent referrer leakage from admin navigation.

5. **Mobile SCA** — the OWASP dependency-check plugin is scoped to the `server` Gradle
   module. The `composeApp` (Android/iOS KMP) and `common` modules have independent
   dependency graphs with no CVE scanning. A vulnerability in a KMP or Android library
   used by the mobile client would not be detected by the existing weekly CI job.

6. **Secret scanning in CI** — no tool is configured to detect accidentally committed
   secrets (API keys, JWT secrets, database credentials) in the git repository or in
   CI pipeline logs. A single accidental commit of a Mistral API key or `JWT_SECRET`
   value would expose it permanently in git history regardless of subsequent deletion.

Doing nothing was not acceptable. Items 3, 5, and 6 represent gaps that could allow
exploitable vulnerabilities to ship undetected. Items 1, 2, and 4 represent gaps in
assurance coverage that leave attack surfaces untested or insufficiently documented.

## Decision

**Extend the security toolchain and phase plan to address all six identified gaps:
add gitleaks secret scanning to CI (Phase 01 extension), add SpotBugs + FindSecBugs
SAST and mobile SCA to the build (Phase 07 extension), add an authenticated ZAP scan,
define the full Nginx security header set including CSP and HSTS, and create an OWASP
Threat Dragon model as a living threat model artifact.**

The six gaps are addressed as concrete tasks assigned to existing phases as follows:

**Phase 01 extension — G6 (Secret scanning):**
Add `gitleaks` to the GitHub Actions CI pipeline as a pre-build step. Configure it to
scan the full commit history on first run and every new commit thereafter. Any detected
secret pattern fails the build with a non-zero exit code. Suppressions require a
documented justification in `.gitleaks.toml`.

**Phase 07 extensions — G2 (Authenticated DAST), G3 (Security SAST),
G4 (Security headers), G5 (Mobile SCA):**

- **Task 7.6 extension:** After the ZAP baseline scan, run a second ZAP scan in
  authenticated mode. Provide ZAP with a valid learner JWT and a valid ADMIN JWT
  (generated from the seeded admin account). Spider and active-scan the authenticated
  surface. Document findings in a dedicated `DAST-REPORT-AUTHENTICATED.md` section.

- **Task 7.9 (new):** Add SpotBugs with the FindSecBugs plugin to the server Gradle
  build. Configure it to run as a CI step after `detektMain`. Fail the build on any
  finding with confidence HIGH and rank SCARY or SCARIEST. Suppress false positives in
  `spotbugs-exclude.xml` with documented justification. The initial run may produce
  findings against existing code; all findings must be triaged before Phase 07 is
  marked DONE — fix, suppress with justification, or log in THREAT-LOG.md as an
  accepted risk.

- **Task 7.10 (new):** Extend the OWASP dependency-check Gradle configuration to also
  scan the `composeApp` and `common` modules. Add a separate `dependency-check.yml`
  CI step that runs `./gradlew :composeApp:dependencyCheckAnalyze :common:dependencyCheckAnalyze`
  on the same weekly schedule. CVE ≥ 7.0 in mobile modules is a build blocker on the
  same terms as the server module.

- **Task 7.7 extension:** Expand the Nginx security header additions to include:
  `Content-Security-Policy` (strict policy for the admin React SPA: `default-src 'self';
  script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:;
  connect-src 'self'; frame-ancestors 'none'`), `Strict-Transport-Security`
  (`max-age=63072000; includeSubDomains`), and `Referrer-Policy` (`strict-origin-when-cross-origin`).
  Document each header's scope (API vs admin SPA vs both) in DAST-REPORT.md.

**Threat model artifact — G1 (Threat Dragon):**
Create `agentic/security/threat-model.td` as an OWASP Threat Dragon JSON model file.
The model must include: all six bounded contexts as process nodes (Identity, Topic,
Enrollment, Struggle, PeerReview, Badge), all external actors (learner mobile client,
admin browser, Mistral AI, Resend email, Google OAuth, Facebook OAuth), all data stores
(PostgreSQL, Redis, MinIO), and trust boundaries separating the public internet, the
DMZ (Nginx), the application tier, and the data tier. Every threat identified in the
Phase 01 and Phase 02 STRIDE tables must appear as a threat in the Threat Dragon model
with its mitigation linked. The model is updated at the start of each new phase that
introduces a new attack surface, alongside the THREAT-LOG.md STRIDE entry.

## Why This, Not Something Else

| Alternative | Why rejected |
|-------------|-------------|
| Semgrep instead of SpotBugs + FindSecBugs for security SAST | Semgrep requires maintaining a custom ruleset for Spring/Kotlin patterns or subscribing to Semgrep Pro rules. FindSecBugs has a mature, maintained Spring Security rule set that runs without external subscription and integrates directly with the existing Gradle build. Semgrep remains an option for future phases if FindSecBugs coverage proves insufficient. |
| GitHub Advanced Security secret scanning instead of gitleaks | GitHub Advanced Security is a paid feature not available on all repository tiers. Gitleaks is open-source, runs in any CI environment, and produces identical detection for the secret patterns relevant to this project (API keys, JWT secrets, database URLs). If the repository moves to a tier with GHAS, gitleaks can be removed in favour of the native scanner. |
| OWASP ZAP authenticated scan via `zap-api-scan.py` instead of baseline + auth | `zap-api-scan.py` is OpenAPI-driven and covers only endpoints declared in the OpenAPI spec. The Play4Change API has endpoints not yet fully specified (struggle path, peer review). The baseline + manual authentication approach covers the full surface including undocumented endpoints, at the cost of a more complex scan setup. |
| Skipping CSP for the API endpoints and applying it only to the admin SPA | Correct — CSP is only meaningful for HTML responses. The API returns JSON and the CSP header in DAST-REPORT.md correctly notes this is a false positive for the API surface. The decision here is to add CSP scoped specifically to the admin React SPA via a location-specific Nginx block, not to the API routes. |
| Threat Dragon cloud (hosted) instead of local `.td` file | Threat Dragon cloud stores models on a third-party server. The Play4Change architecture diagrams describe internal network topology, database schemas, and trust boundaries — externalising this to a hosted service increases the information surface available to a supply-chain attacker. The local `.td` file committed to the repository is reviewed in PRs like any other artifact. |
| Waiting for a full Phase 09 dedicated to security tooling | Splitting secret scanning into Phase 01 and SAST/SCA/headers into Phase 07 keeps the additions adjacent to the work they gate. A standalone Phase 09 would mean shipping Phase 02–06 code with no secret scanning and no security SAST — both gaps that could allow a vulnerable artifact to reach the Phase 07 starting point with already-committed issues. |

## Consequences

**Easier:**

- Secret leaks are caught at commit time, not discovered post-breach
- Security regressions in Kotlin/Spring code are caught by CI before merge, not by a periodic ZAP run
- Mobile library CVEs surface on the same weekly cadence as server CVEs
- The Threat Dragon model provides a single visual source of truth for trust boundaries
  that can be shared with stakeholders without reading THREAT-LOG.md markdown tables
- Authenticated DAST covers the full application surface, not just the login page
- The complete Nginx header set eliminates the most common ZAP Medium findings before
  the authenticated scan runs, producing a cleaner baseline

**Harder:**

- CI build time increases: gitleaks adds ~15–30s; SpotBugs + FindSecBugs adds 60–120s
  on first run (subsequent runs are incremental). The weekly dependency-check job
  already accepts long runtimes; the mobile module addition extends it proportionally.
- SpotBugs findings on existing code must be triaged before Phase 07 can be marked DONE.
  Some findings will be false positives requiring suppression with documented justification.
- The Threat Dragon `.td` file must be kept in sync with the architecture as phases add
  new bounded contexts or external integrations. This is a maintenance obligation.
- The authenticated ZAP scan requires generating and injecting a valid JWT into the scan
  context. The token must be refreshed before each scan run (15-minute access token TTL).
  The scan setup script must handle token generation.
- The CSP for the admin SPA must be tuned after initial deployment. The `'unsafe-inline'`
  allowance for `style-src` is a temporary concession for the React Vite build; it should
  be replaced with a nonce-based policy once the admin SPA build pipeline is stable.

**Must revisit when:**

- A new bounded context is added — update the Threat Dragon model and add a STRIDE
  table to THREAT-LOG.md before the phase is marked DONE
- The admin React SPA adopts a nonce-based CSP or a build-time hash extraction step —
  replace `'unsafe-inline'` in `style-src` with the appropriate directive
- The repository moves to GitHub Advanced Security — evaluate replacing gitleaks with
  the native secret scanning feature
- A CVE ≥ 7.0 is found in a mobile module — triage and fix before the next release,
  following the same process as server-side CVEs
- The `FACEBOOK_APP_ID` + `FACEBOOK_APP_SECRET` environment variables are provisioned
  — complete G4 from ADR-016 (Facebook OAuth app-level token verification)

## Security Considerations

This ADR directly addresses multiple OWASP Top 10 categories:

**A03 — Injection:** SpotBugs + FindSecBugs detects injection vectors (SQL, command,
LDAP) that Detekt does not. The SAST addition closes the gap between code quality
linting and security linting for the server module.

**A05 — Security Misconfiguration:** The full security header set (`CSP`, `HSTS`,
`Referrer-Policy`, `X-Content-Type-Options`, `X-Frame-Options`) addresses the Nginx
misconfiguration surface. `HSTS` ensures the browser enforces HTTPS for two years
regardless of whether a non-HTTPS URL is ever served. `CSP` for the admin SPA
limits the blast radius of any XSS vulnerability by preventing inline script execution
and restricting resource loading to `'self'`.

**A06 — Vulnerable Components:** Extending OWASP dependency-check to `composeApp` and
`common` closes the mobile SCA gap. CVEs in KMP libraries used on Android/iOS were
previously undetected by the weekly CI job.

**A07 — Auth Failures:** The authenticated DAST scan exercises the full auth-protected
surface. Vulnerabilities in the enrollment, task submission, and admin flows that are
invisible to an unauthenticated scan are now covered.

**A09 — Logging Failures:** Gitleaks secret scanning prevents credentials from entering
the git history. A committed secret is a logging failure in the broadest sense — it
permanently exposes a credential in an audit trail that cannot be purged without
rewriting history.

This ADR introduces one residual risk: the `'unsafe-inline'` allowance in the admin SPA
`Content-Security-Policy` `style-src` directive. This is a temporary concession for the
current Vite/React build pipeline which injects styles inline. The risk is accepted for
the initial Phase 07 delivery. Mitigation: replace with nonce-based or hash-based
`style-src` when the admin SPA build pipeline is next modified. This acceptance is
logged in THREAT-LOG.md.
