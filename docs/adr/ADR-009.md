# ADR-009 — Deployment Strategy: Docker Compose over Kubernetes + AWS

**Status:** Accepted  
**Date:** March 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-003 (Batch Generation), ADR-006 (Module Boundary), ADR-007 (AI Pipeline)

---

## Context

During the implementation phase, the question arose whether to deploy the
backend on a managed Kubernetes cluster (AWS EKS) rather than Docker Compose
on a single virtual machine. The argument for Kubernetes centred on
scalability, resilience, and operational maturity — qualities associated with
production-grade deployments.

This ADR documents the evaluation, the explicit rejection of Kubernetes for
this project stage, and the conditions under which that decision should be
revisited.

The decision is recorded not because it was difficult, but because the
reasoning matters: choosing the simpler option for the right reasons is
engineering discipline, not a shortcut. Choosing the simpler option without
understanding the alternative is a gap.

---

## Decision

**Deploy on Docker Compose on a single-node VPS. Do not use Kubernetes or
AWS managed services at this stage.**

---

## What Kubernetes Solves

To evaluate this correctly, the problems Kubernetes solves must be named
precisely:

| Problem | Kubernetes solution |
|---------|-------------------|
| A single process crashes and the service goes down | Pod restart policy + liveness probes |
| Traffic increases beyond one machine's capacity | Horizontal Pod Autoscaler + multi-node scheduling |
| A new version needs to deploy without downtime | Rolling deployment strategy |
| Multiple services need different resource quotas | CPU/memory limits per Pod |
| A nightly batch job competes with request-serving for CPU | Separate Pod with its own resource limits |
| Secrets must not be stored in source control | Kubernetes Secrets + environment injection |

Each of these is a real operational problem. None of them is a problem that
this project currently has.

---

## Current Scale Reality

The U!REKA Play4Change deployment at evaluation time serves:
- A demo environment with 1–5 concurrent users
- A nightly batch job generating tasks for 1 active course
- A single Spring Boot process and a single PostgreSQL instance

At this scale, Docker Compose on a €10–20/month VPS (Hetzner CX21 or
equivalent) provides identical availability characteristics to an EKS cluster.
Both can run 24/7. Both can be restarted on failure. The operational overhead
of EKS — cluster management, node group configuration, IAM roles, VPC setup,
load balancer provisioning — is unjustified cost at zero benefit.

The cost difference is not marginal:

| Setup | Monthly cost (estimate) | Operational overhead |
|-------|------------------------|---------------------|
| Hetzner CX21 VPS + Docker Compose | €4–8 | Low — one `docker compose up -d` |
| AWS EKS (minimum viable cluster) | €150–250 | High — IAM, VPC, node groups, ALB |
| AWS EKS + RDS + ALB + ECR | €300–500 | Very high |

The EKS minimum includes: control plane (€0.10/hour), two t3.small worker nodes
(~€25/month each), a Network Load Balancer (~€18/month), and ECR image storage.
These costs exist regardless of whether a single user accesses the system.

## Why the Architecture Remains Kubernetes-Ready

The decision to not use Kubernetes does not mean the architecture is
incompatible with it. The services are already structured for future extraction:

**Each Docker Compose service maps directly to a Kubernetes Deployment:**

```yaml
# docker-compose.yml → Kubernetes equivalents

services:
  server:       → Deployment: ureka-server
  postgres:     → StatefulSet: ureka-postgres (or RDS)
  prometheus:   → Deployment: prometheus
  grafana:      → Deployment: grafana
```

**Environment variables become ConfigMaps and Secrets:**
```
application.yml properties → ConfigMap
ai.mistral.api-key         → Secret
db.password                → Secret
```

**The nightly batch job maps to a CronJob:**
```yaml
kind: CronJob
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: batch-worker
              image: ureka-server
              command: ["java", "-jar", "app.jar", "--spring.batch.job.enabled=true"]
```

**The `TaskGenerationPort` interface is already the seam for service extraction:**
The module boundary established in ADR-006 means the AI adapter can be extracted
into an independently deployable service without changing `:server`. When that
happens:

1. Extract `:ai-agent:langchain` into a separate Spring Boot application
2. Create `:ai-agent:grpc-client` implementing `TaskGenerationPort` via gRPC
3. The `:server` module has zero changes — it still calls `TaskGenerationPort`

This is the correct migration path. The interface designed today is the
Kubernetes service contract tomorrow.

## Trigger Conditions for Migration

The migration to Kubernetes (on AWS EKS, GKE, or self-hosted) is justified
when at least one of the following is true:

**Trigger 1 — Resource contention between batch and serving**
The nightly batch job at 02:00 consumes significant CPU for LLM embedding and
pgvector queries. At ~50 calls/night this is negligible. At 100,000 users across
50 courses, the batch window may overlap with early-timezone user traffic.
When batch jobs measurably increase P95 API latency, a separate Pod with resource
limits is the correct fix.

**Trigger 2 — Multi-institution deployment requiring tenant isolation**
The U!REKA institution network spans multiple universities. If each institution
requires an isolated database and isolated AI pipeline (for data residency
reasons consistent with ADR-007's GDPR rationale), namespace-level isolation
in Kubernetes is cleaner than separate VPS instances.

**Trigger 3 — Zero-downtime deployment requirement**
Docker Compose on a single node has a restart gap during deployments. For a
student project demo this is acceptable. For a production service with SLA
commitments, Kubernetes rolling deployments eliminate this gap.

**Trigger 4 — LLM inference becomes the latency bottleneck**
Adaptive branch generation (ADR-007) is on-demand and currently tolerates
60-second Mistral timeouts. If concurrent struggle events at scale exhaust the
Mistral rate limit, a separate inference service with its own queue and
backpressure becomes necessary. That service is a natural Kubernetes Deployment
behind an internal load balancer.

## What Was Implemented Instead of Kubernetes

The time and complexity that would have gone into Kubernetes configuration was
applied to the operational concerns that are relevant at the current scale:

**Observability (already implemented):**
Prometheus metrics exposed via Spring Boot Actuator + Micrometer, visualised
in Grafana. This stack runs identically on Docker Compose and Kubernetes — the
Prometheus scrape config changes from `localhost:8080` to a Kubernetes Service
DNS name, but the instrumentation code is identical. Building observability
before scaling is the correct priority order.

**Graceful degradation (already implemented):**
The `NoOpTaskGenerationAdapter` and `CourseService` mock fallback ensure the
application remains functional when Mistral is unavailable. This resilience is
application-level, not infrastructure-level — it is equally valuable on
Docker Compose and Kubernetes.

**Health checks (already implemented):**
Spring Boot Actuator exposes `/actuator/health` with component-level indicators
for the database and the AI provider. Docker Compose can use this endpoint for
`healthcheck` configuration. Kubernetes liveness and readiness probes point
to the same endpoint — the application instrumentation is identical.

These three capabilities are more valuable at the current scale than Kubernetes
would be, and they transfer directly to a Kubernetes deployment when that
migration occurs.

## Alternatives Considered

### AWS Elastic Beanstalk

Beanstalk abstracts infrastructure while providing some managed features.
Rejected because: vendor lock-in to AWS deployment model, opaque configuration,
higher cost than a raw VPS for equivalent workloads, and the abstraction hides
the infrastructure knowledge that is educationally valuable to demonstrate.

### Railway / Render / Fly.io (managed PaaS)

These platforms deploy Docker containers with minimal configuration.
Reasonable for a demo, but they abstract the Docker Compose and environment
variable management that is explicitly part of the system's deployment story.
The current Docker Compose setup demonstrates those concerns directly.

### Minikube (local Kubernetes for development)

Running Minikube locally would demonstrate Kubernetes familiarity without
the AWS cost. However, for the submission artefact (a running demo), a local
Minikube cluster is not accessible to examiners. A deployed Docker Compose
setup on a VPS is.

---

## Consequences

- The deployment artefact is a `docker-compose.yml` + `.env` file combination
  that can be reproduced on any Linux host with `docker compose up -d`.
- The Prometheus/Grafana stack runs as Docker Compose services alongside the
  application, providing full observability without additional infrastructure.
- A complete Kubernetes migration guide can be produced from this ADR and the
  Docker Compose configuration — the mapping is mechanical and documented above.
- The engineering time saved by not configuring EKS was invested in
  application-level resilience, observability, and the AI pipeline — all of
  which provide more value at the current scale.
- This decision must be revisited at first commercial deployment or when any
  trigger condition above is met, whichever comes first.
