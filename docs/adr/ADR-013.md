# ADR-013 — Learning Flow: Day Progression, Struggle Detection, and Caching Strategy

**Status:** Accepted  
**Date:** April 2026  
**Author:** Radesh Govind  
**Relates to:** ADR-003 (Batch Generation), ADR-005 (Adaptive Learning Path), ADR-009 (Deployment), ADR-012 (Topic Content Pipeline)

---

## Context

Phase 4 implements the core user-facing learning loop: enrolling in a topic,
receiving one task per day, submitting answers, and receiving adaptive scaffolding
when struggling. Four independent design problems had to be resolved:

1. **Day progression** — when exactly does a new task unlock, and how is "today"
   calculated for a user in any timezone?

2. **Struggle detection** — what signals determine that a user is struggling, and
   how is the type of struggle classified without incurring AI cost on every failure?

3. **Roadmap** — how does the user's linear task sequence adapt when an adaptive
   branch is inserted mid-sequence?

4. **Caching** — the task retrieval path hits the database on every request. At
   scale this is unsustainable. What caching strategy protects the database without
   making the system brittle?

Each decision is documented separately because the reasoning is independent,
even though all four were implemented together.

---

## Decision Summary

| Concern | Decision |
|---------|----------|
| Day boundary | Local midnight in the user's timezone (from `X-Timezone` header) |
| Day index calculation | Calendar days elapsed since enrollment, in user's local timezone |
| Struggle threshold | 2 wrong answers on the same `TaskAssignment` |
| Error pattern classification | Automatic rule-based (4-rule priority chain, no AI) |
| Adaptive branch placement | Inserted as extra roadmap nodes after today's task |
| Post-branch flow | User continues forward — no retry of original task |
| Caching | Redis via Spring Data + Lettuce, with silent fallback on unavailability |

## Decision 1 — Day Progression and Timezone Handling

### The problem

One task per day sounds simple. It is not, because "day" is ambiguous:
- A user in Lisbon and a user in Tokyo enrolled at the same moment in UTC.
  At 23:00 UTC, the Lisbon user is still on the same day; the Tokyo user
  has already crossed midnight into the next.
- If days are counted in UTC, a user in a UTC+9 timezone gets their new
  task at 09:00 local time — not midnight. This feels arbitrary and breaks
  the psychological contract of "daily task."

The correct unit is **calendar days in the user's local timezone** — the
same way a diary works.

### The algorithm

```kotlin
object DayIndexCalculator {
    fun compute(enrolledAt: OffsetDateTime, timezoneHeader: String?): Int {
        val zone = timezoneHeader
            ?.let { runCatching { ZoneId.of(it) }.getOrNull() }
            ?: ZoneOffset.UTC
        val enrolledDate = enrolledAt.atZoneSameInstant(zone).toLocalDate()
        val todayDate    = ZonedDateTime.now(zone).toLocalDate()
        return ChronoUnit.DAYS.between(enrolledDate, todayDate).toInt()
            .coerceAtLeast(0)
    }
}
```

**Step by step:**

1. Parse the timezone from the `X-Timezone` HTTP header (e.g. `"Europe/Lisbon"`).
   If missing or invalid, fall back silently to UTC.
2. Convert `enrolledAt` (stored as UTC offset) to a `LocalDate` in the user's
   timezone. This is the enrollment date as the user experienced it.
3. Get today's `LocalDate` in the same timezone.
4. Count full calendar days between enrollment date and today.
   `ChronoUnit.DAYS.between(LocalDate, LocalDate)` counts calendar days, not
   86400-second intervals — a midnight crossing always advances the count by 1
   regardless of DST or leap seconds.
5. `coerceAtLeast(0)` guards against server clock skew: if the server clock is
   somehow behind the enrollment timestamp, the result is 0 rather than negative.

**Example:**

```
User enrolls: 2026-04-01 22:00 UTC (= 23:00 Europe/Lisbon, still Apr 1 locally)
X-Timezone: Europe/Lisbon

Apr 1 23:00 Lisbon → dayIndex = 0  (enrolled today, same calendar day)
Apr 2 00:01 Lisbon → dayIndex = 1  (midnight crossed, new calendar day)
Apr 2 22:00 Lisbon → dayIndex = 1  (still Apr 2 locally)
Apr 3 00:01 Lisbon → dayIndex = 2
```

A UTC calculation would give dayIndex=1 at 00:00 UTC on Apr 2 (23:00 Lisbon time
on Apr 1) — the user's task would unlock an hour before local midnight.

### Why `X-Timezone` header over other approaches

**Alternatives considered:**

- **Store timezone at enrollment time** — the user's timezone may change
  (travel, relocation). Storing it once at enrollment produces incorrect day
  boundaries after a timezone change. The header approach uses the current
  timezone on every request.

- **Infer timezone from IP geolocation** — imprecise (VPNs, shared IPs), adds
  a geolocation service dependency, and fails entirely for API clients. The
  header is an explicit, reliable signal.

- **Admin-configured unlock time per topic** — adds topic configuration
  complexity and still doesn't solve the user's local midnight expectation.
  A user in Tokyo and a user in Lisbon studying the same topic would get their
  tasks at different absolute times but the same local time.

- **UTC calendar day** — simplest to implement. Rejected because it breaks the
  "midnight in your timezone" contract, which is the natural expectation for a
  daily habit product.

### Timezone failure handling

`runCatching { ZoneId.of(it) }.getOrNull()` silently returns `null` on any
invalid timezone string. The null falls through to `ZoneOffset.UTC`. No error
is returned to the client — an invalid or missing timezone is treated as UTC.
This is intentional: the client sending a bad timezone should not cause the
learning flow to fail. UTC is a safe, predictable fallback.

### Where this logic lives

`DayIndexCalculator` is a pure Kotlin `object` in `application/enrollment/`.
It has no Spring context, no JPA, no injected dependencies. It takes two
parameters and returns an `Int`. This makes it trivially testable — the full
test suite for this function requires only a few `OffsetDateTime` + timezone
string combinations, no mocking.

## Decision 2 — Struggle Detection and Error Pattern Classification

### The struggle threshold

The adaptive branch opens after **2 wrong answers on the same `TaskAssignment`**.

The threshold of 2 is a deliberate choice:
- **1 wrong answer** is normal. Even confident learners misread questions.
  Triggering adaptive scaffolding on a single wrong answer would be
  patronising and would generate an enormous volume of adaptive tasks at scale.
- **3 wrong answers** means the user has already spent significant time on a
  task they do not understand before help arrives. The intervention comes too late.
- **2 wrong answers** signals genuine difficulty — the user tried, failed, tried
  again with a different answer, and failed again. The pattern is unambiguous.

The threshold is checked in `TaskService.submitAnswer()` at the moment
`wrongAttemptCount` would reach 2. It triggers `HandleStruggleService.triggerAsync()`
which runs on the `generationExecutor` thread pool — the learning flow response
returns immediately and adaptive task generation happens in the background.

### Error pattern classification — automatic, no AI

When the struggle threshold is crossed, the system must characterise the
*type* of struggle so the AI generation prompt can produce appropriate scaffolding.
Four error patterns are defined: `WRONG_CONCEPT`, `PARTIAL_UNDERSTANDING`,
`READING_ERROR`, `TIME_PRESSURE`.

**Why not ask the AI to classify it?**

The AI generation prompt already receives the struggle context. Asking a separate
LLM call to classify the error pattern before generating the adaptive branch
would add 10–30 seconds of latency to the trigger response, increase cost, and
introduce a new failure point. The error pattern is an input to the generation
prompt — it does not need to be AI-derived.

**Why not ask the user?**

Interrupting a struggling user with "why do you think you got this wrong?" is a
poor experience. The system should diagnose silently.

**The classification algorithm**

`ErrorPatternClassifier` applies a 4-rule priority chain at the moment of the
second wrong answer. The data available at that point:

- `assignment` — the current `TaskAssignment`: contains `dueAt`, `optionOrder`
  (the shuffled option display order), and `selectedOption` (the first wrong answer)
- `newSelectedOption` — the shuffled index being submitted right now (second wrong)
- `template` — the `TaskTemplate`: contains `correctAnswer` (original unshuffled index)

```kotlin
object ErrorPatternClassifier {
    fun classify(
        assignment: TaskAssignment,
        newSelectedOption: Int,
        template: TaskTemplate
    ): ErrorPattern {

        // Priority 1: TIME_PRESSURE
        // Submitted after the 24-hour window expired
        if (OffsetDateTime.now().isAfter(assignment.dueAt))
            return ErrorPattern.TIME_PRESSURE

        // Priority 2: READING_ERROR
        // The selected option maps to an index adjacent to the correct answer
        // in the original (non-shuffled) options array
        val originalIndexSelected = assignment.optionOrder[newSelectedOption]
        if (abs(originalIndexSelected - (template.correctAnswer ?: -99)) <= 1)
            return ErrorPattern.READING_ERROR

        // Priority 3: PARTIAL_UNDERSTANDING
        // User tried two different wrong options — they have some knowledge
        // but are uncertain between options, not simply guessing randomly
        if (assignment.selectedOption != null
            && assignment.selectedOption != newSelectedOption)
            return ErrorPattern.PARTIAL_UNDERSTANDING

        // Priority 4: WRONG_CONCEPT (default)
        return ErrorPattern.WRONG_CONCEPT
    }
}
```

**Why this priority order:**

`TIME_PRESSURE` first because late submission is a behavioural signal independent
of which option was chosen — the scaffolding should address the time management
aspect, not the content.

`READING_ERROR` second because selecting an option adjacent to the correct answer
in the unshuffled array is a strong signal of inattention or misreading, not
conceptual failure. The shuffling means the user sees options in a different order,
but if the option they chose was "next to" the correct answer in the original
question, it is likely they read too quickly.

`PARTIAL_UNDERSTANDING` third because choosing two different wrong options across
two attempts is a stronger signal than choosing the same wrong option twice.
Choosing the same wrong answer twice (`selectedOption == newSelectedOption`) is
a strong `WRONG_CONCEPT` signal — the user is confident in an incorrect belief.
Choosing different wrong options suggests uncertainty between plausible-sounding
answers, which is scaffolded differently.

`WRONG_CONCEPT` as the default — if none of the above conditions match, the user
is consistently wrong in a way that does not fit a more specific pattern.

**Where this logic lives:**

`ErrorPatternClassifier` is a pure Kotlin `object` in `application/struggle/`.
No Spring context, no dependencies. Like `DayIndexCalculator`, it is directly
testable with plain inputs.

**Mapping to the AI agent's error pattern enum:**

The server's `domain/struggle/ErrorPattern` maps to the ai-agent's
`com.ureka.play4change.model.ErrorPattern` inside `HandleStruggleService`:

| Server domain | AI agent |
|---|---|
| `WRONG_CONCEPT` | `CONCEPTUAL_MISUNDERSTANDING` |
| `PARTIAL_UNDERSTANDING` | `PROCEDURAL_ERROR` |
| `READING_ERROR` | `UNCLEAR_INSTRUCTIONS` |
| `TIME_PRESSURE` | `UNKNOWN` |

`UNKNOWN` is used for `TIME_PRESSURE` because the AI agent's adaptive branch
generator does not have a specific scaffolding strategy for time-based failures —
the generated tasks are general reinforcement of the concept.

## Decision 3 — Roadmap Design: Linear Path with Inserted Adaptive Nodes

### The mental model

The roadmap is a straight road. Every user enrolled in a topic sees the same
sequence of task nodes, one per day. When a struggle session opens, adaptive
tasks are inserted as extra nodes between today's task and tomorrow's. There is
no branching, no parallel paths, no "return to original task" mechanic. The road
simply gets longer temporarily.

```
Before struggle:
  Day 1 ✓ → Day 2 ✓ → Day 3 [today] → Day 4 🔒 → Day 5 🔒

After 2nd wrong answer on Day 3:
  Day 1 ✓ → Day 2 ✓ → Day 3 [today] → A1 📍 → A2 🔒 → A3 🔒 → Day 4 🔒 → Day 5 🔒

After completing all adaptive tasks:
  Day 1 ✓ → Day 2 ✓ → Day 3 ✓ → A1 ✓ → A2 ✓ → A3 ✓ → Day 4 [today] → Day 5 🔒
```

### Why no retry of the original task

Earlier designs considered returning the user to the original task after
completing the adaptive branch. This was rejected for three reasons:

**Pedagogical:** The adaptive tasks scaffold the user back to the level required.
If the scaffolding is well-designed, the user understands the concept by the
time they finish the adaptive sequence. Forcing them to answer the original
question again is a test, not learning — and a test they may still fail,
causing frustration.

**UX:** The roadmap is a straight road. Users understand "complete these tasks
and move forward." A loop — complete adaptive tasks, then retry the original
before advancing — introduces a conditional that is hard to communicate
visually and surprising on first encounter.

**Simplicity:** No retry means the `StruggleSession` resolves automatically
when all adaptive tasks are completed. The `currentDayIndex` advances at the
next calendar midnight regardless. No special "retry gate" logic, no conditional
advancement.

### Roadmap node statuses

The roadmap endpoint assembles a flat ordered list of `RoadmapNode` objects:

| Status | Meaning |
|--------|---------|
| `COMPLETED` | Past day, submitted on time, correct |
| `LATE` | Past day, submitted after `dueAt` |
| `SKIPPED` | Past day, not submitted |
| `PENDING` | Today's task, not yet submitted |
| `LOCKED` | Future day |
| `ADAPTIVE_PENDING` | Adaptive task not yet completed |
| `ADAPTIVE_COMPLETED` | Adaptive task completed |

Adaptive nodes carry `isAdaptive = true` and are inserted in `orderIndex`
sequence from the open `StruggleSession`. Future regular tasks remain `LOCKED`
until all adaptive tasks are completed and the next calendar day arrives.

### Where the roadmap lives

`RoadmapService` assembles the roadmap from three data sources:

1. `TaskTemplateRepository.findCurrentByModuleId()` — all task templates for
   the enrollment's module (the fixed path)
2. `EnrollmentRepository.findAssignmentsByEnrollmentId()` — all existing
   task assignments (past + today's submissions)
3. `StruggleRepository.findOpenByEnrollmentId()` — the open struggle session
   and its adaptive tasks, if any

The assembly is purely in-memory: templates are indexed by `dayIndex`, assignments
are indexed by `taskTemplateId`, and adaptive tasks are appended after the current
day's node. No database joins, no complex queries.

## Decision 4 — Caching Strategy: Redis with Silent Fallback

### The problem

The `GET /tasks/today` endpoint is the most frequently called endpoint in the
system — every active user calls it at least once per day, and potentially
multiple times (app open, app close, re-open). Without caching, every call
hits the database: load enrollment, derive dayIndex, load task template,
check or create assignment. At 10,000 active users with a morning peak, this
is 10,000 database round-trips in a short window.

### The solution: Redis with `CachePort`

Redis is added as a Docker Compose service. Spring Data Redis with Lettuce
(the default driver, auto-configured) is used.

The cache is accessed exclusively through `CachePort`:

```kotlin
// application/port/CachePort.kt
interface CachePort {
    fun get(key: String): String?
    fun put(key: String, value: String, ttlSeconds: Long)
    fun evict(key: String)
}
```

`RedisCacheAdapter` implements this interface. Every operation is wrapped in
`try-catch`:

```kotlin
@Component
class RedisCacheAdapter(
    private val redis: RedisTemplate<String, String>
) : CachePort {

    override fun get(key: String): String? = try {
        redis.opsForValue().get(key)
    } catch (ex: Exception) {
        log.warn("Redis get failed for key {}: {}", key, ex.message)
        null  // cache miss — caller falls through to DB
    }

    override fun put(key: String, value: String, ttlSeconds: Long) = try {
        redis.opsForValue().set(key, value, Duration.ofSeconds(ttlSeconds))
    } catch (ex: Exception) {
        log.warn("Redis put failed for key {}: {}", key, ex.message)
        // no-op — DB is source of truth, cache miss on next read is acceptable
    }

    override fun evict(key: String) = try {
        redis.delete(key)
    } catch (ex: Exception) {
        log.warn("Redis evict failed for key {}: {}", key, ex.message)
        // no-op — TTL will expire the entry eventually
    }
}
```

**Redis unavailability never breaks the learning flow.** A Redis outage means
cache misses on every request — all traffic falls through to the database.
This is a performance degradation, not a service outage.

### Cache keys and TTLs

| What | Key pattern | TTL | Invalidated on |
|------|------------|-----|----------------|
| Today's task | `task:{userId}:{topicId}:{dayIndex}` | 24h | Submission |
| Topic metadata | `topic:{topicId}` | 1h | Admin update |
| Enrollment state | `enrollment:{userId}:{topicId}` | 15m | Submission or progress update |
| Task template | `template:{templateId}:{version}` | 6h | New version created |

**Why these TTLs:**

`task:{userId}:{topicId}:{dayIndex}` — 24h because the task does not change
within a calendar day. The key includes `dayIndex` so a key from day N
naturally expires before day N+1 is accessed (the new day generates a new key).
Active invalidation on submission is belt-and-suspenders: if a user submits,
the cached pending task is stale.

`enrollment:{userId}:{topicId}` — 15m because enrollment state changes on
every submission (points, streak, dayIndex). A 15-minute window means a user
who submits and immediately checks their profile sees updated data within 15
minutes at worst. This is acceptable for a daily-cadence app.

`topic:{topicId}` — 1h because topic metadata (title, status, description)
changes rarely. An admin updating a topic description does not need instant
global propagation — 1h staleness is invisible to users.

`template:{templateId}:{version}` — 6h because task templates are immutable
once created. The version is in the key, so a versioned edit (which creates a
new template record with a new ID) naturally bypasses the old cache entry.

### Why Spring Data Redis over alternatives

**Jedis** — the older Redis client for Java. Spring Boot auto-configures Lettuce
by default. Using Jedis requires explicitly excluding Lettuce and adding Jedis.
No benefit justifies this additional configuration.

**In-memory map (Caffeine)** — a local JVM cache. Appropriate for single-instance
deployments. Rejected because it does not survive application restarts and is not
shared between application instances if the system scales horizontally. Redis is
process-external, so a restart does not invalidate the cache.

**Spring Cache abstraction (`@Cacheable`)** — Spring's annotation-driven caching.
Rejected because it hides cache key construction and TTL management behind
annotations, making it hard to reason about exactly what is cached and when.
The explicit `CachePort.put(key, value, ttl)` approach makes every caching
decision visible in the use case code.

## The Complete Learning Loop

The four decisions above compose into a single coherent user experience:

```
User opens app (morning)
  │
  ├── GET /topics/{topicId}/tasks/today
  │     X-Timezone: Europe/Lisbon
  │
  │   TaskService.getTodayTask(userId, topicId, timezone)
  │     │
  │     ├── DayIndexCalculator.compute(enrollment.enrolledAt, timezone)
  │     │     → dayIndex = 3
  │     │
  │     ├── CachePort.get("task:{userId}:{topicId}:3")
  │     │     → null (cache miss)
  │     │
  │     ├── Load TaskTemplate(dayIndex=3, isCurrent=true)
  │     ├── Find or create TaskAssignment
  │     ├── Shuffle options → store optionOrder
  │     ├── CachePort.put("task:{userId}:{topicId}:3", json, 86400)
  │     └── Return TaskResponse (shuffled options, NO correctAnswer)
  │
  └── User submits answer
        POST /tasks/{assignmentId}/submit { selectedOption: 2 }
        │
        TaskService.submitAnswer(command)
          │
          ├── Validate ownership (userId matches assignment.userId)
          ├── Map shuffled index → original index via optionOrder
          ├── Compare with template.correctAnswer
          │
          ├── [CORRECT]
          │     Update: isCorrect=true, pointsAwarded, status=SUBMITTED
          │     Update enrollment: totalPoints++, streak++, lastActivityAt
          │     CachePort.evict("task:{userId}:{topicId}:3")
          │     CachePort.evict("enrollment:{userId}:{topicId}")
          │     Return: { isCorrect: true, points: 80, streak: 4 }
          │
          └── [WRONG, 2nd attempt]
                wrongAttemptCount = 2 → threshold crossed
                ErrorPatternClassifier.classify(assignment, 2, template)
                  → PARTIAL_UNDERSTANDING
                HandleStruggleService.triggerAsync(...)
                  [background, @Async]
                  → Creates StruggleSession
                  → Calls taskGenerationPort.generateAdaptiveBranch(context)
                  → Saves AdaptiveTasks to StruggleSession
                Return immediately: { isCorrect: false, struggleTriggered: true }

User checks roadmap
  GET /topics/{topicId}/roadmap
  → [Day 1 ✓] [Day 2 ✓] [Day 3 ✓] [A1 📍] [A2 🔒] [A3 🔒] [Day 4 🔒] ...
```

## Alternatives Considered (Summary)

| Concern | Rejected alternative | Reason |
|---------|---------------------|--------|
| Day boundary | UTC calendar day | Breaks local midnight contract |
| Day boundary | Store timezone at enrollment | Fails after user travel/relocation |
| Day boundary | IP geolocation | Imprecise, adds external dependency |
| Struggle threshold | 1 wrong answer | Too sensitive — patronising, costly |
| Struggle threshold | 3 wrong answers | User struggles too long before help |
| Error classification | AI-based classification | Adds latency, cost, failure point |
| Error classification | User self-report | Interrupts struggling user |
| Post-branch flow | Retry original task | UX loop, adds gate logic, not pedagogically necessary |
| Caching | Jedis | Requires removing Lettuce auto-config, no benefit |
| Caching | In-memory map | Not shared across instances, lost on restart |
| Caching | `@Cacheable` annotations | Hides cache key and TTL logic from use cases |

## Consequences

- Every user-facing endpoint that derives dayIndex reads the `X-Timezone` header.
  An absent header silently defaults to UTC — no error, no broken flow.
- `DayIndexCalculator` and `ErrorPatternClassifier` are pure objects. They require
  zero mocking to test — all Phase 4 unit tests are fast and dependency-free.
- Redis unavailability degrades performance (cache misses) but never breaks
  correctness. The database is always the source of truth.
- The correct answer index is never included in any API response. Option shuffling
  is stored server-side in `optionOrder` JSONB, which maps the displayed shuffled
  index back to the original answer index at submission time.
- The roadmap assembles from three queries with no joins. Adding new roadmap
  node types (achievements, bonus tasks) requires only extending the assembly
  logic in `RoadmapService` — no schema changes.
- Struggle sessions resolve automatically when all adaptive tasks are completed.
  No manual admin action or user confirmation is required to resume the fixed path.
