## Group 1 ‚Äî Auth / Token Refresh Race (Critical, affects every iOS user)

### F-01 ‚Äî Concurrent 401s cause family revocation and force-logout (B11 / B-04)

**What is broken:** When the access token expires while the home screen loads, 2‚Äì3 requests
fire simultaneously. All get 401. All call `refreshTokens()` concurrently. The second call
presents an already-rotated (used) refresh token ‚Üí server revokes the entire token family ‚Üí
`sessionExpired()` ‚Üí user is sent back to login. Happens every ~15 minutes per session.

**Option A ‚Äî Mutex in HttpClientFactory (documented fix in ISSUES.md)**
Wrap the `refreshTokens { }` block with a `Mutex`. First coroutine does the refresh;
subsequent ones wait, then read the new tokens from `TokenStorage` without hitting the server.
- Risk: low. Ktor coroutines compose well with `Mutex`.
- Effort: ~20 lines in `HttpClientFactory.kt`.
- Does NOT fix the extra round-trip (see F-02), just prevents the race.

**Option B ‚Äî Reduce token TTL granularity on the server**
Reduce the 15-minute access token TTL to something longer (e.g., 60 min).
Reduces refresh frequency but doesn't fix the race ‚Äî it just happens less often.
- Risk: low server change. Doesn't fix the root cause.
- Not recommended as the sole fix.

**Option C ‚Äî Both A + proactive refresh (before expiry)**
Mutex (Option A) + add a background coroutine that calls `POST /auth/refresh` 30 seconds
before the access token expires, so it's fresh by the time requests fire.
- Effort: moderate. Requires tracking token expiry in `TokenStorage`.
- Eliminates both the race and the extra round-trip.

**‚Üí DECISION:**
Option A

### F-02 ‚Äî Post-auth 401 always fires (Ktor BearerAuthProvider cache not invalidated)

**What is broken:** After `POST /auth/magic-link/verify` writes tokens to Keychain, the Ktor
plugin still has a cached `null` from the cold-start `loadTokens()` call. The very next request
goes out without an `Authorization` header, gets 401, triggers `refreshTokens()`, and spends
the original token pair immediately. This is **architecturally guaranteed** with the current design.
Live test confirmed this on every single login.

**Option A ‚Äî Call a custom `invalidate()` or re-init the Ktor client after login**
After `verifyMagicLink` stores tokens, recreate or reinitialise the Ktor client so
`loadTokens()` is called fresh. This is not natively supported in Ktor's bearer plugin.
Would require Koin scope manipulation or a custom `BearerAuthProvider` wrapper.
- Risk: moderate. Touching DI wiring mid-session can cause hard-to-debug issues.
- Effort: ~1‚Äì2 days of investigation + implementation.

**Option B ‚Äî Accept the extra round-trip; fix only the race (F-01 Option A)**
The 401‚Üírefresh sequence works correctly (live-tested). The overhead is ~31ms and invisible
to the user. Accept it as a known architectural limitation (documented in ADR-021 D1).
Only fix F-01 (Mutex) to prevent B11 activation.
- Risk: none. Current behaviour is functionally correct.
- Effort: zero beyond F-01.

**Option C ‚Äî Replace Ktor bearer plugin with a custom interceptor**
Write a Ktor `HttpSend` plugin that proactively reads from `TokenStorage` on every request
and injects the `Authorization` header before sending. Skip the bearer plugin entirely.
- Risk: high. Replaces a battle-tested plugin with custom code.
- Effort: significant. Token refresh logic must be reimplemented.

**‚Üí DECISION:**
B

## Group 2 ‚Äî iOS Client Hardcoded Values (High ‚Äî blocks real-device testing)

### F-03 ‚Äî iOS base URL hardcoded to `http://localhost:8080`

**What is broken:** `PlatformModule.ios.kt` hardcodes `http://localhost:8080`. Works on
the simulator only. Any real device, TestFlight build, or staging environment fails
because it cannot reach the simulator host's localhost.

**Option A ‚Äî Compile-time xcconfig / Info.plist variable**
Add `BASE_URL` to an `.xcconfig` file. Read it in `PlatformModule.ios.kt` via
`Bundle.main.object(forInfoDictionaryKey:)`. Debug xcconfig points to localhost; Release
xcconfig points to the production URL.
- Risk: low. Standard iOS pattern.
- Effort: small. Requires Xcode project file edit and a new .xcconfig.

**Option B ‚Äî KMP `BuildKonfig` / `BuildConfig` equivalent for iOS**
Use a KMP-compatible build config plugin (e.g., `buildkonfig`) to inject the base URL
at compile time, same as Android's `BuildConfig.BASE_URL`.
- Risk: low. Consistent with the Android approach already in the project.
- Effort: small-medium. Plugin already may be in the build if Android uses it.

**Option C ‚Äî Runtime environment file (plist lookup)**
Bundle an `env.plist` file (gitignored) that the app reads at startup. `.xcconfig`
sets the path. Dev environment plist ‚Üí localhost; prod ‚Üí real URL.
- Risk: low. But adds a manual step for each developer setup.
- Not recommended ‚Äî more friction than xcconfig.

**‚Üí DECISION:**
B

### F-04 ‚Äî `X-Timezone` header never sent by mobile client

**What is broken:** Every server endpoint that uses `X-Timezone` (tasks, roadmap, `dueAt`
calculation) falls back to UTC. Users in any other timezone get the wrong day index, wrong
`dueAt`, and wrong `X-Task-Available-At` header. Live test confirmed this for Lisbon (UTC+1).
Affects 100% of non-UTC users.

**Option A ‚Äî Send `X-Timezone` on every authenticated request via Ktor `defaultRequest`**
In `HttpClientFactory.kt`, add `header("X-Timezone", TimeZone.currentSystemDefault().id)`
to the `defaultRequest { }` block. Kotlin `kotlinx-datetime` has `TimeZone.currentSystemDefault()`.
- Risk: minimal. One-line addition per platform.
- Effort: trivial. Already have `X-Request-ID` set in `defaultRequest`.

**Option B ‚Äî Send `X-Timezone` only on endpoints that need it (task, roadmap)**
Override per-call in `HttpTaskRepository` and `HttpEnrollmentRepository`.
- Risk: none. But creates drift risk ‚Äî future endpoints will need the same treatment.
- Not recommended over Option A.

**Option C ‚Äî Store timezone on the user profile server-side (Phase 03 Task 3.4 preference)**
Send timezone once during login/profile update. Server uses stored preference instead of
header. Eliminates per-request header overhead.
- Risk: medium. Server-side change. Header and stored preference can diverge if user travels.
- Could complement Option A (server uses header if present, falls back to profile preference).

**‚Üí DECISION:**
A

### F-05 ‚Äî App clears Keychain tokens on any 400 response (not just 401)

**What is broken:** `SplashRepository` or the session check on cold start gets a 400 from
`GET /profile` (server returns 400 when no `Authorization` header, instead of 401).
The client treats any non-2xx as "session expired" and clears the Keychain.
A transient 400 (proxy, server restart, network glitch) silently logs the user out.

**Sub-issue:** `GET /profile` should return 401 when unauthenticated, not 400.
The 400 itself is a server-side bug (see F-13).

**Option A ‚Äî Fix the server to return 401 on unauthenticated GET /profile**
The endpoint is guarded by Spring Security. The 400 likely comes from the Ktor request
construction (missing required parameter). Ensure Spring Security returns 401 before any
controller logic runs on unauthenticated requests.
- Fixes root cause. Client then receives standard 401 ‚Üí correct session-expired path.

**Option B ‚Äî Make the client only clear tokens on explicit 401 with a specific body**
Change client: only clear tokens if status == 401. Currently it clears on any error.
- Doesn't fix the 400 from the server, but stops false-positive logouts.
- Combined with Option A is the right fix.

**‚Üí DECISION:**
A + B

## Group 3 ‚Äî Logout (Medium ‚Äî security concern)

### F-06 ‚Äî Logout clears local tokens before the server call (BUG 5)

**What is broken:** `HttpAuthRepository.logout()` clears `TokenStorage` (Keychain) first,
then tries to call `POST /auth/logout`. If the network is down, the server refresh token
remains valid for up to 7 days. A stolen device with a cached refresh token at that moment
could re-authenticate.

**Option A ‚Äî Reverse the order: call server first, then clear on success**
```kotlin
val refreshToken = tokenStorage.getRefreshToken()
if (refreshToken != null) {
    client.post("/auth/logout") { ... }  // throws on failure
}
tokenStorage.clear()
```
- Risk: if the server call fails (network down), tokens are NOT cleared locally.
  User might think they logged out but their session persists in local storage.
- Standard security recommendation: server-first.

**Option B ‚Äî Clear locally always, but retry the server call in the background**
Clear immediately (so the local session is gone), then enqueue a background `WorkManager`
job to notify the server. On next app start, re-attempt any pending server-side revocations.
- Risk: adds complexity. The server might have already expired the token by retry time.
- Effort: medium. Requires persistent queue.

**Option C ‚Äî Clear locally always (current behaviour), accept the risk, document it**
The threat model for a stolen device is: attacker must have the device AND extract the
Keychain (requires bypass of iOS Data Protection). The 7-day window is a residual risk,
not an active threat. Document it in THREAT-LOG as an accepted risk.
- Risk: formally accepted, not ignored.
- Effort: zero. Only adds a THREAT-LOG entry.

**‚Üí DECISION:**
C

## Group 4 ‚Äî Server Configuration (High for production, Low for local dev)

### F-07 ‚Äî `APP_BASE_URL` defaults to `http://localhost:5173` (admin web, wrong port)

**What is broken:** `application.yml` default for the magic link email URL is the React
admin dev server port, not the Spring server port. Any bare-server deployment (no Docker
Compose override) sends magic link emails pointing to the wrong host. Docker Compose
correctly overrides it, so local dev works, but this is a foot-gun for any other deployment.

**Option A ‚Äî Change the default to `http://localhost:8080`**
One-line change in `application.yml`.
- Correct for bare-server local development.
- Still wrong for any real deployment, but at least consistent.

**Option B ‚Äî Remove the default; require `APP_BASE_URL` to be explicitly set**
```yaml
base-url: ${APP_BASE_URL}  # no default ‚Äî fail fast if not set
```
- Server fails to start if `APP_BASE_URL` is not set. Forces explicit configuration.
- Risk: breaks existing setups that rely on the default. Requires `APP_BASE_URL=http://localhost:8080` in local `.env`.

**Option C ‚Äî Keep default but add a startup validation warning**
Log a loud `WARN` at startup if `APP_BASE_URL` contains `5173` or `localhost:5173`.
- Non-breaking. Alerts developers without failing the build.
- Effort: trivial.

**‚Üí DECISION:**
B

### F-09 ‚Äî Spring HTTP Basic auth implicitly enabled (default UserDetailsService active)

**What is broken:** At startup, Spring logs `Using generated security password: <UUID>`.
This means Spring's HTTP Basic auth filter is active alongside the JWT filter. An attacker
who observes the startup log (e.g., via a logging aggregator) gets a valid HTTP Basic
credential for the server's lifetime. Currently not tracked in THREAT-LOG.

**Option A ‚Äî Disable HTTP Basic in SecurityConfig**
Add to `SecurityConfig.securityFilterChain`:
```kotlin
http.httpBasic { it.disable() }
```
- One-line fix. Eliminates the attack surface entirely.
- Risk: none ‚Äî the app uses JWT, not HTTP Basic.

**Option B ‚Äî Also remove the default user auto-configuration**
Add to `application.yml`:
```yaml
spring:
  security:
    user:
      password: disabled-not-used
```
Or exclude `UserDetailsServiceAutoConfiguration` from the Spring Boot auto-configuration.
- Belt-and-suspenders with Option A.
- Effort: trivial.

**‚Üí DECISION:**
A+B

## Group 5 ‚Äî Task Lifecycle Bugs

### F-11 ‚Äî Options in task response may not be re-ordered through shuffled `optionOrder`

**What is broken:** ADR-021 ¬ß4.1 flags this as "needs live verification." The `optionOrder`
(e.g., `[2, 0, 3, 1]`) is stored in `TaskAssignment`. The response DTO must re-order
`options[]` using `optionOrder` before serialising. If it doesn't, the client sees
canonical option ordering but `selectedOption` is interpreted as a shuffled index ‚Üí
wrong answers evaluated. This is a **data-correctness bug** if unverified.

**Before choosing a fix option, this needs code verification:**
- Read `TaskService.getTodayTask()` response construction
- Read the web controller DTO mapping
- Confirm whether `options` in the response are re-ordered or not

**Option A ‚Äî Verify + fix in one step (most likely path)**
Read the DTO mapper. If options are not re-ordered:
```kotlin
val reorderedOptions = optionOrder.map { template.options[it] }
```
- Effort: small if the mapper is wrong. Unknown until verified.

**Option B ‚Äî Add a test that proves the shuffle is end-to-end correct**
Write a `@WebMvcTest` that submits a known `selectedOption` for a known template and
asserts `isCorrect` matches the expected canonical answer. This would catch the bug if
it exists and prevent regression.
- Should be done regardless of Option A.

**‚Üí DECISION:**
A+B

### F-12 ‚Äî Badge issuance compares against `topic.taskCount` not actual template count (BUG 13)

**What is broken:** `BadgeIssuanceService` checks `correctCount >= topic.taskCount`.
`topic.taskCount` is the number requested at creation. If Mistral generates fewer tasks
(e.g., due to 429 rate limiting ‚Äî confirmed live in A.2), a user who correctly completes
all *actual* tasks never reaches the threshold. Badge is silently never issued.

**Option A ‚Äî Compare against actual template count in the module**
```kotlin
val actualCount = taskTemplateRepository.countByTopicModuleId(module.id)
if (correctCount >= actualCount) { issueIfNotYetEarned(...) }
```
- Fixes the root cause.
- Requires a small repository method addition.
- Effort: small.

**Option B ‚Äî Compare against actual assignment count for the enrollment**
Count assignments the user has that are SUBMITTED (not the template count).
- Closer to "did the user do all the work that was given to them."
- More correct semantically, same effort.

**Option C ‚Äî Fix Mistral 429 handling so `taskCount` always matches generated count**
Update `topic.taskCount` after generation to reflect what was actually created.
Fixes the inconsistency at the source.
- Complements Option A/B ‚Äî both fixes should be applied.

**‚Üí DECISION:**
A+B+C

### F-13 ‚Äî `GET /struggle/enrollment/{id}` returns empty/400 on RESOLVED session (Live BUG)

**Option B ‚Äî Return the session regardless of status; add a `status` field to the response**
Change the controller to not filter by OPEN. Include `status: RESOLVED | OPEN | ABANDONED`
in the response DTO. Client shows appropriate UI per status.
- Better UX. Client can differentiate "in progress" from "complete" from "failed."
- Effort: small server change + small client change.

**‚Üí DECISION:**

### F-14 ‚Äî `dueAt` for first assignment uses `now + 24h` instead of next midnight (BUG 8/I03)

**Option A ‚Äî Use `DayIndexCalculator.startOfTomorrow(UTC)` for the first assignment**

- Requires a test update.

**Option B ‚Äî Pass `X-Timezone` header through the enrollment endpoint**
Propagate timezone from the enrollment HTTP request down to `EnrollmentService`.
Server uses the user's actual timezone for first `dueAt`.
- More correct but requires API contract change.
- Depends on F-04 being fixed first (client must send X-Timezone).

## Group 6 ‚Äî Struggle Flow Bugs

### F-16 ‚Äî Adaptive task shuffle uses unseeded `java.util.Random()` (BUG 14)

**What is broken:** AdaptiveTask option shuffle calls `.shuffle()` (no seed), which uses
`java.util.Random()`. Different random per generation. The shuffle is stored at generation
time, so it's stable once created. But if the session is regenerated (after ABANDONED),
the same user gets a different ordering. Minor inconsistency vs the deterministic main
task shuffle.

**Option A ‚Äî Use a deterministic seed (sessionId + taskId)**
```kotlin
val seed = (sessionId.toString() + task.id.toString()).hashCode().toLong()
(options.indices.toMutableList()).also { it.shuffle(Random(seed)) }
```
- Consistent with the anti-cheat approach used for regular tasks.
- Effort: trivial.

**Option B ‚Äî Accept unseeded shuffle for adaptive tasks**
Adaptive tasks are remediation content, not graded. The anti-cheat concern (option order
sharing between users) is lower here ‚Äî each struggle session is unique to one user.
Document the difference as intentional.
- Effort: zero. Add a DECISIONS.md entry.

**‚Üí DECISION:**
B

### F-19 ‚Äî Struggle GET may return wrong session when multiple exist (BUG 18)

**What is broken:** `findOpenByEnrollmentId(enrollmentId)` ‚Äî if a user has multiple OPEN

In `HandleStruggleService.triggerAsync()`: before creating a new session, close any

### F-20 ‚Äî Struggle uses same thread pool as topic generation (BUG 15 / Y1)

**What is broken:** `@Async("generationExecutor")` is used for both topic generation

## Group 8 ‚Äî Tech Debt (Yellow ‚Äî not broken, but will bite later)

### F-24 ‚Äî `findByTaskTemplateId` has no ORDER BY ‚Äî non-deterministic instance selection

**What is broken:** `taskInstanceRepository.findByTaskTemplateId(templateId)` returns
instances in Postgres insertion order (not guaranteed after vacuums, restores, etc.).
`instances[Math.floorMod(seed, instances.size)]` selects by index. Same user could get
a different instance after a DB restore. Anti-cheat guarantee breaks silently.

### F-26 ‚Äî Broken topic (0 tasks generated) shows "no tasks yet" ‚Äî misleading (BUG 10)

- Effort: small.

### F-27 ‚Äî Streak may be permanently reduced by struggling (product decision, undocumented)

intentional (struggle = failure event) or unintentional (struggle = learning, not failure).

- Risk: requires persisting the pre-struggle streak count (new field or DECISIONS.md note).

**Option B ‚Äî Treat the streak reset as intentional (document in DECISIONS.md)**

consequence. Document this choice.
- Effort: zero code. One DECISIONS.md entry.
